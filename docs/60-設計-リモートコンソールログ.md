# リモートコンソールログ設計

## 背景

iPhone Safari ではブラウザの開発者ツールが使えず、デバッグが困難。
ブラウザ側の `console.log/warn/error/info` や未キャッチ例外を
Nakama match message（新 opcode）経由でサーバに送信し、
`docker logs` で閲覧できるようにする。

## 設計方針

- 新 opcode `OP_REMOTE_LOG = 18`（C->S のみ、サーバは broadcast しない）
- デフォルト OFF、DebugOverlay のトグルボタンで ON/OFF（Cookie 永続化）
- match 参加前のログはバッファリングし、接続後にフラッシュ
- Babylon.js 等サードパーティのログや `window.onerror` も全て捕捉

## キャプチャ対象

| 対象 | 方法 | tommieChat 以外も含む？ |
| --- | --- | --- |
| `console.log/warn/error/info` | 元のメソッドを上書き | Yes |
| 未キャッチ例外 | `window.onerror` | Yes |
| 未処理 Promise reject | `unhandledrejection` | Yes |

## メッセージペイロード

```json
{
  "entries": [
    { "l": "error", "m": "TypeError: Cannot read ...", "t": 1713000000000 },
    { "l": "log",   "m": "snd initPos x=10 z=20",    "t": 1713000000050 }
  ],
  "ua": "Mozilla/5.0 (iPhone; CPU iPhone OS 17_4 like Mac OS X) ..."
}
```

- `l` (level): `"log"` / `"warn"` / `"error"` / `"info"` / `"onerror"` / `"unhandled"`
- `m` (message): 文字列化した引数（500 文字で切り詰め）
- `t` (timestamp): `Date.now()` ミリ秒エポック
- `ua` (userAgent): デバイス識別用（バッチごとに 1 回送信）

## サーバ出力例

```text
12:34:56.7 REMOTE_LOG [12:34:55] uid=abc123(taro) sid=def4 level=error msg=TypeError: Cannot read ...
```

## レートリミット

### クライアント側

- バッチタイマー: 2 秒間隔でフラッシュ、最大 20 エントリ/バッチ
- バッファ上限: 200 エントリ（古いものから破棄）
- メッセージ長: 500 文字/エントリ

### サーバ側

- `rateLimitLog = 5` メッセージ/秒/セッション（既存パターンと同じ）
- メッセージテキスト: 1000 バイトで切り詰め
- ドロップ統計は既存の RATE_DROP ログに統合

**最大スループット**: 5 バッチ/秒 x 20 エントリ = 100 行/秒

## バッファリングとマッチライフサイクル

`sendMatchState` はアクティブなマッチが必要だが、
console 出力はマッチ参加前から発生する。

1. `remoteLogSink` は `enabled` 設定時からバッファリング開始
   （DebugOverlay の Cookie 復元で起動時に ON になる）
2. `_sender` が `null` の間、`flush()` は no-op
   -- エントリはバッファに蓄積
3. `joinWorldMatch` 完了時に `setSender()` で送信関数を登録
   - 即座にバッファをフラッシュ
4. disconnect/logout 時に `_sender = null` にリセット
   -- エントリは再びバッファに蓄積、再接続時にフラッシュ

## 再帰ループ防止

`sendRemoteLog` 内で `console.log` を呼ばない。
`.catch(() => {})` でサイレント処理し、送信エラーがログに再入しないようにする。

## 実装順序

1. **サーバ** (`main.go`): opcode + rate limit + ハンドラ
   -- クライアント変更前にデプロイ可能（opcode 18 を送る側がまだいないため安全）
2. **クライアント opcode + 送信** (`NakamaService.ts`)
3. **console ラッパー + sink** (`main.ts`)
4. **UI** (`index.html` + `DebugOverlay.ts`)

## 検証手順

1. `npm run check` -- TypeScript 型チェック
2. `npm run test` -- テスト通過確認
3. `bash nakama/doBuild.sh --fresh` -- Go ビルド
4. `bash nakama/doRestart.sh` -- dev 環境再起動
5. ブラウザで DebugOverlay -> 52.RemoteLog を ON にして操作
6. `docker logs nakama 2>&1 | grep REMOTE_LOG` でログ確認
