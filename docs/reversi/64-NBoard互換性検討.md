## 1. 根本的な違い

メッセージ書式の短縮（FPGA 向け 2 文字ニモニック）以外の、思想・機能レベルの差分。

### 1.1 局面伝達の形式

| | 本書 | NBoard |
| ---- | ---- | ------ |
| 形式 | `BO` + 64 文字（盤面スナップショット） | GGF（**棋譜全体**） |
| 持つ情報 | 現在の石配置のみ | 初期局面＋着手履歴＋プレイヤー名＋時計＋結果 |
| 復元性 | 「この瞬間の局面」だけ伝わる | 初手から今までの指し手が全部伝わる |

→ **NBoard は履歴を伝える、本書は状態を伝える**。思想が真逆。

### 1.2 手番・色の指定方法

| | 本書 | NBoard |
| ---- | ---- | ------ |
| 色の伝達 | `SB` / `SW` で明示指示 | GGF の `PC[Black]` `PB[...]` タグに埋め込み |
| 手番計算要求 | `SB` 受信でそのまま「計算せよ」 | `set game <GGF>` → `go` の 2 段階 |

→ NBoard は「局面設定」と「思考要求」を分離、本書は一体化（状態遷移に組み込み）。

### 1.3 思考制御

| | 本書 | NBoard |
| ---- | ---- | ------ |
| 探索深さ指定 | **なし** | `set depth <n>` |
| 時間制限 | サーバー側で一方的に切るだけ | GGF の `TB[...]` `TW[...]` タグ |
| 強度切替 | なし | depth で自在に切替 |

### 1.4 思考の可視化（進捗 / 評価 / 候補）

| | 本書 | NBoard |
| ---- | ---- | ------ |
| 進捗通知 | **なし** | `status "d=20 nodes=..."` を思考中に何度も返す |
| 性能情報 | なし | `nodestats <nodes> <time>` |
| 候補手ランキング | なし | `hint <n>` で上位 n 手＋評価値 |
| 着手評価値 | なし | `search <mv> <eval> <depth>` |

→ NBoard は「なぜその手を選んだか」を外から観察可能、本書はブラックボックス。

### 1.5 学習・解析機能

| | 本書 | NBoard |
| ---- | ---- | ------ |
| 学習コマンド | なし | `learn`（直前対局を学習） |
| 解析モード | なし | `hint` を時間をかけて呼ぶと解析になる |

### 1.6 ハンドシェイク

| | 本書 | NBoard |
| ---- | ---- | ------ |
| プロトコルバージョン | `VE` で識別名を返すのみ（バージョンは文字列） | `nboard <version>` で**数値バージョンネゴシエーション** |
| CPU ID 自由度 | 文字列 16 文字任意 | 同等 |

### 1.7 運用モデル

| | 本書 | NBoard |
| ---- | ---- | ------ |
| ライフサイクル | ずっと接続維持（BOOT→IDLE→対局→IDLE→...） | プロセス起動/終了で対局区切り（subprocess） |
| 単独リセット対応 | `RE\n` で CPU が能動送信＝サーバーが直前指示再送 | なし（プロセスが死んだら作り直すだけ） |
| 終局通知 | `EB` / `EW` / `ED` を明示送信 | GGF の結果タグ、または `go` で合法手なし |

### 1.8 エラー処理

| | 本書 | NBoard |
| ---- | ---- | ------ |
| エラー応答 | `ER\n`（任意、ログのみ） | 標準化なし（stdin 閉じると終了） |
| 不正着手対応 | 検討事項で未確定 | 同じく曖昧 |

## 2. 互換性を高める拡張案

FPGA 向けの 2 文字ニモニックは残したまま、**機能面を NBoard に寄せる**方向の拡張候補。難度順。

### 2.1 ★ 低コスト・高効果

どれも **CPU 側が任意実装でよい** ので、FPGA 作者にペナルティなく追加可能。

1. **`ST<text>\n`（STATUS）コマンド追加**
    - CPU → サーバーで思考中の状態を任意送信。例: `ST d=12 n=45000\n`
    - NBoard の `status` 相当
    - 進捗表示できて UX 向上（「思考中 depth=12」などを UI 表示）
2. **`NC<nodes>,<ms>\n`（NODESTATS）**
    - 着手の後に探索統計を任意返送。NBoard の `nodestats` 相当
    - 大会運用で CPU 性能比較ができる
3. **`VE` 応答にプロトコルバージョンを追加（採用済み）**
    - `VE<NN><識別名>` 形式（`<NN>` は 2 桁数字、先頭固定）。例: `VE01tommie-CPU v1`
    - NBoard の `nboard <version>` 相当
    - 将来拡張時にサーバーがプロトコル機能切替できる

### 2.2 ★★ 中コスト・機能拡張

1. **`SD<n>\n`（SET DEPTH）**
    - サーバー → CPU で思考深さ指定
    - CPU が対応しなければ無視（`ER` 任意返送）
    - 大会で強度を揃えられる
2. **`HI<n>\n`（HINT）**
    - サーバー → CPU で上位 n 手要求
    - 応答は `HI<mv1>,<eval1>;<mv2>,<eval2>...\n`
    - 解析モード実装可能
3. **`BO` に履歴形式（BH）を追加**
    - `BH<Thor 形式棋譜>\n` で初手からの全手を渡す
    - 既存の NBoard / Edax との互換レイヤ作成が容易

### 2.3 ★★★ 大改造

1. **GGF 互換モード追加**
    - `GG<GGF 文字列>\n` でフル GGF 受理
    - FPGA CPU は対応しないが、ソフトウェア CPU なら既存 NBoard 実装を流用可能
2. **プロトコルバージョン分岐**
    - v1 = 現行、v2 = NBoard 互換機能付き
    - `VE` 応答の `p<N>` でサーバーが機能切替

## 3. 現実的な提案

tommieChat のユースケースを考えると:

- **自作 CPU 大会** → 本書そのままで OK（FPGA 作者が楽）
- **強豪 AI 連携（Edax など）** → 別プロトコル（NBoard Protocol そのもの）で叩く

「同じプロトコルで両立させる」より、「用途別にアダプタを分ける」設計が筋。
本書を NBoard に近づけると FPGA 作者に負担がかかる。ただし §2.1 の **★ 低コスト拡張**
は CPU 側が任意実装でよいのでペナルティなく追加可能。

**結論**: 2.1 の 3 項目を本書に取り込む方向を推奨。2.2 以降は需要が出てから検討。
