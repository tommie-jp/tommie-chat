# リバーシ UART リプレイテスト

CPU 側（`test/reversi/reversi_cpu.py`）の挙動を回帰テストする仕組み。
シリアルテストパネルで収録した対局ログをそのままシナリオファイルとして
`--replay` で再生し、CPU の応答を期待値と突き合わせる。

実シリアルポート・実機なしで回せるので、ロジック変更時の確認に使う。

## シナリオファイル形式

1 行 1 メッセージのテキストファイル。

```text
TX <ascii>    # ホスト → CPU (CPU 入力)
RX <ascii>    # CPU → ホスト (CPU 応答)
# 先頭 # はコメント
```

- LF/CR は付けない（テスト側で付加）
- 大文字小文字は [61-UARTプロトコル仕様.md](61-UARTプロトコル仕様.md) §4 に準拠
- PI/PO ハートビートは通常含めない（時刻依存で非決定的なため）

例: [test/reversi/scenarios/01-opening.txt](../../test/reversi/scenarios/01-opening.txt)

```text
TX SB
RX MOd3
TX MOc3
RX MOb3
```

## 記録手順（シリアルテストパネル経由）

1. `/test-web-serial-api.html` を開く（dev サーバの Web Serial テストパネル）
2. 表示オプションで **Hex: Replay (TX/RX)** を選択
3. **PI/PO除外** チェックボックスを ON（デフォルト）
4. 実機 CPU or Python CPU と接続、対局プレイ or 手動コマンド送信
5. ログ欄右上の **Copy** でクリップボードにコピー
6. `test/reversi/scenarios/NN-<名前>.txt` として保存

## 再生手順

```bash
cd test/reversi

# 単一シナリオ
python3 reversi_cpu.py --replay scenarios/02-auto-pass.txt

# 全シナリオ一括 (WSL / Linux)
./doTest-replay.sh

# 全シナリオ一括 (Windows PowerShell)
.\doTest-replay.ps1
```

CPU 応答が期待値と一致すれば `[OK]`、不一致なら `[FAIL]` と差分を stderr に出力して exit 1。

リプレイ比較から除外されるコマンド:

- `PI` / `PO` (ハートビート、シナリオ側で扱わない)
- `ST` (盤面スナップショット等の状態通知、診断用)

ブラウザの `Replay (TX/RX)` モード出力にタイムスタンプ接頭辞 `[HH:MM:SS.mmm]` が付いていても、
`reversi_cpu.py --replay` のパーサーが自動で剥がす（行頭の `[...]` と空白）。

## Adapter 側ユニットテスト (Vitest)

リプレイテストは CPU (Python) の挙動を検証するが、`SerialReversiAdapter` (TypeScript) の
状態機械も Vitest で単体テストしている。

ファイル: [test/SerialReversiAdapter.test.ts](../../test/SerialReversiAdapter.test.ts)

ブリッジ (`window.__serialTestBridge`) と `SerialReversiMovePort` をモックし、
`onGameStateUpdate` を順に駆動して以下を検証する:

- ゲーム開始時の `SB` / `SW` / `BO` 発行
- 通常着手 (case 2)
- サーバー auto-pass 時の連続 `MOxx` 発行 (case 2b) — `05-opp-multi-move.txt` と対の TS テスト
- CPU からの `MO` 受信 → `othelloMove` RPC 呼び出し
- `RS` 受信 → `BO` + 直前 `MO` 再送
- `RS` 上限超過 → `othelloResign` (反則負け)
- `EN` 受信 → `othelloResign`
- 終局 `EB` / `EW` / `ED` 発行

実行:

```bash
npx vitest run test/SerialReversiAdapter.test.ts
```

## 統合実行 (Phase 1 + Phase 2)

Python シナリオと Adapter Vitest を続けて実行する統合スクリプト:

```bash
# WSL / Linux (推奨)
./test/reversi/doTest-all.sh

# Windows PowerShell 版 (WSL 経由で vitest を呼ぶ)
.\test\reversi\doTest-all.ps1
```

WSL 版が素直で高速。PowerShell 版は UNC パス上から `npx` を叩けないため、
内部で `wsl bash -c ...` に切り替えて fnm 経由の Node を呼ぶ (`~/.bashrc` が
非インタラクティブで early-return するので `fnm env` を明示 eval している)。

## ファイル配置

| パス | 役割 |
| ---- | ---- |
| `test/reversi/reversi_cpu.py` | CPU 本体。`--replay` でシナリオ再生 |
| `test/reversi/scenarios/*.txt` | シナリオ集 |
| `test/reversi/doTest-replay.sh` / `.ps1` | 全シナリオ一括実行 |

## 新しいシナリオを足すとき

- 番号は 2 桁の連番（既存との衝突回避）
- ファイル冒頭にコメントで狙い（何を検証するか）を書く
- CPU の戦略は first-legal 固定なので応答は決定的。AI 化したら期待値も更新する
- 相手の auto-pass は `TX PA` で表現する（[SerialReversiAdapter](../../src/SerialReversiAdapter.ts) が送る通知と同じ）
- CPU 側が auto-pass されるケース (サーバが CPU の手番をスキップして相手が連続着手する)
  は `TX MOxx` が連続するだけ。CPU は毎回 `PA` を返す → シナリオに `RX PA` を
  正しく記述する（例: [05-opp-multi-move.txt](../../test/reversi/scenarios/05-opp-multi-move.txt)）
