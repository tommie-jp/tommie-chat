GOについて

Nakama の場合、Go 一択がおすすめです。理由:

Go は遅くない — 「JSと同程度」は誤解です。Go はコンパイル言語で、一般的にJSの2〜5倍高速です。Cより遅いのは事実ですが、ゲームサーバ用途では十分すぎる性能です。

Nakama 公式の推奨 — Nakama 自体が Go で書かれており、Go プラグインはネイティブ実行されます。JS/Lua ランタイムはインタプリタ経由なので、Go プラグインの方が高速です。

Go vs Lua vs JS（Nakama内）:

Go: ネイティブ速度、型安全、並行処理が得意 Lua: 軽量だが機能制限あり JS: TypeScript→JSにトランスパイル必要、VMオーバーヘッドあり 今回のチャンク排他制御のように sync.RWMutex が使えるのも Go の強みです。JS/Lua ではこのレベルの並行制御はできません。

結論: Nakama 拡張には Go が最適です。現在の選択は正しいです。

AOIメモ

おすすめの対策（優先度順） ✔1. AOIをFarClipから切り離す（必須・効果大）

AOI半径 = 固定48セル（3チャンク分） → 7×7 = 49チャンク / 256チャンク = カバー率19% → setBlockの送信先: 平均190人/1000人 デバッグツールで調整可能にしておく。

  1. ワールドを広くする（効果最大） チャンク数 セル数 AOI半径48のカバー率 setBlock送信先 16×16（現在） 256×256 19% ~190人 32×32 512×512 5% ~50人 64×64 1024×1024 1.2% ~12人 64×64チャンクなら1000人でも1回のsetBlockで約12人にしか送らない。

  2. MatchLoopのtick rate削減 現在: tickRate=10（100ms毎） 1000人のメッセージを毎tick処理するのは重い AOI更新は低頻度でよいのでtickRate=5で十分

  3. マッチ分割（シャーディング） 1マッチ1000人はNakamaには重い ワールドをエリア別に複数マッチに分割 例: 4×4エリア = 16マッチ、各マッチ最大~63人 実装コストは大きいが、スケーラビリティは最高

  4. setBlockの送信レート制限 1プレイヤーあたり秒間1〜2回に制限 連打によるサーバ負荷スパイクを防止 推奨ロードマップ フェーズ 対策 想定同時接続 工数 Phase 1 AOIをFarClipから独立（固定半径） ~200人 小 Phase 2 ワールド拡大（64×64チャンク） ~500人 中 Phase 3 setBlockレート制限 ~700人 小 Phase 4 マッチ分割（シャーディング） ~1000人+ 大
