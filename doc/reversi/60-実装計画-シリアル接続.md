## CPU 参加モデル（案 B2 採用）

オーナーの立ち位置を明確化しておく。

- **対局スロット (黒/白)**: オーナーは **座らない**。CPU 対戦ゲームの 1 席は自作 CPU（オーナーのシリアル先）が占有し、もう 1 席を他プレイヤーが埋める
- **オーナーの役割**: 観戦 + シリアル中継のみ。盤面は `othelloSubscribe` 経由で取得し、相手手を UART で CPU へ送る → CPU 応答を `othelloMove` で nakama に返す
- **サーバ側制約**:
  - CPU 対戦ゲームはオーナー自身が `othelloJoin` で対局席に座れないようにブロック
  - CPU 側手番の `othelloMove` は **オーナーのみ** 発行可（他プレイヤー発行なら拒否）
- **招待フロー** (`?ot=<N>` URL で起動した場合):
  - オーナー以外: リバーシロビーを開く（通常と同じ）
  - オーナー: 「自分が作成した CPU 対戦ゲームへの誘い」としてトースト表示。YES → リバーシプレイパネルへ遷移して観戦開始 / NO → ロビー
  - `applyState` 内の自動遷移（誘い受諾後）は OK

### RPC 拡張ポイント

| RPC | 拡張 |
| --- | --- |
| `othelloCreate` | `isCpu: boolean` を追加。true で作成された試合はサーバが CPU 対戦ゲームとしてマーク |
| `othelloMove` | CPU 手番の中継時に `cpu: true` 等のフラグを付与。サーバはオーナー判定 & マッチの CPU フラグを見て承認 |

> 新規 RPC (`othelloCpuMove` 等) は作らない。既存 RPC への最小拡張で済ませる。

## 想定手順（当面の運用）

1. シリアルテストパネルを開く
2. CPU とのシリアル接続を確認（COM1 ⇔ COM2 エミュでの往復確認）
3. シリアルテストパネルの「新ゲーム作成」ボタンを押す（遷移なし）
4. リバーシロビーに CPU 対戦ゲームが表示される（オーナー=作成者、コメント=「〇〇のCPU」）
5. 他プレイヤーが「参加」ボタンで参戦 → オーナーの自作 CPU と対戦
6. オーナーはシリアル経由で CPU の着手を中継（閲覧画面で盤面確認）
7. 終局後、オーナーがリバーシロビーから削除

> 自動ループ（CPU ホストモード）は [59 §CPUホストモードの動作仕様](59-設計-外部CPU接続.md) のフェーズ 5 以降。当面は毎回人が作成・削除する。

## 実装に入る前の設計詰め

コードを書き始める前に確定させたい点。

### 1. 状態機械の図式化

SerialReversiAdapter の状態遷移を [59 §ブラウザ側実装](59-設計-外部CPU接続.md) の責務リストから PlantUML で起こす。

- 状態候補:
  - 未接続 / 接続済み・対局なし / 対局中・自分手番 / 対局中・相手手番 / 対局中・CPU応答待ち / 接続エラー
- 遷移トリガ:
  - ユーザ操作（接続・切断・ゲーム作成・投了）
  - シリアル受信（着手・`PASS` / `RESIGN` / `READY`）
  - nakama 通知（`othelloUpdate` / 対局終了）
  - タイムアウト

### 2. 盤面同期プロトコルの擬似コード

1 手の往復を疑似コードで書き下す:

```text
自分手番になる
  → CPU へ "S B\r\n" (初手) or "M <相手の手>\r\n"
  → CPU 応答 "<手>\r\n" を受信
  → othelloMove RPC 呼び出し
  → nakama から othelloUpdate 受信
  → 相手手番 → 相手の手を受信したら CPU へ "M <相手の手>\r\n"
  → 以下ループ
```

エッジケース:

- CPU から受信した手が不正 → 再送要求 / 手動入力 / 強制投了
- CPU が `PASS` を返した → nakama 側でパス扱いにできるか確認
- nakama から切断 → 再接続後の局面同期（`B <盤面全体>\r\n` を投入）

### 3. CPU タイムアウト動作

無応答時の挙動を決める:

- タイムアウト時間: 30 秒? 大会ルール `max(60-8*log10(clock), 0.1)` 秒?
- 無応答時の UI: 警告表示のみ? 強制パス/投了?
- 再試行: `READY` 受信で直前指示を再送するか

### 4. 手動入力モードとの共存

同じパネル内でシリアル接続中と手動入力モードをどう切り替えるか:

- モード切替ボタン
- 切替時に盤面はそのまま引き継ぐ
- シリアル切断時は自動で手動モードへフォールバックするか

### 5. 再接続時の bootstrap

対局中に USB を挿し直した場合:

- 選択肢 A: `B <盤面全体>\r\n` で局面投入、手番から再開
- 選択肢 B: 諦めて投了させる
- 選択肢 C: 手動入力モードへフォールバック

MCU 実装の自由度によって選択が変わる。

### 6. 代理モデルの表示（決定済み）

- **案 B2** を採用。オーナーは対局席に座らず観戦のみ、対局席は「自作 CPU」が直接占有する形にする
- ロビー上は CPU 対戦ゲームとしてマーク（コメントに「〇〇のCPU」）。他プレイヤーは普通に「参加」で CPU と対戦できる
- サーバはマッチの CPU フラグとオーナー ID を保持し、
  - オーナーの `othelloJoin` を拒否
  - CPU 手番の `othelloMove` はオーナー発行のみ承認

### 決めなくていい点（後回し推奨）

- Role ベース API 制限（[doc/51](../51-設計-リバーシCPUブリッジ.md)）— 代理モデルなら不要
- 24/7 CPU ホストモード（[59 フェーズ 5](59-設計-外部CPU接続.md)）— Android 持ち込み運用では無縁
- Google 認証移行 — 通常の人間ログインで済む
- サーバ側不正手検証の強化 — まず成功手順を通す方針
