# ゴーストアバター問題と対策

- 作成日: 2026-04-04
- 更新日: 2026-04-05（対策実装完了）

## 問題の概要

iPhone Safari でページをリロードすると、古いセッションの WebSocket が
バックグラウンドで生き続け、サーバー上にゴーストアバターが残る。
ping/pong も正常に応答するため、pong タイムアウトでは消えない。

## 対策前のログインフロー（3段階 — アトミックでない）

```text
クライアント                    サーバー
─────────                     ─────────
1. joinChat()    ──────►      チャットプレゼンスに追加（他全員に通知）
2. joinMatch()   ──────►      マッチプレゼンスに追加（他全員に通知）
3. sendInitPos() ──────►      アバター位置を登録

1〜3の間にリロードすると、サーバーに中途半端な状態が残る → ゴースト
```

## 対策後のログインフロー（1段階 — アトミック）

```text
クライアント                    サーバー
─────────                     ─────────
login()
  └ WebSocket接続のみ（joinChat 廃止）

joinWorldMatch(metadata)
  └ joinMatch(metadata) ────► MatchJoinAttempt: metadata から初期位置を PendingInit に保存
                               MatchJoin:
                                 1. prevSid があれば古いセッションを MatchKick + ステート即削除
                                 2. PendingInit から初期位置を即座に登録
                                 3. OP_SYSTEM_MSG (join) を全員に送信
                               → joinMatch 1回で全て完結 ✓
```

## 実装した対策

### 1. joinMatch アトミック化

- `joinChat` を廃止、チャットメッセージはマッチ経由（`OP_CHAT=13`）に統合
- `sendInitPos` を廃止、`joinMatch` のメタデータに初期位置を含める
- サーバーの `MatchJoin` で `PendingInit` から即座に位置登録

### 2. prevSid によるゴーストキック

```text
フロー:
  1. ログイン成功 → sessionId を localStorage に保存（タブ固有キー）
  2. リロード → localStorage から前回 sessionId を読み取り
  3. joinMatch(metadata: { prevSid: "old-session-id", ... })
  4. サーバー MatchJoin: prevSid を全マップから即座に削除 + MatchKick
     + AOI_LEAVE を全クライアントに送信（既にフラッシュ済みの AOI_ENTER を打ち消す）
  5. 新セッション参加
```

- `sessionStorage` にタブ固有ID（`tabId`）を保存 → タブ間で干渉しない
- `localStorage` に `prevSessionId_{tabId}` として保存
- 同一ブラウザの複数タブでのログインが可能

### 3. クライアント側ガード

- `onAOIEnter`: `userMap` にないセッション → スキップ
- `onAvatarInitPos`: `userMap` にないセッション → スキップ

MatchLoop が MatchJoin より先に実行されるため、キック対象の AOI_ENTER が
既にフラッシュ済みの場合がある。クライアント側で userMap を参照し、
退出済みセッションのアバター作成を防止する。

## 廃止した対策（不要になったため削除）

| 対策 | 理由 |
| ---- | ---- |
| ログインクールダウン（5秒待機） | prevSid キックにより不要 |
| `beforeunload`/`pagehide` の `leaveMatch` | iOS Safari で確実に送信されないため効果なし |
| `pageshow` bfcache リロード | 上記と同様 |
| `cleanupBeforeUnload()` メソッド | 上記と同様 |
| デバイスID ベースのキック | prevSid 方式に置き換え |
| `?multilogin` フラグ | タブ固有 prevSid により不要 |
| タイマーベースの切断検知/孤立アバター削除 | 根本対策ではないため削除 |
| 同一ユーザーID フィルタ | 複数ブラウザログインをブロックするため削除 |

## アーキテクチャ変更まとめ

### 新設 opcode

| opcode | 名前 | 方向 | 用途 |
| ------ | ---- | ---- | ---- |
| 13 | `OP_CHAT` | C→S→C | チャットメッセージ（joinChat 廃止の代替） |
| 14 | `OP_SYSTEM_MSG` | S→C | システムメッセージ（ログイン/ログアウト通知） |

### matchState の変更

| フィールド | 変更 |
| ---------- | ---- |
| `PendingInit` | 追加: joinMatch メタデータの初期位置を一時保存 |
| `PrevSIDs` | 追加: 前回セッションID（ゴーストキック用） |
| `DeviceIDs` | 削除: prevSid 方式に置き換え |
| `MultiLogin` | 削除: タブ固有 prevSid により不要 |

### サーバー処理順序の注意点

Nakama のマッチ tick 内処理順序:

```text
1. MatchLoop（メッセージ処理 + PendingAOIEnter フラッシュ）
2. MatchJoin（新セッション参加）
3. MatchLeave（退出処理）
```

MatchLoop が先に実行されるため、キック対象の移動メッセージが先に処理され
AOI_ENTER が送信済みになる場合がある。対策:

- サーバー: MatchJoin で AOI_LEAVE を明示送信して打ち消す
- クライアント: userMap にないセッションの AOI_ENTER/InitPos をスキップ

Cookie/localStorage の共有範囲と制限事項は
[doc/12-クライアント永続化設計.md](12-クライアント永続化設計.md) を参照。
