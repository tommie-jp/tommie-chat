# 検討事項: ログインキュー方式

現在のレートリミット方式（`MAX_LOGIN_RATE_PER_SEC=50`）の代替として、
大規模化した場合に検討するログインキュー方式の設計メモ。

---

## 現状: レートリミット方式

`nakama/go_src/main.go` の `MatchJoinAttempt` でレートリミッターが
秒あたりの joinMatch 数を制限している。超過時はエラーで拒否し、
クライアント側（テストコード）がリトライする。

```text
クライアント → 認証 → WS接続 → joinMatch → 拒否 → リトライ → joinMatch → 入場
```

| 項目 | 値 |
| ---- | -- |
| 制限値 | `MAX_LOGIN_RATE_PER_SEC=50`（docker-compose.yml） |
| 制限箇所 | `MatchJoinAttempt`（Go プラグイン） |
| 超過時 | `"too many logins, try again later"` エラー |
| クライアント対応 | 500〜1000ms 後にリトライ（最大10回） |

---

## キュー方式の概要

レートリミットが「拒否して再試行させる」のに対し、キューは「受け付けて順番待ちさせる」。

```text
クライアント → 認証 → WS接続 → joinMatch → キューに入る → 順番待ち → 入場許可
                                                ↓
                                          「あなたは342番目です」
                                          「推定待ち時間: 2分」
```

大規模 MMO（WoW, FF14 等）で広く採用されている方式。
認証とゲーム入場を分離し、認証は無制限に受け付け、ゲーム入場だけを制御する。

---

## レートリミット vs キュー

| | レートリミット（現在） | キュー方式 |
| --- | --- | --- |
| 超過時の挙動 | エラーで拒否 → クライアントがリトライ | 待機列に入れる → 順番が来たら入場 |
| クライアント負荷 | リトライでサーバに再リクエスト（負荷増大） | 1回のリクエストで待機（負荷なし） |
| UX | エラー表示 or 見えないリトライ | 「待ち人数: N人」の表示が可能 |
| 公平性 | リトライタイミング次第（不公平） | 先着順（公平） |
| 実装複雑度 | 低 | 中〜高 |
| サーバ負荷 | リトライが集中すると逆に負荷増 | 安定（キューから一定レートで処理） |

---

## 参考: Minecraft のレートリミット

Minecraft は「秒あたり何人」ではなく「同一 IP の再接続間隔（ms）」で制限する設計。

| サーバ種別 | 制限方式 | デフォルト値 |
| ---- | ---- | ---- |
| Bukkit/Spigot | connection-throttle（同一IPの再接続間隔） | 4000ms |
| BungeeCord | connection_throttle（同上） | 4000ms |
| Vanilla | パケットレート制限 | 約200パケット/秒 |

## tommieChat での実装案

### Go プラグイン側

`MatchJoinAttempt` で拒否する代わりに、キューに追加する。

1. キューに追加（Go 側で slice や channel で管理）
2. 毎 tick で N 人ずつキューから取り出して入場許可
3. 待機中のクライアントには WebSocket で順番を通知

### ブラウザ側

キュー待ち中の UI を表示する。

- 「接続待ち: あなたは N 番目です」
- 推定待ち時間の表示
- キャンセルボタン

## 導入判断基準

現在の規模（目標 1000〜2000 人）ではレートリミット＋リトライで十分。
以下の条件に該当する場合にキュー方式への移行を検討する。

- 同時接続数が数千〜数万人規模になった
- ログイン集中時の UX が問題になった（サービス公開後）
- リトライによるサーバ負荷がボトルネックになった
