# nakama チューニングパラメータ

同時接続1000人を目標としたnakamaサーバの設定チューニング。
設定は `nakama/docker-compose.yml` の entrypoint に `--パラメータ名 値` 形式で追加する。

---

## 設定済みパラメータ

### `--match.input_queue_size 4096`

| 項目 | 内容 |
| ------ | ------ |
| デフォルト | 128 |
| 設定値 | 4096 |
| 単位 | メッセージ件数 |

**概要:**
MatchLoop の1tick（100ms）あたりに受け付けられる WebSocket メッセージの最大キューサイズ。
これを超えたメッセージはドロップされる。

**選定理由:**
1000人が同時に `initPos` + `AOI_UPDATE` を送信すると、1tick に最大 **2000件** のメッセージが集中する。
デフォルト128では約180人でドロップが発生することを確認（`initPos 178/180` 等）。
2000件に対して余裕を持たせ 4096 に設定。

---

### `--match.call_stack_size 1024`

| 項目 | 内容 |
| ------ | ------ |
| デフォルト | 128 |
| 設定値 | 1024 |
| 単位 | 呼び出し件数 |

**概要:**
Match 関連の内部関数呼び出し（MatchJoin, MatchLeave 等）のキューサイズ。
多人数が同時に joinMatch を実行するとキュー溢れが起きる可能性がある。

**選定理由:**
1000人同時 joinMatch に対応できるよう 1024 に設定。

---

### `--socket.outgoing_queue_size 1024`

| 項目 | 内容 |
| ------ | ------ |
| デフォルト | 64 |
| 設定値 | 1024 |
| 単位 | メッセージ件数（コネクション毎） |

**概要:**
各 WebSocket コネクションの送信バッファサイズ。
サーバからクライアントへの送信が追いつかない場合、このキューが溢れると接続が切断される。

**選定理由:**
AOI_ENTER バルク配信では、1人のプレイヤーが参加するたびに最大 N-1 件の送信が発生する。
1000人規模では送信バッチが大きくなるため、デフォルト64では溢れるリスクがある。
1000人規模のピーク送信量を考慮して 1024 に設定。

---

### `--runtime.max_count 4096`

| 項目 | 内容 |
| ------ | ------ |
| デフォルト | 256 |
| 設定値 | 4096 |
| 単位 | goroutine 数 |

**概要:**
Lua/Go ランタイムで同時実行できる goroutine（RPC処理スレッド）の上限。
`syncChunks`、`getServerInfo`、`storeLoginTime` などの RPC が同時に大量に来た場合、
この上限を超えると RPC がキューで待機またはエラーになる。

**選定理由:**
1000人が同時にログインシーケンスを実行すると、複数のRPCが並列で実行される。
デフォルト256に対して 4096 に設定し、RPC拒否が起きないようにする。

## バリデーション設定（runtime.env）

`docker-compose.yml` の `--runtime.env` で設定する。
Go プラグイン起動時にログ出力される（`validationConfig: ...`）。

### 入力制限

| 環境変数 | デフォルト | 単位 | 説明 |
| -------- | ---------- | ---- | ---- |
| `MAX_CHAT_LEN` | 200 | 文字数 | チャットメッセージの最大文字数。超過分は切り捨て。制御文字（`\n`以外）も除去される |
| `MAX_BLOCK_ID` | 255 | ID値 | 有効なブロックIDの上限。超過したブロック設置はドロップ |

### レート制限（セッション単位、1秒ごとにリセット）

| 環境変数 | デフォルト | 単位 | 説明 |
| -------- | ---------- | ---- | ---- |
| `MAX_LOGIN_RATE_PER_SEC` | 50 | 人/秒 | ログイン試行の制限（全セッション共通） |
| `RATE_LIMIT_CHAT` | 5 | 回/秒 | チャットメッセージの送信制限 |
| `RATE_LIMIT_BLOCK` | 20 | 回/秒 | ブロック設置/削除の制限 |
| `RATE_LIMIT_MOVE` | 30 | 回/秒 | 移動コマンドの制限 |

> レート超過分はサイレントドロップ（エラー返却なし）。
> 正常なクライアント操作では上限に達しない値に設定されている。

### 座標・範囲チェック（設定不要、自動適用）

| 対象 | チェック内容 |
| ---- | ------------ |
| `opInitPos` / `opMoveTarget` | ワールド座標 x, z が ±512 範囲内か。範囲外はドロップ+ログ |
| `opAOIUpdate` / `syncChunks` | チャンク座標をクランプ後、min > max なら拒否+ログ |
| `opBlockUpdate` | ブロック座標 gx, gz が 0〜1023 範囲内か |

## チューニング経緯

| 時期 | 最大同時接続 | 施策 |
| ------ | ------------ | ------ |
| 初期 | ~100人 | — |
| AOI_ENTER バルク化 | 170人 | WebSocketメッセージ数を O(N²) から O(N) に削減 |
| BroadcastMessage 統合 | 170人 | Go側のBroadcastMessage呼び出し回数削減 |
| nakamaキュー拡張 | 測定中 | `input_queue_size` 等を1000人規模に拡張 |

---

## 参考: メッセージ数の理論値（全員が同一AOI内の場合）

| 同時接続数 N | AOI_ENTER 送信件数（バルク化前） | AOI_ENTER 送信件数（バルク化後） |
| ------------ | ------------------------------- | ------------------------------- |
| 100人 | 9,900件 | ~5,050件 |
| 500人 | 249,500件 | ~125,250件 |
| 1,000人 | 999,000件 | ~500,500件 |

> バルク化後でも N=1000 では約50万件。AOI範囲を近隣のみに限定することで
> メッセージ数を大幅に削減できる（O(N²) → O(N)）。
