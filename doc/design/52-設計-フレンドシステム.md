# 設計: フレンドシステム

tommieChat のフレンドリスト設計方針。
他 MMO・Pigg・SNS（X 等）を比較し、tommieChat 向けの提案をまとめる。

部屋システムの VRChat 風 Friends+ 公開設定（[50-設計-部屋システム.md 第10章](50-設計-部屋システム.md#10-vrchat-の-5-段階公開設定詳細)）の前提となる機能。

## 1. SNS/MMO のフレンドモデル比較

### 関係モデルの 4 分類

| モデル | 例 | 関係の方向 | 承認 | 用途 |
| ---- | ---- | ---- | ---- | ---- |
| **双方向承認** | Pigg, Facebook, LINE, Mixi, FF14, Habbo | 対称 | 必要 | 信頼関係・親密な交流 |
| **非対称承認** | Twitter 鍵アカ, Instagram 非公開 | 非対称 | 必要 | 情報発信＋ファン管理 |
| **一方向フォロー** | Twitter/X, Instagram, TikTok | 非対称 | 不要 | 情報発信・気軽な追跡 |
| **階層型（サークル）** | Steam, Discord, WoW, Google+ | 対称＋分類 | 必要 | 関係性の使い分け |

### サービス別の詳細

#### アメーバPigg（双方向承認）

- フレンド申請 → 承認 → 相互フレンド
- 「お気に入り」と「フレンド」が別概念
- フレンドの**現在地**（どの部屋にいるか）が見える
- フレンドの部屋に「ピンポン」（ノック）して入室可能
- 上限あり（初期は 100 人 → 課金で拡張）
- 無視リスト（一方的にブロック）

**特徴**: フレンドが「現在地表示」と「優先入室」の権限とセットになっている。tommChat の VRChat 風 Friends+ にこれが必要。

#### Twitter / X（一方向フォロー）

| 機能 | 内容 |
| ---- | ---- |
| フォロー | 相手の許可不要、TL に流れてくる |
| 鍵アカ | フォローに承認必要（非対称承認） |
| ミュート | 相手に気付かれず TL から消す |
| ブロック | 相手から自分が見えなくなる、相互フォロー解除 |
| リスト | フォロー以外で「読みたい人」を分類管理 |
| ブックマーク | 投稿単位の保存（リレーションではない） |

**特徴**: 関係を**情報の流れ**として捉える。「フレンド」概念はなく、相互フォロー（mutual）が暗黙的にそれに相当。

#### Discord（階層型）

| レイヤー | 関係 |
| ---- | ---- |
| フレンド | 双方向承認、DM 可能、オンライン状態見える |
| サーバー | 同じサーバーに所属するだけ（フレンドではない） |
| ブロック | 一方向、メッセージ受信拒否 |

**特徴**: フレンドとサーバーメンバーが分離。tommChat の「フレンド」と「同じ部屋にいる人」の関係に近い。

#### Steam（双方向＋自動分類）

- 双方向承認のフレンド
- 「最近一緒に遊んだ人」自動リスト（フレンドではない一時関係）
- **ニックネーム** — 自分だけが見える相手の愛称
- お気に入りフラグ（フレンドの中で更にピン留め）

**特徴**: 「自動的に関係が形成される」緩いリストと、「明示的に友達になる」リストの併用。

#### FF14 / WoW（MMO 系）

- 双方向承認のフレンドリスト
- フレンドにメモ書きを残せる（Steam のニックネームと類似）
- **ブラックリスト** vs **ミュート**を区別
- フレンド数上限あり（200〜500）
- ギルド/フリーカンパニー（別レイヤーの所属）

#### LINE（連絡先＋承認）

- 電話番号/QR コードで「知り合い候補」自動検出
- 承認後に双方向トーク可能
- ブロック・非表示
- 特徴的: 「知り合いかも？」の自動サジェスト

### モデル比較の整理

```text
                  非対称 ←──────────────→ 対称

   承認なし    Twitter/X        Steam最近遊んだ人
                  │                   │
                  │                   │
   承認あり    Insta鍵           Pigg/LINE/FF14/Discord
              Twitter鍵          Habbo/Facebook
```

## 2. tommChat への提案

### 設計の前提

tommChat は「**チャット主体のアバターSNS（Pigg 立体版）**」を目指しており、

1. **部屋システムの権限基盤**として VRChat 風 Friends+ を実現したい
2. **すれ違い → 仲良くなる** 社交ゲーム性が大事
3. Nakama 標準のフレンド API がそのまま使える（`ListFriends`/`AddFriends`/`BlockFriends`）
4. 商用品質を目指すのでブロック機能必須

→ **基本は Pigg/MMO 系の双方向承認**を採用しつつ、X 風の**軽い片方向リレーション**も併設するのが最適。

### 推奨モデル: 4 種類のリレーション

| 種別 | 方向 | 承認 | 通知 | 用途 |
| ---- | ---- | ---- | ---- | ---- |
| **フレンド** | 双方向 | 必要 | あり | 部屋アクセス権、現在地共有、ピンポン |
| **お気に入り** | 一方向 | 不要 | なし | 「気になる人」を私的にマーク |
| **ミュート** | 一方向 | — | なし | チャット・通知を表示しない |
| **ブロック** | 一方向 | — | なし | 相互に存在を隠す、接触不可 |

#### なぜこの 4 種か

- **フレンド**は VRChat Friends+ の基礎。`worldDef.Visibility=Friends` を実装するために必須
- **お気に入り**は Pigg「お気に入り」と Steam「ピン留め」と X「ブックマーク」の合成。承認不要で軽く、再会したい人をマークできる
- **ミュート**は荒らしまで行かないが鬱陶しい人用。X が証明したように、ブロックより需要が大きい
- **ブロック**は商用品質に必須。荒らし対策の最後の砦

### 各種別の挙動詳細

#### フレンド（双方向承認）

```text
A → B に申請  B が承認  双方フレンド
  Nakama 標準: state=2(InviteReceived) → state=0(Friend)
```

| 機能 | 内容 |
| ---- | ---- |
| 現在地表示 | フレンド一覧で「○○の部屋にいます」 |
| オンライン状態 | オンライン/オフライン/取り込み中 |
| 合流ボタン | フレンドの居る部屋にワープ（VRChat の Join） |
| ピンポン | 招待制部屋にノックを送る |
| ニックネーム | 自分だけ見える相手の愛称（Steam 方式） |
| メモ | フレンドに 1 行メモ（FF14 方式） |

**上限**: 初期 200 人（Pigg・MMO の標準的な値）

#### お気に入り（一方向、相手に通知しない）

- 部屋ですれ違った人を「☆」でマーク
- 自分のお気に入り一覧に追加
- **相手には一切通知しない・見えない**
- フレンド申請の前段階として使う想定
- お気に入り → フレンド申請の導線をつける

Pigg の「お気に入り」が一方的だったのに対し、tommChat ではより明確にプライベートな機能とする。

#### ミュート（一方向、UI 上だけ非表示）

- チャット欄でその人の発言を非表示
- プレイヤーリストでもグレーアウト or 非表示
- 相手のアバターは見える（同じ部屋に居られる）
- 相手には気付かれない

#### ブロック（一方向、相互不可視）

- お互いのアバター・チャット・存在が見えなくなる
- 同じ部屋には**入れない**（ブロック側の判定）
- フレンド関係があれば自動解除
- Nakama 標準の `BlockFriends` を利用

### 軸の使い分け

```text
                  軽い ←─────────────────→ 重い

   非通知    お気に入り    ミュート         ブロック

   通知      （なし）      フレンド申請
   双方向                  フレンド
```

- **お気に入り**: 自分用メモ、軽い
- **フレンド**: 関係が成立、相手に通知あり
- **ミュート**: 不快を軽く回避
- **ブロック**: 完全遮断、商用必須

## 3. データモデル

Nakama 標準の `friends` テーブルで足りるもの:

```text
state=0  Friend          (双方向承認済み)
state=1  Invite Sent     (申請中)
state=2  Invite Received (申請受信)
state=3  Blocked         (ブロック)
```

不足分は Storage API で補う:

```typescript
// collection: "user_relations", key: "rel_{targetUid}", user_id: 自分
interface UserRelation {
    targetUid: string;
    favorite?: boolean;       // お気に入り
    muted?: boolean;          // ミュート
    nickname?: string;        // 自分だけ見える愛称
    note?: string;            // メモ
    addedAt: number;          // 追加日時
    lastMet?: number;         // 最後にすれ違った日時
}
```

- **フレンド/ブロック** は Nakama 標準 API
- **お気に入り/ミュート/愛称/メモ** は Storage API（自分のレコード）
- 一覧取得時に両者をマージしてクライアントに返す `getRelations` RPC を作る

## 4. UI 設計案

### フレンドパネル（メニュー → フレンド一覧）

```text
┌─────────────────────────────────────┐
│ フレンド一覧               [×]      │
├─────────────────────────────────────┤
│ [フレンド] [お気に入り] [申請] [BL]  │ ← タブ
├─────────────────────────────────────┤
│ ★ たろう (太郎ちゃん)                │
│   メイン部屋          [合流] [DM]   │
│ ─────────────────────────────────── │
│ ★ じろう                             │
│   公園広場            [合流] [DM]   │
│ ─────────────────────────────────── │
│ ○ さぶろう                           │
│   オフライン (3時間前)        [DM]   │
└─────────────────────────────────────┘
```

### 右クリック/長押しメニュー（プレイヤー or アバター）

```text
○○さん
├ プロフィール
├ お気に入りに追加  ☆
├ フレンド申請を送る
├ ──────────
├ ミュート
└ ブロック
```

### 部屋の入室時のフレンド権限チェック

- 部屋の Visibility が `Friends` → オーナーのフレンドのみ入室可
- Visibility が `FriendsPlus` → 中の誰かのフレンドなら入室可
- Visibility が `Public` → 誰でも

## 5. 段階的実装計画

### Phase 1: 最小フレンド（MVP）

- Nakama 標準 API でフレンド申請・承認・削除・ブロック
- フレンド一覧パネル（オンライン/オフライン表示）
- プレイヤー一覧から右クリック → フレンド申請
- ブロックリスト

### Phase 2: 部屋連携

- フレンドの**現在地表示**（どの部屋にいるか）
- フレンド一覧の「合流」ボタン
- 部屋の Visibility=Friends/Friends+ 連携

### Phase 3: お気に入り・ミュート・愛称

- お気に入り（Storage API）
- ミュート（チャット表示制御）
- 愛称・メモ機能

### Phase 4: ソーシャル機能

- フレンドのオンライン通知（押し付けがましくない範囲で）
- DM（Nakama channel chat の private channel）
- フレンド数ランキング・最近すれ違った人の自動サジェスト
- 「ピンポン」機能（招待制部屋へのノック）

## 6. 注意点・トレードオフ

1. **承認制 vs 自由フォロー** — Pigg 方式（双方向承認）を選んだのは「Pigg 立体版」という方向性に合わせるため。X 風にすると「アバターをフォローする」という感覚が薄くなる
2. **オンライン状態のプライバシー** — フレンドにすら現在地を見せたくない人向けに「オンライン状態を隠す」設定を Phase 2 で追加
3. **荒らし対策** — フレンド申請のスパム対策（1日最大N件、ブロックされたら申請不可）
4. **匿名性** — チャット主体 SNS なら Twitter のような気軽さも欲しい。お気に入り（一方向）でその気軽さを担保
5. **Nakama の制約** — フレンド数上限なし（実用上は数百で頭打ち）、双方向承認のみ標準サポート

## 7. Pigg と X のいいとこ取り

| 観点 | Pigg | X | tommChat 採用 |
| ---- | ---- | ---- | ---- |
| 関係構築 | 双方向承認 | 一方向 | **双方向（フレンド）** |
| 軽い興味 | お気に入り | フォロー/ブックマーク | **お気に入り（一方向、非通知）** |
| 嫌な人対策 | 無視 | ミュート＋ブロック | **ミュート＋ブロック** |
| 現在地共有 | あり | なし | **あり（フレンドのみ）** |
| 上限 | あり (100〜) | なし | **あり (200)** |
| 通知 | 申請時のみ | 申請なし | **申請時のみ** |
