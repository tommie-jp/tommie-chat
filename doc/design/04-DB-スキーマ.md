# DB: ストレージスキーマ

## 概要

Nakama は PostgreSQL の `public.storage` テーブルにデータを格納する。
主キーは `(collection, key, user_id)` の複合キー。
本アプリが使用するコレクションは 3 つ: `world_data`, `ccu_data`, `bookmarks`。

## コレクション一覧

| Collection | 用途 | レコード数の目安 |
| --- | --- | --- |
| `world_data` | ワールドメタ情報 + チャンクデータ | 1 + チャンク数 x ワールド数 |
| `ccu_data` | 同接履歴（1 分間隔） | 1 |
| `bookmarks` | ユーザーブックマーク | ユーザー数 |

## 1. ワールドメタ情報（worlds_meta）

| 項目 | 値 |
| --- | --- |
| Collection | `world_data` |
| Key | `worlds_meta` |
| UserID | `00000000-...`（システムユーザー） |
| PermissionRead | 2（公開） |
| PermissionWrite | 0（書込不可） |

### Value（JSON）

```json
{
  "worlds": [
    {
      "id": 0,
      "chunkCountX": 64,
      "chunkCountZ": 64,
      "name": "ワールド名",
      "ownerUid": "ユーザーID"
    }
  ],
  "nextId": 2
}
```

| フィールド | 型 | 説明 |
| --- | --- | --- |
| `worlds[].id` | `int` | ワールド ID |
| `worlds[].chunkCountX` | `int` | X 方向チャンク数（デフォルト 64） |
| `worlds[].chunkCountZ` | `int` | Z 方向チャンク数（デフォルト 64） |
| `worlds[].name` | `string` | ワールド名 |
| `worlds[].ownerUid` | `string` | オーナーの Nakama ユーザー ID |
| `nextId` | `int` | 次のワールド ID |

## 2. チャンクデータ（地形ブロック）

| 項目 | 値 |
| --- | --- |
| Collection | `world_data` |
| Key | `w{worldId}_chunk_{cx}_{cz}`（例: `w0_chunk_31_32`） |
| UserID | `00000000-...`（システムユーザー） |
| PermissionRead | 2（公開） |
| PermissionWrite | 1（オーナー書込） |

キー生成関数: `chunkStorageKey(worldId, cx, cz)` (main.go:794)

### チャンク Value（JSON）

```json
{
  "table": [1, 0, 255, 200, 100, 200, 0, 0, 0, 0, 0, 0, ...]
}
```

| フィールド | 型 | 説明 |
| --- | --- | --- |
| `table` | `int[]` | 1 チャンク分のブロックデータ（`CHUNK_SIZE * CHUNK_SIZE * 6` 要素） |

1 ブロックは 6 要素（バイト）:

| オフセット | 内容 | 範囲 |
| --- | --- | --- |
| 0 | BlockID 下位バイト | 0-255 |
| 1 | BlockID 上位バイト | 0-255 |
| 2 | R（赤） | 0-255 |
| 3 | G（緑） | 0-255 |
| 4 | B（青） | 0-255 |
| 5 | A（不透明度） | 0-255 |

BlockID = `table[i] | (table[i+1] << 8)`。BlockID=0 はブロックなし。

### 保存タイミング

- **定期保存**: MatchLoop 内で 60 秒ごとに `flushDirtyChunks()` でバッチ保存
- **シャットダウン時**: `MatchTerminate` で `flushDirtyChunks()` を呼び出し
- **ワールド削除時**: `deleteRoom` RPC で全チャンクを `StorageDelete`

## 3. 同接履歴（CCU）

| 項目 | 値 |
| --- | --- |
| Collection | `ccu_data` |
| Key | `history_1m` |
| UserID | `00000000-...`（システムユーザー） |
| PermissionRead | 0（非公開） |
| PermissionWrite | 0（非公開） |

### CCU Value（JSON）

```json
{
  "timestamps": [1741500060, 1741500120, 1741500180],
  "values":     [5, 3, 0]
}
```

| フィールド | 型 | 説明 |
| --- | --- | --- |
| `timestamps` | `int64[]` | 各サンプルの Unix 秒（1 分間隔） |
| `values` | `int[]` | その時点の同接数（1 分間の 1 秒サンプル平均値） |

最大 14,400 件（10 日分）。詳細は [04-DB-同接データ.md](04-DB-同接データ.md) 参照。

## 4. ユーザーブックマーク

| 項目 | 値 |
| --- | --- |
| Collection | `bookmarks` |
| Key | `list` |
| UserID | 各ユーザーの ID |
| PermissionRead | 1（オーナーのみ読取） |
| PermissionWrite | 1（オーナーのみ書込） |

### ブックマーク Value（JSON）

```json
{
  "items": [
    {
      "name": "お気に入りの場所",
      "x": 123.45,
      "z": -67.89,
      "ry": 1.57,
      "worldId": 0
    }
  ]
}
```

| フィールド | 型 | 説明 |
| --- | --- | --- |
| `items[].name` | `string` | ブックマーク名（1-20 文字、HTML エスケープ済み） |
| `items[].x` | `float64` | X 座標 |
| `items[].z` | `float64` | Z 座標 |
| `items[].ry` | `float64` | Y 軸回転 |
| `items[].worldId` | `int` | ワールド ID |

1 ユーザーあたり最大 50 件。

## レガシーフォーマット

旧バージョンでは以下のキー形式が使用されていた:

| 旧キー | 新キー | 備考 |
| --- | --- | --- |
| `ground_table` | `w0_chunk_{cx}_{cz}` | 100x100 単一テーブル → チャンク分割 |
| `chunk_{cx}_{cz}` | `w0_chunk_{cx}_{cz}` | ワールド ID プレフィックスなし |

`InitModule` 起動時に `ground_table` → チャンク形式への自動マイグレーションあり（main.go:2767-2843）。
`chunk_X_Z` → `w0_chunk_X_Z` の自動マイグレーションは未実装（手動 SQL で対応）。
