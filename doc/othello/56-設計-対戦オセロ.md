# 対戦オセロ設計メモ

## コンセプト

部屋の中でオセロ対戦ができる機能。チャットSNSのコンセプトに合わせ、
既存の部屋内に「オセロ卓」を置く形で実装する。

## 方針：パネル＋ブロックのハイブリッド

- **操作・判定はパネル（HTML）** で行い、対局者はパネルで打つ
- **ブロックにも同期反映** して、部屋にいる人が 3D 空間でも盤面を見られる

| 方式 | メリット | デメリット |
| --- | --- | --- |
| パネル主体 | タッチ／クリックで直感的、モバイル対応しやすい | 3D 空間との一体感は薄い |
| ブロック主体 | 没入感が高い、観戦が自然 | 操作しづらい、カメラ制御が必要 |
| **ハイブリッド** | 両方の利点を取れる | 実装量は増える |

## ゲームステート（インメモリ）

`othelloGames sync.Map` で管理。キー: `oth_<timestamp>_<seq>`

```go
type OthelloGame struct {
    GameID    string
    Board     [64]int8      // 0=空, 1=黒, 2=白
    BlackUID  string
    WhiteUID  string
    Turn      int8          // 1=黒, 2=白
    Status    string        // waiting / playing / finished
    Winner    int8          // 0=未定, 1=黒勝, 2=白勝, 3=引分
    LastMove  int           // 最後の手の index (-1=なし)
    WorldID   int           // 所属ワールド
    BoardGX   int           // ブロック盤面の左上ワールドX座標
    BoardGZ   int           // ブロック盤面の左上ワールドZ座標
    PrevBoard [64]int8      // 前回盤面（ブロック差分検出用、-1 初期化）
}
```

## 対戦フロー

シーケンス図: [56-seq-othello.puml](56-seq-othello.puml)

```text
プレイヤーA: othelloCreate RPC → gameId 取得、パネルにゲーム画面表示（待機中）
             → 3D: 10x10 ブロック配置（8x8 盤面 + 外枠）
プレイヤーB: othelloList → 一覧表示 → othelloJoin RPC → 対戦開始
             → 両者に OP_OTHELLO_UPDATE 配信
毎ターン:   othelloMove RPC(gameId, row, col)
             → サーバ判定 → 盤面更新 → OP_OTHELLO_UPDATE 配信
             → OP_BLOCK_UPDATE でブロック差分配信
投了:       othelloResign RPC → status=finished → 結果配信
             → 60秒後にブロッククリーンアップ（goroutine）
終了:       パス2連続 or 盤面埋まり → status=finished → 結果表示
```

## Pub/Sub モデル

```text
クライアント → OP_OTHELLO_SUB (19) → サーバ
  payload: {"sub": true/false}
  - パネル表示時に subscribe、非表示時に unsubscribe
  - MutationObserver でパネルの style.display を監視

サーバ → OP_OTHELLO_UPDATE (18) → 購読者
  payload: OthelloUpdatePayload (board, turn, status, etc.)
  - MatchSignal → MatchLoop で購読者のみに配信
```

## RPC 一覧

| RPC 名 | 引数 | 戻り値 | 説明 | 副作用 |
| --- | --- | --- | --- | --- |
| `othelloCreate` | `{worldId}` | `OthelloUpdatePayload` | 卓作成（status=waiting） | 木枠+盤面ブロック配置 |
| `othelloJoin` | `{gameId}` | `OthelloUpdatePayload` | 対戦参加 → status=playing | 購読者に OP_OTHELLO_UPDATE 配信 |
| `othelloMove` | `{gameId, row, col}` | `OthelloUpdatePayload` | 着手 → 合法手判定・裏返し | 購読者に OP_OTHELLO_UPDATE + ブロック差分配信 |
| `othelloResign` | `{gameId}` | `OthelloUpdatePayload` | 投了 → status=finished | 60秒後にブロッククリーンアップ |
| `othelloList` | `{worldId}` | `{games: [...]}` | ワールド内の未終了ゲーム一覧 | なし |

## WebSocket Opcode

| Opcode | 名前 | 方向 | ペイロード | 説明 |
| --- | --- | --- | --- | --- |
| 18 | `OP_OTHELLO_UPDATE` | S→C | `OthelloUpdatePayload` | 盤面更新通知。購読者のみに配信 |
| 19 | `OP_OTHELLO_SUB` | C→S | `{subscribe: true/false}` | 購読登録/解除 |
| 4 | `OP_BLOCK_UPDATE` | S→C | `[{gx,gz,blockId,r,g,b,a}...]` | 3Dブロック差分。AOI内プレイヤーに配信 |

## 内部シグナル（MatchSignal 経由）

| 関数 | signal type | タイミング | 内容 |
| --- | --- | --- | --- |
| `othelloSignalBroadcast` | `othelloUpdate` | join / move / resign | 購読者に OP_OTHELLO_UPDATE を配信 |
| `othelloBlockSignal` | `othelloBlocks` | create / move | 盤面の差分ブロックを AOI 内に配信 |
| `othelloFrameSignal` | `othelloBlocks` | create | 外枠36ブロック（木目色）を配置 |
| `othelloClearBlocks` | ―（直接チャンク操作） | move(終局) / resign 後60秒 | 10x10エリアを blockId=0 でクリア |

## 3D ブロック盤面

### 盤面レイアウト（10x10 = 100 ブロック）

```text
BoardGX-1 ～ BoardGX+8, BoardGZ-1 ～ BoardGZ+8

  木枠 木枠 木枠 木枠 木枠 木枠 木枠 木枠 木枠 木枠
  木枠 盤面 盤面 盤面 盤面 盤面 盤面 盤面 盤面 木枠
  木枠 盤面 盤面 盤面 盤面 盤面 盤面 盤面 盤面 木枠
  ...
  木枠 木枠 木枠 木枠 木枠 木枠 木枠 木枠 木枠 木枠
```

### ブロック色

| 要素 | RGB | 説明 |
| --- | --- | --- |
| 空マス | (0, 100, 0) | 深緑（地面グリッドより暗い） |
| 黒石 | (15, 15, 15) | ほぼ黒 |
| 白石 | (250, 250, 250) | ほぼ白 |
| 木枠 | (60, 35, 10) | 木目色 |

### ブロック同期の仕組み

1. `othelloBlockSignal()` — `Board` vs `PrevBoard` の差分を検出
2. 差分をJSON化して `nk.MatchSignal()` で送信
3. MatchLoop の SignalHandler が受信 → チャンク保存 + AOI内ブロードキャスト
4. クライアントの `onBlockUpdate` が `placeBlock()` で thin instance 更新

### クリーンアップ

- 投了/終了時に 60 秒タイマーの goroutine を起動
- `othelloClearBlocks()` が 10x10 エリアを `blockId=0` でクリア
- クライアントは次回 `syncAOIChunks` でハッシュ不一致を検出 → ブロック除去

## サーバ側ロジック（Go）

### 合法手判定

- 8 方向（上下左右 + 斜め 4 方向）に探索
- 相手の石を挟める場所だけが合法手
- 合法手がない場合は自動パス（`othelloAutoPass`）
- パス 2 連続で終局

## 時間ルール（実装方針）

### 一般的なオセロの時間ルール（参考）

| 方式 | 典型値 | 採用例 |
| --- | --- | --- |
| 持ち時間制（切れ負け） | 1ゲーム 5〜30分 | 世界選手権 30分 / 全日本 25-30分 / Othello Quest 3-10分 |
| フィッシャー方式 | 初期 5分 + 1手ごとに +3秒 | チェス系。切れづらい |
| 1手制限制 | 1手 10〜30秒 | オンライン即戦 / 実装が容易 |

### tommieChat の採用方針

カジュアルチャットSNSという特性上、**1手 30秒制** をベースに段階実装する。

| 段階 | 内容 | 状態 |
| --- | --- | --- |
| 1 | クライアント側でタイマー表示のみ（時間切れでも何も起きない） | 実装済み |
| 2 | クライアント側で時間切れ検出 → 自動投了（`rpcOthelloResign`） | 未 |
| 3 | サーバ側で権威的タイマー（切断・チート対策） | 未 |

### 採用理由

- **30秒** は世界標準で直感的
- **カジュアル性**: 60秒以上は観戦/対戦相手の離脱リスクが上がる
- **段階実装**: 段階1は UI のみでサーバ変更不要。UX 検証してから段階2/3へ
- **将来拡張**: フィッシャー方式や持ち時間制は必要になってから追加

### 段階1 の表示仕様（チェスクロック風）

SVG 円形プログレスリングを **黒/白 2つ並べて** 表示するチェスクロック風を採用。
アクティブ側（現在のターンの色）のみがカウントダウンし、非アクティブ側は 30 のまま 0.35 にディム。

- **2つの SVG 円形リング**: 黒用・白用を横並び（`display: flex; gap: 24px;`）
- **アクティブ側**: `stroke-dasharray` で 30 秒で 1 周減る / `opacity: 1`
- **非アクティブ側**: リング満タン・数字 30 固定 / `opacity: 0.35` にディム
- **中央に大きな数字**: 残り秒数をカウントダウン
- **下に石アイコン**: ⚫ / ⚪ で「どちらのクロックか」を示す
- **YOU 表示**: 自分の色のクロックに「YOU」を赤で付記（`.oth-clock.you .oth-timer-label::after`）
- **色遷移**（アクティブ側の残り秒数に応じて）:
  - 15 秒超: 緑（#3bb273）
  - 10-15 秒: 黄（#f0c541）
  - 5-10 秒: 橙（#f28c28）
  - 5 秒以下: 赤（#d93025）＋パルスアニメ（`.oth-clock.warning`）
- **時間切れでも強制アクションは行わない**（段階1は表示のみ）

### 段階1 で採用しなかった案（参考）

| 案 | 概要 | 不採用理由 |
| --- | --- | --- |
| A: 横バー | `div width` でアニメ＋絵文字 | カジュアル感はあるが商用クオリティ感が弱い |
| 単一リング | 現在ターンのクロックのみ表示 | 相手側の存在感がない / 初期採用したがチェスクロックに変更 |

## 既知の制限・今後の検討

- **盤面位置**: 現在 (504, 504) に固定。複数同時対戦時に盤面が重なる
- **切断対応**: タイムアウト投了は未実装（時間ルール段階2/3 で対応）
- **観戦モード**: pub/sub で盤面は受信できるが、観戦専用 UI は未実装
- **対戦招待**: チャットコマンド `/othello @名前` は未実装
- **インメモリ**: サーバ再起動でゲーム状態が消える（永続化なし）
- **ブロッククリーンアップ（サーバ再起動耐性あり）**:
  通常運用では終局/投了後の 60 秒遅延 goroutine が
  `othelloClearBlocks` → `MatchSignal("othelloBlocks", BlockID:0)`
  を発行し、接続中プレイヤーへの `opBlockUpdate` ブロードキャスト＋サーバチャンク更新
  が行われる。再接続プレイヤーには `markChunkDirty` → `flushDirtyChunks` 経由で
  DBに保存された空チャンクがハッシュ同期で配信される。
  **シャットダウン対策**: `docker compose down` 等で `stop_grace_period`（10秒）
  内にプロセスが停止する場合、60秒遅延 goroutine は完走できないが、
  `worldMatch.MatchTerminate` 内で未クリーンな `othelloGames` を全て iterate し、
  `othelloClearBlocksDirect` で **MatchSignal を経由せず直接チャンクを書き換え**、
  その後の `flushDirtyChunks` で DB に永続化する。
  これにより再起動後の再接続時にハッシュ同期で空チャンクが配信される。
  ※ `othelloClearBlocksDirect` は match loop 停止中に呼ばれるため
  クライアントへのリアルタイム通知は行わない（全員切断される前提）。
