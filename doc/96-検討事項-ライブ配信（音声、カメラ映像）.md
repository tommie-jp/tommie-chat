# 検討事項: ライブ配信（音声、カメラ映像）

リアルタイムの音声・カメラ映像をアバターに反映する機能の設計メモ。

---

## 用途

| 種類 | 内容 |
| ---- | ---- |
| 音声配信 | エモーション効果音、ボイス付きリアクション |
| カメラ映像配信 | カメラ映像を板ポリテクスチャとして貼付（「動くアバター」） |

---

### ライブ音声

| タイトル | 音声 | 方式 |
| ---- | ---- | ---- |
| VRChat | ◎ 全員ボイチャ | 距離減衰あり（近い人の声が大きい） |
| Second Life | ◎ ボイチャ | 距離減衰あり |
| Fortnite | ◎ パーティ内 | パーティ単位 |
| World of Warcraft | ○ パーティ/ギルド | チャンネル単位 |

### ライブカメラ映像

| タイトル | カメラ映像 | 用途 |
| ---- | ---- | ---- |
| Gather Town | ◎ | 近づくとカメラ映像が見える（メイン機能） |
| Roblox | ○ | 表情トラッキング → アバターの顔に反映 |
| Spatial | ○ | ビデオ通話的に顔を表示 |
| VRChat | × | VR ヘッドセットのトラッキングで代替 |
| Cluster | × | — |

### tommChat に近いモデル

**Gather Town** が最も近い:
- 2D のバーチャルオフィス/SNS
- アバターに近づくと自動でカメラ映像・音声が繋がる
- 距離が離れると自動で切断
- ゲームではなくコミュニケーションが主目的

**Roblox の表情トラッキング** も参考になる:
- カメラで顔を撮影 → 表情をリアルタイムでアバターに反映
- 映像そのものは送らず表情パラメータだけ送信（帯域極小）

### アプローチ比較

| アプローチ | 帯域 | 没入感 | 実装難易度 |
| ---- | ---- | ---- | ---- |
| Gather Town 方式（カメラ映像そのまま） | 大 | 中 | 低 |
| Roblox 方式（表情トラッキング） | 極小 | 高 | 高 |
| VRChat 方式（音声のみ距離減衰） | 小 | 高 | 低 |

tommChat には **VRChat 方式の距離減衰ボイチャ + Gather Town 方式のカメラ映像**
（配信モード ON の人のみ）の組み合わせが合う。

## サーバ選定

| サーバ | プロトコル | 遅延 | ライセンス | おすすめ度 |
| ---- | ---- | ---- | ---- | ---- |
| LiveKit | WebRTC | 〜100ms | Apache 2.0 | ◎ |
| Mediasoup | WebRTC | 〜100ms | ISC | ○ |
| Janus Gateway | WebRTC | 〜100ms | GPL-3 | ○ |
| SRS | WebRTC / RTMP / HLS | 100ms〜数秒 | MIT | ○ |
| MediaMTX | WebRTC / RTMP / HLS | 100ms〜数秒 | MIT | △ |
| Nginx-RTMP | RTMP / HLS | 5〜30秒 | BSD-2 | × |

**LiveKit を推奨。** 1つのサーバで音声・映像の両方をカバーでき、Docker 1コンテナで動作する。

---

## SFU 方式の動作

LiveKit は SFU（Selective Forwarding Unit）方式。
サーバでミキシングせず、個別ストリームをそのまま転送する。

```
配信者A → LiveKit → そのまま転送 → 受信者
配信者B → LiveKit → そのまま転送 → 受信者
配信者C → LiveKit → そのまま転送 → 受信者
```

受信者はブラウザ側で個別の音声/映像ストリームを同時再生する。

| 方式 | 処理場所 | サーバ CPU | 遅延 |
| ---- | ---- | ---- | ---- |
| SFU（LiveKit） | ブラウザで個別再生 | 低 | 低 |
| MCU | サーバでミックスして1本配信 | 高 | 高 |

## 構成

```
配信者ブラウザ
  → getUserMedia() でカメラ/マイク取得
  → LiveKit に映像/音声トラック送信
  → サーバが AOI 内のプレイヤーにのみ配信
  → 受信側:
      音声: Web Audio API で再生（距離で音量減衰）
      映像: VideoTrack → Canvas → DynamicTexture → 板ポリに貼付
```

### Nakama 連携

```
クライアント → Nakama で認証 → Nakama が LiveKit 用 JWT 発行
  → クライアントが JWT で LiveKit に接続
```

### Docker Compose 追加例

```yaml
livekit:
  image: livekit/livekit-server:latest
  ports:
    - "7880:7880"   # HTTP API
    - "7881:7881"   # WebRTC TCP
  environment:
    - LIVEKIT_KEYS=devkey:secret
```

## さくらVPS 2G でのキャパシティ

スペック: 3 vCPU / 2GB RAM / 100Mbps

### 音声のみ

| 条件 | 値 |
| ---- | ---- |
| 1人あたりビットレート | 〜32 kbps (Opus) |
| 配信者10人 × 各20人視聴 | 6.4 Mbps |
| 100Mbps での上限 | 配信者50人 × 各50人視聴 程度 |

音声のみなら十分余裕がある。

### 映像あり

| 解像度 | ビットレート | 配信者1人 → 視聴20人 | 100Mbps での配信者上限 |
| ---- | ---- | ---- | ---- |
| 160x120（サムネ） | 100 kbps | 2 Mbps | 〜50人 |
| 320x240（低画質） | 300 kbps | 6 Mbps | 〜16人 |
| 640x480（中画質） | 800 kbps | 16 Mbps | 〜6人 |
| 1280x720（HD） | 2,500 kbps | 50 Mbps | 〜2人 |

板ポリテクスチャには 160x120〜320x240 で十分。

### リソース

| リソース | LiveKit 消費 | 2G VPS | 判定 |
| ---- | ---- | ---- | ---- |
| RAM | 〜200MB | 2GB（Nakama 等と共用） | やや厳しい |
| CPU | 映像10配信で 0.5〜1 vCPU | 3 vCPU | OK |

### 現実的な構成

| プラン | 同時映像配信者数 | 備考 |
| ---- | ---- | ---- |
| 2G VPS 1台（現状） | 5〜10人 | 低解像度限定、Nakama と同居 |
| 2G + 1G VPS 追加 | 〜20人 | LiveKit を別サーバに分離 |
| 4G VPS 1台 | 〜20人 | 全部同居 |

## 受信者側の負荷

AOI 内の配信者が多いとブラウザでの同時デコード負荷が上がる。

| 同時受信数 | ブラウザ負荷 |
| ---- | ---- |
| 〜10本 | 問題なし |
| 〜30本 | やや重い |
| 50本〜 | スマホだと厳しい |

### 対策

| 対策 | 内容 |
| ---- | ---- |
| 距離で購読制限 | 近い人の音声/映像のみ受信（例: 半径5m以内の上位10人） |
| 距離で音量減衰 | Web Audio API の PannerNode で自然に聞こえなくなる |
| Simulcast | 近い人=高画質、遠い人=低画質を自動切替（LiveKit 標準機能） |
| 配信モード | 全員が常時配信ではなく、希望者のみ ON にする |

## 懸念事項

| 課題 | 対策 |
| ---- | ---- |
| プライバシー | カメラ映像は個人情報。録画禁止の明示、利用規約が必要 |
| 不適切映像 | 通報機能必須。モデレーターによるキック/BAN |
| サーバコスト | 映像は音声の10〜50倍の帯域。利用状況を見て VPS 増強 |
| 通信事業者届出 | 1対1ボイスメッセージは届出が必要な可能性あり。1対多の配信はコンテンツ配信に該当し不要と考えられる |

## 実装の優先順

| 段階 | 内容 |
| ---- | ---- |
| Phase 1 | LiveKit を Docker Compose に追加、Nakama から JWT 発行 |
| Phase 2 | 音声配信（エモーション効果音の AOI 内リアルタイム再生） |
| Phase 3 | カメラ映像配信（配信モード ON → 板ポリに DynamicTexture） |
| Phase 4 | Simulcast、距離制御、購読制限の最適化 |

Phase 1〜2 は音声のみなので 2G VPS で十分動作する。
Phase 3 の映像配信は同時配信者 5〜10人の範囲で開始する。
