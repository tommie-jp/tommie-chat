# 23-メモリ削減計画

2026-04-25

VPS のメモリ消費を抑えるための段階的計画メモ。
コンテナ統合（5→1）は overhead 30〜45 MB しか削減できず、運用リスクが大きい
ため不採用とし、不要サービスの停止と各サービスのチューニングで段階的に削減
していく方針。

## 現状の構成と実測メモリ (2026-04-25)

VPS sakura-vps2 (RAM 1.9 GB / Swap 2 GB) で `docker stats --no-stream` 実測値:

| サービス | prod 実測 | test 実測 | 必須性 | 備考 |
| ---- | ---: | ---: | ---- | ---- |
| postgres | 13.5 MB | 31.8 MB | 必須 | 軽量 |
| nakama | 32.1 MB | 23.7 MB | 必須 | Go 効率良 |
| web (nginx) | 3.0 MB | 4.5 MB | 必須 | 極軽量 |
| prometheus | 40.9 MB | (なし) | 任意 | Tier 1 A 対象 |
| minio | 84.2 MB | 112.6 MB | UGC 用 | わずか 28 KB のために 84-112 MB 使用 |
| **小計** | **173.7 MB** | **172.6 MB** | | 両環境合計 ~346 MB |

ホスト全体: `free -h` で 741 MB used / 1.9 GB total。コンテナ外
(host OS + Docker daemon + cache) で ~395 MB。

## なぜコンテナ統合しないか

- Heroic Labs 公式 Nakama image は単独プロセス前提の entrypoint を持つ
- 5→1 統合には s6-overlay / supervisord 採用の自前 image 構築が必要
- 削減できるのは **コンテナ overhead 30〜45 MB のみ**
- 失うもの: 部分再起動の柔軟性、ログ分離、observability、本家更新追従の容易さ

ROI が悪く、運用リスクの方が大きいので採用しない。

## 削減計画 (優先度順)

### Tier 1: ノーリスクで効果大

#### A. 本番から Prometheus を停止 (実測 ~41 MB 削減) ✅ 実施

`docker-compose.prod.yml` で `prometheus` サービスに `profiles: ["with-metrics"]`
を付けて、デフォルト起動から外す。
必要時は `docker compose --profile with-metrics up -d prometheus` で起動。

Nakama 単体の `/metrics` (port 9100) は維持されるので、外部から `curl
http://localhost:9100/metrics` で必要時にメトリクス取得可能。

#### B. MinIO 利用状況確認 → 当面維持 (2026-04-25 判断)

実測:

```text
prod  /data/avatars/    28 KB (UGC ユーザアップロード)
prod  /data/assets/     4 KB
prod  /data/uploads/    4 KB
test  /data/avatars/    28 KB
```

合計 36 KB のために MinIO が prod 84 MB / test 112 MB 食っている。
容量効率は極端に悪いが、`/s3/avatars/<NNN-xxx>.png` (UGC 投稿アバター) を
[nakama/doDeploy.sh:491](../nakama/doDeploy.sh#L491) の nginx で
`proxy_pass http://minio:9000/avatars/` 経由で配信中なので、削除すると
既存ユーザのカスタムアバターが壊れる。

将来 MinIO 廃止する場合の移行先候補:

1. ホスト bind mount + nginx 静的配信（実装中程度）
2. Nakama Storage + base64 エンコード（実装小、性能悪）

**当面は維持。Tier 3 として将来課題に置く。**

#### C. test 環境 (mmo-test.tommie.jp) を平時停止 (実測 ~173 MB 削減) — 未実施

ステージング検証時のみ起動する運用に変更:

```bash
# 停止
docker compose -f docker-compose.yml -f docker-compose.dev.yml -p tommchat-test down

# 検証時のみ起動
docker compose -f docker-compose.yml -f docker-compose.dev.yml -p tommchat-test up -d
```

[CLAUDE.md](../CLAUDE.md) §ステージング検証 の「本番デプロイ前に検証」運用と整合。

### Tier 2: チューニング (推定合計 50〜150 MB 削減)

#### C. PostgreSQL の shared_buffers 等を絞る

dev/小規模本番ならデフォルト 128 MB は過剰:

```yaml
postgres:
  command: postgres -c shared_buffers=32MB -c work_mem=4MB -c max_connections=50
```

→ 同接 100 規模なら十分。

#### D. Nakama のバッファサイズ調整

`docker-compose.yml` の Nakama entrypoint パラメータ:

- `--match.input_queue_size 8192` → 1024〜2048
- `--socket.outgoing_queue_size 2048` → 512〜1024
- `--runtime.max_count 8192` → 1024

50〜100 MB 効くケースあり。

#### E. Nginx worker チューニング

```nginx
worker_processes 1;
events { worker_connections 256; }
```

同接 100 規模なら十分。

### Tier 3: 構成変更 (推定 ~200 MB 削減、リスク有り)

#### F. ホスト OS の Nginx を活用

本番で既にホスト nginx が TLS 終端していれば、Docker 側の web コンテナを
廃止し、ホスト nginx → `proxy_pass http://127.0.0.1:7350` 直接。
コンテナ 1 個削減。

ただしデプロイスクリプト (`doDeploy.sh`) のインフラ前提が変わるので
本番固有のリスクがあり、最後に検討。

## 実施履歴

### 2026-04-25 Prometheus 本番停止

`docker-compose.prod.yml` の prometheus に `profiles: ["with-metrics"]` を追加し、
deploy。実測:

```text
prod 全コンテナ合計
  Before: 173.7 MB (5 コンテナ)
  After:  113.7 MB (4 コンテナ)
  削減:    -60 MB (Prometheus -41 MB + Nakama 起動直後 -18 MB + MinIO -12 MB の合計)
```

ホスト全体 (`free`) の変化:

| 指標 | Before | After | 差分 |
| ---- | ----: | ----: | ----: |
| Mem used | 741 MB | 743 MB | ~ 同じ |
| buff/cache | ~395 MB | **1.2 GB** | +830 MB（拡張） |
| **Swap used** | **357 MB** | **111 MB** | **-246 MB** ✅ |
| プロセス + Swap | 1098 MB | 854 MB | **-244 MB** |

「used が同じなのに改善した」のは、Linux が swap 退避していた 246 MB を物理 RAM
に戻し、空きを buff/cache に振り分けた結果。Swap I/O 削減で体感速度も向上する。

prometheus 再起動が要るときは:

```bash
docker compose -f docker-compose.yml -f docker-compose.prod.yml \
  --profile with-metrics up -d prometheus
```

## メモリ削減を検討すべき目安 (2 GB VPS の場合)

### 平常時の健全ライン

```text
Mem available:  > 600 MB    (30%+ 余裕)
Swap used:      < 100 MB    (5% 未満)
ロード平均:      < 1.0
```

この範囲で運用できているなら追加対策不要。

### 注意ライン（Tier 1 を進めるサイン）

```text
Mem available:  300〜600 MB  (15〜30%)
Swap used:      100〜400 MB  (5〜20%)、上昇傾向
レスポンス:      時々遅延を感じる
```

このゾーンに入ったら次のチェックを行う:

- `docker stats --no-stream` で各コンテナの肥大化チェック
- `vmstat 1 10` で si (swap-in) / so (swap-out) の頻度を確認
- 不要サービス（test 環境、Prometheus 等）を止めて様子見

### 危険ライン（Tier 2 着手）

```text
Mem available:  100〜300 MB  (5〜15%)
Swap used:      400〜800 MB  (20〜40%)、継続的に上昇
レスポンス:      明らかに遅い、API タイムアウト
ロード平均:      > 2.0
```

ここまで来たら:

- PostgreSQL の shared_buffers / Nakama バッファサイズを調整 (Tier 2)
- test 環境を完全停止
- 同時起動コンテナを 1 環境のみに制約

### 緊急ライン（即対応）

```text
Mem available:  < 100 MB
Swap used:      > 800 MB (40%+)
OOM Killer 発動 (dmesg | grep -i "killed process" でログ)
```

OOM 寸前。即座に:

- 不要サービスを `docker stop`
- VPS スペックアップを検討 (2 GB → 4 GB)
- Tier 3（ホスト nginx 統合・MinIO 廃止）を本気で実施

## 監視の運用提案

### 簡易チェック (週 1 回)

```bash
# free と docker stats を同時確認
echo "=== free ==="
free -h
echo "=== docker stats ==="
docker stats --no-stream --format "table {{.Name}}\t{{.MemUsage}}\t{{.MemPerc}}"
echo "=== swap activity (10秒) ==="
vmstat 1 10
```

`vmstat` の si/so が 0 のままなら健全。継続的に > 0 なら swap I/O 多発。

### 自動アラート (任意)

systemd timer or cron で 5 分毎に `free` を読み、`available` が 200 MB を切ったら
通知:

```bash
# /etc/cron.d/mem-alert (例)
*/5 * * * * tommie /usr/local/bin/check-mem.sh
```

```bash
# /usr/local/bin/check-mem.sh
#!/bin/bash
AVAIL=$(free -m | awk '/^Mem:/ {print $7}')
if [ "$AVAIL" -lt 200 ]; then
    echo "[$(date)] LOW MEM: available=${AVAIL}M" >> /var/log/mem-alert.log
    # ここで Slack/メール通知を入れてもよい
fi
```

## 容量見積り (将来の規模拡大時)

| 同接ユーザ数 | 推定 Nakama RSS | postgres RSS | VPS 推奨 |
| ----: | ----: | ----: | ---- |
| 〜100 | 30〜60 MB | 30〜80 MB | 2 GB (現状で十分) |
| 100〜500 | 100〜200 MB | 80〜200 MB | 2〜4 GB |
| 500〜1000 | 200〜400 MB | 200〜400 MB | 4 GB |

時は 4 GB 以上にスペックアップが必要。
