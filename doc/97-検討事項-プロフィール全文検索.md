# 検討事項: プロフィール全文検索

ユーザープロフィール本文（自己紹介・興味タグ等）を日本語キーワードで検索する機能の設計メモ。

例: 「猫」で検索すると、自己紹介に「猫が好き」と書いたユーザーや `interests: ["猫"]` を持つユーザーがヒットする。

---

## 現状

現在 Nakama に保存されている「プロフィール」はランタイム情報のみ:

- [nakama/go_src/main.go:2124-2132](../nakama/go_src/main.go#L2124-L2132) `profileEntry` = `DisplayName` / `TextureUrl` / `CharCol` / `CharRow` / `LoginTime` / `NameColor`
- 自由記述の `bio` や興味タグは未実装

そのため、まず **プロフィール本文の保存場所と更新フロー** を設計する必要がある。

---

## 1. プロフィール保存場所

| 方式 | 場所 | メリット | デメリット |
| ---- | ---- | ---- | ---- |
| A. Nakama `users.metadata` | 既存 `users` テーブルの JSONB | Nakama API で完結、マイグレ不要 | 他用途と混在しやすい |
| B. `storage` コレクション | `collection=profile`, `key=self` | 既存パターン踏襲（`bookmarks` と同じ） | FTS は `value` JSONB を走査 |
| C. 専用テーブル | `user_profile` を新設 | インデックス・カラム設計が自由 | マイグレーション必要、運用負担 |

推奨は **B**:

- `bookmarks` と同じ作法で済む（[doc/04-DB-スキーマ.md](04-DB-スキーマ.md) 参照）
- `PermissionRead=2`（公開）にすればクライアントから直接読める
- PGroonga インデックスは `WHERE collection='profile'` でスコープ限定できる

---

## 2. プロフィール項目（初期案）

```json
{
  "bio": "猫が好きな新宿在住エンジニアです",
  "interests": ["猫", "ゲーム", "Babylon.js"],
  "location": "東京",
  "twitter": "@xxx",
  "updatedAt": "2026-04-14T12:34:56Z"
}
```

| フィールド | 型 | 上限 | 備考 |
| ---- | ---- | ---- | ---- |
| `bio` | `string` | 200 文字 | 自由記述。改行可 |
| `interests` | `string[]` | 10 個 × 20 文字 | タグ |
| `location` | `string` | 30 文字 | 任意 |
| `twitter` | `string` | 30 文字 | `@` 付き。URL ではなく ID |
| `updatedAt` | `string` | — | 更新日時（ISO 8601） |

- HTML エスケープ必須（`bookmarks` 同様）
- 不適切ワードは初期は通報ベース、規模拡大後に自動フィルタ

---

## 3. 日本語 FTS の方式

`storage.value`（JSONB）を SQL で全文検索する。日本語対応で現実的な選択肢:

| 拡張 | 強み | 弱み | 導入コスト |
| ---- | ---- | ---- | ---- |
| **PGroonga** | 日本語形態素解析、表記ゆれに強い、LIKE 互換 | PostgreSQL 拡張イメージ差し替え | 中 |
| **pg_bigm** | bigram で軽量、LIKE 高速化 | 語境界なし（「京都」で「東京都」もヒット） | 低 |
| `to_tsvector('simple', ...)` | 拡張不要 | 日本語の語境界が取れず精度低 | 小（ただし実用性低） |

推奨は **PGroonga**:

- 商用 1000 人規模・日本語前提では精度が重要
- `nakama/docker-compose.yml` の `postgres:` サービスを `groonga/pgroonga:*` 系イメージに差し替えるだけ
- LIKE 演算子互換なので移行ハードル低い

### PGroonga 導入手順（概要）

1. `docker-compose.yml` の postgres イメージを `groonga/pgroonga:3.x-alpine-17` 系に変更
2. `InitModule` で `CREATE EXTENSION IF NOT EXISTS pgroonga`
3. インデックス作成（下記参照）
4. 検索 RPC 追加

---

## 4. 実装イメージ

### マイグレーション（`InitModule` 内で 1 回実行）

```sql
CREATE EXTENSION IF NOT EXISTS pgroonga;
CREATE INDEX IF NOT EXISTS idx_profile_pgroonga
  ON storage USING pgroonga ((value::text))
  WHERE collection = 'profile';
```

### RPC: `setProfile` / `getProfile`

- `setProfile`: 入力を検証（文字数・HTML エスケープ）して `storage` に保存
- `getProfile`: 指定 `user_id` のプロフィールを返す

### RPC: `searchProfiles`

```go
rows, _ := db.QueryContext(ctx, `
  SELECT user_id, value FROM storage
  WHERE collection = 'profile' AND value::text &@~ $1
  LIMIT 50
`, keyword)
```

- `&@~` = PGroonga の全文検索演算子（表記ゆれ対応）
- クライアントから `socket.rpc("searchProfiles", "猫")` で呼ぶ
- レート制限: 1 ユーザー 10 回/分 程度

---

## 5. UI

### プロフィール編集パネル

- メニューに「プロフィール編集」項目を追加
- `bio` のテキストエリア + `interests` のタグ入力 + `location` / `twitter`
- 保存ボタンで `setProfile` RPC 呼び出し

### プロフィール検索パネル

- メニューに「プロフィール検索」項目を追加
- 検索バー + 結果リスト（アバター画像・表示名・`bio` ハイライト）
- クリックでそのユーザーの詳細を表示、または世界内のその人の位置にテレポート（同じワールドにいる場合）

---

## 6. 段階的実装の順序

1. プロフィール編集 UI をメニューに追加（`bio` テキストエリアのみでまず開始）
2. RPC `setProfile` / `getProfile` を作って `storage` に保存
3. PGroonga を `docker-compose.dev.yml` で先に試す（dev 環境のみ）
4. RPC `searchProfiles` を追加、検索 UI を実装
5. ステージング（`mmo-test.tommie.jp`）で検証 → 本番デプロイ

---

## 7. 要検討事項

- **プライバシー**: プロフィール公開範囲（全公開 / フレンドのみ / 非公開）を選択可能にするか
- **検索除外**: 「検索結果に表示しない」設定を用意するか
- **ブロック機能**: ブロックしたユーザーを検索結果から除外する仕組み（[52-設計-フレンドシステム.md](52-設計-フレンドシステム.md) と連携）
- **多言語**: 英語ユーザー向けに `en_bio` フィールドを用意するか、それとも 1 フィールドで言語自動判定か
- **インデックス更新負荷**: プロフィール更新頻度が低ければ PGroonga の書き込みコストは問題にならないはず（要計測）
