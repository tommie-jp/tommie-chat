## 1. デプロイテスト（WSL2 / 開発環境）

本番実行前に、別ディレクトリでデプロイ〜テストまで一括確認できます。
`doDeployTest.sh` は git clone → doDeploy.sh → doAll.sh（疎通テスト + 全テスト）を一括実行します。
**開発環境（WSL2 Ubuntu 24.04）** で実行してください。

```bash
# 作業ディレクトリを作成
mkdir -p ~/deploy-test && cd ~/deploy-test

# doDeployTest.sh を取得して実行
curl -fsSL https://raw.githubusercontent.com/open-tommie/tommie-chat/main/test/doDeployTest.sh -o doDeployTest.sh
chmod +x doDeployTest.sh
./doDeployTest.sh
```

2回目以降は `./doDeployTest.sh` だけで再実行できます（既存ディレクトリの削除を確認後、再クローンします）。

## 2. 自動スクリプト

手動手順の代わりに、以下のスクリプトで一括セットアップできます。
`doDeploy.sh` は **VPS 上で** 実行します（SSH でログイン後）。
フロントエンドのビルドと転送は **開発環境（WSL2 Ubuntu 24.04）で** 事前に行います。

```bash
# --- 開発環境（WSL2 Ubuntu 24.04）で実行 ---
npm run build
rsync -avz --delete dist/ <deploy_user>@<VPS_IP>:~/tommie-chat/dist/

# --- VPS（SSH ログイン後）で実行 ---
# VPS デプロイ（セクション 4〜6 を自動化）
./nakama/doDeploy.sh

# HTTPS 設定（セクション 7 を自動化）※未テスト
./nakama/doSetupHTTPS.sh <ドメイン名>
```

> SSH 鍵認証の設定（3.1）は手動で行ってください。

### 2.1 doDeploy.sh の処理内容

| ステップ | 内容 | スキップ条件 |
| -------- | ---- | ----------- |
| 0. 既存コンテナ停止 | `docker compose down -v` でコンテナ・ボリューム削除 | なし（毎回実行） |
| 1. ファイアウォール | ufw で 22/80/443 のみ許可 | ufw が無い、または既に active |
| 2. スワップ | 2GB スワップ作成 | メモリ 2GB 超、または /swapfile 既存 |
| 3. Docker | Docker Engine + Compose インストール | `docker` コマンドが既に存在 |
| 4. (予約) | — | — |
| 5. 環境変数 | `.env` 生成（PG・MinIO・console パスワード自動生成、server_key は `tommie-chat` 固定） | なし（毎回再生成） |
| 6. nginx.conf | 本番用 nginx.conf 生成（静的配信 + `/s3/` MinIO プロキシ + `/v2/` `/ws` Nakama プロキシ） | なし（毎回上書き） |
| 7. フロントエンド配置 | 開発環境でビルド済みの `dist/` の存在を確認 | なし（毎回確認） |
| 8. Docker ログローテーション | `/etc/docker/daemon.json` 作成 | daemon.json が既に存在 |
| 9. サーバー起動 | `docker compose up -d`（Go プラグインはビルド済み前提） | なし（毎回実行） |
| 10. MinIO バケット初期化 | `avatars`, `assets`, `uploads` バケット作成 + public-read ポリシー設定 | なし（毎回実行、既存バケットはスキップ） |

> パスワード（PostgreSQL・MinIO・console）は毎回再生成されます（ボリュームも毎回削除するため）。server_key は `tommie-chat` 固定です。

以下は手動で行う場合の詳細手順です。

## 4. VPS 初期設定

### 4.1 SSH 接続・ユーザー設定

#### VPS 側: 作業ユーザーの作成

root（またはさくらVPS の初期ユーザー）で VPS にログインして実行する。

> **セキュリティ:** 以下の `<deploy_user>` は任意のユーザー名に置換すること。
> `admin`, `deploy`, `ubuntu` 等の一般的な名前は SSH 総当たり攻撃の対象になりやすい。
> 推測されにくい独自のユーザー名を推奨。

```bash
# 作業ユーザーを作成
sudo adduser <deploy_user>

# sudo 権限を付与（doDeploy.sh 内で sudo を使用するため）
sudo usermod -aG sudo <deploy_user>

# docker 権限を付与（doDeploy.sh / docker compose を使うため）
# ※ Docker 未インストール時は doDeploy.sh が自動インストール後にグループ追加する
sudo usermod -aG docker <deploy_user>
```

#### 開発環境（WSL2）側: SSH 鍵認証の設定

WSL2 の開発ユーザー（例: `tommie`）で実行する。

```bash
# 1. SSH 鍵がなければ生成
ls ~/.ssh/id_ed25519.pub || ssh-keygen -t ed25519

# 2. VPS に公開鍵を登録（作業ユーザーのパスワードを聞かれるので入力）
ssh-copy-id <deploy_user>@<VPS_IP>

# 3. パスワードなしで接続できるか確認
ssh <deploy_user>@<VPS_IP> "echo ok"
```

> `ssh-copy-id` が使えない場合は手動で公開鍵を登録する:
>
> ```bash
> cat ~/.ssh/id_ed25519.pub | ssh <deploy_user>@<VPS_IP> \
>   "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"
> ```

#### VPS 側: パスワードログインの無効化（鍵認証の確認後）

```bash
sudo sed -i 's/^#*PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config
sudo systemctl restart sshd
```

> **注意:** パスワードログインを無効化する前に、必ず鍵認証で接続できることを確認すること。
> 無効化後にパスワードでログインできなくなるため、鍵を紛失するとアクセス不能になる。

#### 開発環境（WSL2）側: nakama/.env.deploy の作成

リモート操作系スクリプト（`doDeploy-remote.sh` / `doStop-remote.sh` / `doMigrateGround.sh` /
`doMigrateMinIO.sh`）が SSH ユーザー名をハードコードしないよう、`nakama/.env.deploy` に設定する。
このファイルは `.gitignore` 済みでリポジトリに公開されない。

```bash
cat > nakama/.env.deploy <<'EOF'
# SSH デプロイ設定（git 管理外）
DEPLOY_SSH_USER=<deploy_user>
# DEPLOY_SSH_HOST=mmo.tommie.jp                                           # 任意。指定するとスクリプト引数を省略可
# DEPLOY_REMOTE_DIR=~/mmo.tommie.jp                                       # 任意。VPS 上のインストール先（既定: ~/<VPSホスト名>）
# DEPLOY_GOOGLE_CLIENT_ID=999-prod.apps.googleusercontent.com             # 任意。本番用 Client ID で差し替え
EOF
chmod 600 nakama/.env.deploy
```

| 変数 | 必須 | 説明 |
| ---- | ---- | ---- |
| `DEPLOY_SSH_USER` | ◯ | VPS 上の作業ユーザー名（4.1 で作成したもの） |
| `DEPLOY_SSH_HOST` | △ | VPS のホスト名/IP。指定すれば一部スクリプトの第1引数省略可 |
| `DEPLOY_REMOTE_DIR` | △ | VPS 上のインストール先ディレクトリ。`doDeploy-remote.sh` で使用。既定は `~/<VPSホスト名>`（第一引数のホスト名と同じディレクトリ）。ステージング `mmo-test.tommie.jp` → `~/mmo-test.tommie.jp`、本番 `mmo.tommie.jp` → `~/mmo.tommie.jp` のように自動で分離される |
| `DEPLOY_GOOGLE_CLIENT_ID` | △ | `doDeploy-remote.sh` がビルド後に `dist/index.html` の `<meta name="google-oauth-client-id">` を書き換えるための Client ID。本番と dev/staging で OAuth クライアントを分離する運用（doc/55 §4-5）で使う。未指定なら差し替えなし（ソースの Client ID のまま） |

**解決順位** （上が優先）:

1. コマンドライン引数（例: `./doDeploy-remote.sh -d /opt/tommie-chat mmo-test.tommie.jp myuser`）
2. `nakama/.env.deploy` の `DEPLOY_SSH_USER` / `DEPLOY_SSH_HOST` / `DEPLOY_REMOTE_DIR`
3. スクリプトのデフォルト値（`DEPLOY_SSH_USER` は `deploy`、`DEPLOY_REMOTE_DIR` は `~/<VPSホスト名>`）

> デフォルトの分離動作:
>
> ```bash
> # ホスト名と同じディレクトリに自動で展開される
> ./nakama/doDeploy-remote.sh mmo-test.tommie.jp   # → ~/mmo-test.tommie.jp
> ./nakama/doDeploy-remote.sh mmo.tommie.jp        # → ~/mmo.tommie.jp
>
> # 明示的に指定したい場合のみ -d
> ./nakama/doDeploy-remote.sh -d /opt/tommie-chat mmo.tommie.jp
> ```
>
> 旧バージョンでは SSH ユーザー名がスクリプト内に `deploy` でハードコードされており、
> ユーザー名を変更するとソース修正が必要だった。`.env.deploy` 化により、ユーザーごとに
> 環境を分離できる。

### 4.2 ファイアウォール

```bash
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 22/tcp    # SSH
sudo ufw allow 80/tcp    # HTTP
sudo ufw allow 443/tcp   # HTTPS
sudo ufw enable
```

> **注意:** 7350, 7351, 5432, 9090 等は外部に公開しない。nginx 経由でアクセスする。

#### 本番環境のポート構成

| ポート | サービス | 外部公開 | 説明 |
| ----- | ------- | ------- | ---- |
| 80 (dev) / 8080 (prod) | nginx | **公開** | Web 配信 + Nakama API/WS プロキシ + MinIO `/s3/` プロキシ |
| 443 | nginx | **公開** | HTTPS（Let's Encrypt 設定時） |
| 7350 | Nakama API | `127.0.0.1` のみ | nginx が `/v2/`, `/healthcheck`, `/ws` をプロキシ |
| 7351 | Nakama Console | `127.0.0.1` のみ | 管理画面（SSH トンネルでアクセス） |
| 5432 | PostgreSQL | `127.0.0.1` のみ | データベース |
| 9000 | MinIO S3 API | `127.0.0.1` のみ | nginx が `/s3/` をプロキシ |
| 9001 | MinIO Console | `127.0.0.1` のみ | 管理画面（SSH トンネルでアクセス） |
| 9090 | Prometheus | `127.0.0.1` のみ | メトリクス収集 |

ブラウザクライアントおよびリモートテストは nginx 経由（ポート 80/8080）で Nakama API にアクセスします。
Nakama API ポート（7350）を外部に公開する必要はありません。

`docker-compose.prod.yml` では Nakama・PostgreSQL のポートを `127.0.0.1` にバインドし、外部からの直接アクセスを防止しています。
開発用の `docker-compose.dev.yml` では全ポートを `0.0.0.0` にバインドしています。

### 4.3 スワップ設定（メモリ 2GB 以下の場合）

```bash
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab
```

## 5. Docker インストール

```bash
# 前提パッケージ
sudo apt-get update
sudo apt-get install -y ca-certificates curl gnupg lsb-release

# Docker 公式リポジトリの追加
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg

echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# インストール
sudo apt-get update
sudo apt-get install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin

# ユーザーを docker グループに追加
sudo usermod -aG docker $USER
newgrp docker
```

## 6. アプリケーションのデプロイ

### 6.1 リポジトリの取得

```bash
cd ~
git clone https://github.com/open-tommie/tommie-chat.git
cd tommie-chat
```

### 6.2 フロントエンドのビルドと転送

フロントエンドは **開発環境（WSL2 Ubuntu 24.04）** でビルドし、VPS に rsync で転送する。
VPS 上での Node.js インストールやビルドは不要。

```bash
# --- 開発環境（WSL2 Ubuntu 24.04）で実行 ---

# 本番用の環境変数を設定してビルド
cat > .env <<'EOF'
VITE_SERVER_KEY=tommie-chat
VITE_DEFAULT_HOST=mmo.tommie.jp
VITE_DEFAULT_PORT=443
EOF
npm run build
rm .env  # ビルド後は不要（server_key は dist/ に埋め込み済み）

# VPS に転送
rsync -avz --delete dist/ <deploy_user>@<VPS_IP>:~/tommie-chat/dist/
```

> `rsync` は差分転送なので、2回目以降は変更ファイルのみ転送されます。

### 6.3 自動デプロイ（推奨）

```bash
cd nakama
bash doDeploy.sh
```

これにより以下が自動で行われます:

- `.env` にランダムなパスワード（PostgreSQL・MinIO・console）を生成（server_key は `tommie-chat` 固定）
- 本番用 nginx.conf を生成（静的配信 + MinIO `/s3/` プロキシ + Nakama API/WS プロキシ）
- フロントエンド（`dist/`）の存在確認（開発環境でビルド・rsync 済み前提）
- Docker コンテナ起動（本番構成: `docker-compose.prod.yml`）
- MinIO バケット初期化（avatars, assets, uploads）
- Go プラグイン（`world.so`）は開発環境でビルド済み（git に含まれる）

### 6.4 テスト実行

```bash
cd ../test
set -a; source ../nakama/.env; set +a
./doAll.sh
```

疎通テスト（`doTest-ping.sh`）が最初に実行され、5項目を確認します:

1. コンテナ healthy 待ち
2. HTTP ヘルスチェック
3. server_key 認証
4. RPC getWorldMatch（Go プラグイン確認）
5. Web 配信（nginx）

### 6.5 リモートサーバへのテスト

デプロイ後、別マシンからリモートサーバに対してテストを実行できます。
`--host` / `--port` オプションまたは `NAKAMA_HOST` / `NAKAMA_PORT` 環境変数で接続先を指定します。
リモートホスト指定時は Go ビルド・サーバ再起動・Docker ログ収集が自動スキップされます。

```bash
# 疎通テスト
./test/doTest-ping.sh --host mmo.tommie.jp

# 全テスト（リモート）
NAKAMA_SERVER_KEY=tommie-chat ./test/doAll.sh --host mmo.tommie.jp

# ポート指定（HTTPS 経由等）
NAKAMA_SERVER_KEY=tommie-chat ./test/doAll.sh --host mmo.tommie.jp --port 443

# 個別テスト
NAKAMA_SERVER_KEY=tommie-chat ./test/doTest-snd-rcv.sh --host mmo.tommie.jp -n 100
```

> リモートテストでは `NAKAMA_SERVER_KEY=tommie-chat` の指定が必要です（ローカルの `.env` は自動読み込みされますが、リモートサーバの server_key と一致させる必要があります）。
> リモートホスト指定時は Go ビルド・サーバ再起動・Docker ログ収集が自動スキップされます。
> テスト間のサーバ再起動もスキップされるため、前テストのプレイヤーが残った状態で次テストが実行されます。
> 各テストは固有のユーザー名を使うため大半のケースで問題ありませんが、厳密なテスト分離が必要な場合はローカルで実行してください。

## 7. HTTPS 対応（Let's Encrypt）

### 7.1 certbot のインストール

```bash
sudo apt-get install -y certbot
```

### 7.2 証明書の取得

```bash
# nginx を一時停止してポート80を解放
cd ~/tommie-chat/nakama
docker compose -f docker-compose.yml -f docker-compose.prod.yml stop web

# 証明書取得
sudo certbot certonly --standalone -d mmo.tommie.jp

# nginx を再起動
docker compose -f docker-compose.yml -f docker-compose.prod.yml start web
```

### 7.3 ホスト nginx の HTTPS 設定

HTTPS はホスト nginx（`/etc/nginx/`）が終端し、Docker nginx（8080）に転送する構成です。
Docker nginx（`nakama/nginx.conf`）は HTTP のまま変更不要です。

```text
ブラウザ → ホスト nginx (443/HTTPS) → Docker nginx (8080/HTTP) → Nakama (7350)
```

`/etc/nginx/sites-available/mmo.tommie.jp` を作成:

```nginx
server {
    listen 80;
    server_name mmo.tommie.jp;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    server_name mmo.tommie.jp;

    ssl_certificate     /etc/letsencrypt/live/mmo.tommie.jp/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/mmo.tommie.jp/privkey.pem;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

有効化と反映:

```bash
sudo ln -sf /etc/nginx/sites-available/mmo.tommie.jp /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl reload nginx
```

### 7.4 証明書の自動更新

```bash
# cron で自動更新（月2回）
echo "0 3 1,15 * * certbot renew --pre-hook 'systemctl stop nginx' --post-hook 'systemctl start nginx'" | crontab -
```

### 7.5 HTTPS デプロイ確認テスト（VPS 上で実行）

```bash
./test/doTest-https.sh mmo.tommie.jp
```

11項目を確認します:

| # | チェック内容 | 期待値 |
| --- | ---------- | ------ |
| 1 | ホスト nginx 起動 | active |
| 2 | SSL 証明書存在 + 有効期限 | ファイル存在、期限内 |
| 3 | Docker nginx (8080) 応答 | HTTP 200 |
| 4 | Content-Type | text/html |
| 5 | HTTP → HTTPS リダイレクト | 301 |
| 6 | HTTPS 応答 | HTTP 200 |
| 7 | HTTPS Content-Type | text/html |
| 8 | WebSocket プロキシ | 101 / 400 / 401 |
| 9 | Nakama API プロキシ | 401 |
| 10 | ポート 7350 バインド | 127.0.0.1 のみ |
| 11 | ポート 7350 ローカル応答 | HTTP 200 |

### 7.6 リモート疎通テスト（WSL2 から実行）

デプロイ後、開発環境から本番サーバへの疎通を確認できます。
`--port 443` を指定すると自動で HTTPS に切り替わります。

```bash
# 疎通テスト（5項目）
NAKAMA_SERVER_KEY=tommie-chat ./test/doTest-ping.sh --host mmo.tommie.jp --port 443

# 全テスト（リモート）
NAKAMA_SERVER_KEY=tommie-chat ./test/doAll.sh --host mmo.tommie.jp --port 443

# 個別テスト
NAKAMA_SERVER_KEY=tommie-chat ./test/doTest-snd-rcv.sh --host mmo.tommie.jp --port 443 -n 100
```

> リモートテストでは `NAKAMA_SERVER_KEY=tommie-chat` の指定が必要です。
> `--port 443` 指定時は自動で `https://` プロトコルを使用します。

## 8. クライアント側の接続先変更

開発環境でビルド時に `.env` で接続先を設定するため、手動設定は通常不要です。

| 設定 | 開発時（`npm run dev`） | 本番（`npm run build`） |
| ---- | ---------------------- | --------------------- |
| `SERVER_KEY` | `defaultkey` | `tommie-chat`（固定値） |
| `HOST` | `127.0.0.1` | `mmo.tommie.jp` |
| `PORT` | `7350` | `443` |
| `USE_SSL` | 自動（`false`） | 自動（`true`） |

> ポート `443` の場合、SSL は自動で有効になります（`NakamaService.ts`）。
> デフォルト値は `/.env.development`（開発）と開発環境でビルド時に作成する `/.env`（本番）で切り替わります。

`doDeploy.sh` が生成する本番用 `nakama/nginx.conf` には以下のプロキシが設定されます:

- `/v2/` — Nakama REST API（認証・RPC）
- `/ws` — WebSocket（ブラウザ ↔ Nakama のリアルタイム通信）
- `/s3/` — MinIO S3 API（アバタースプライト等のアセット配信）

これにより、ブラウザクライアントおよびリモート疎通テスト（`doTest-ping.sh --host`）が
nginx 経由（ポート 80/8080）で Nakama API にアクセスできます。
Nakama API ポート（7350）を外部に公開する必要はありません。

## 9. 運用

### 9.1 Docker Compose 構成

開発と本番で Docker Compose ファイルを分離しています:

| ファイル | 用途 |
| ------- | ---- |
| `docker-compose.yml` | 基本構成（postgres, nakama, web, prometheus, minio。ポート定義なし） |
| `docker-compose.dev.yml` | 開発用（全ポート `0.0.0.0` バインド） |
| `docker-compose.prod.yml` | 本番用（`127.0.0.1` バインド、Web は 8080、MinIO は 9000/9001、`name: tommchat-prod`） |

```bash
# 開発環境
docker compose -f docker-compose.yml -f docker-compose.dev.yml up -d

# 本番環境
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d
```

本番の Web ポートは 8080 です（既存の nginx が 80 を使用している前提、mmo.tommie.jp からリバースプロキシ）。

### 9.2 環境変数ファイル（.env）

開発とサーバで `.env` ファイルが分かれています。

| ファイル | 読み込むツール | 用途 |
| ------- | ------------ | ---- |
| `/.env.development` | Vite（`npm run dev` 時） | 開発ブラウザ用。`VITE_SERVER_KEY=defaultkey` や `VITE_DEBUG=true` 等を設定 |
| `/nakama/.env` | Docker Compose | サーバ用（`POSTGRES_PASSWORD`, `NAKAMA_SERVER_KEY`, `MINIO_ROOT_USER`, `MINIO_ROOT_PASSWORD` 等）。`doRestart.sh` / `docker compose up` 時に使用 |

> `/.env` は開発環境でのビルド時に手動で作成・削除します（セクション 6.2 参照）。
> Vite は `package.json` のあるディレクトリ（プロジェクトルート）の `.env` を読み、
> Docker Compose は `docker-compose.yml` のあるディレクトリ（`nakama/`）の `.env` を読みます。

### 9.7 アップデート手順


## 10. デプロイ方式について

本プロジェクトは公式 Docker イメージ（Nakama, PostgreSQL, nginx, Prometheus, MinIO）をそのまま使い、Go プラグインとフロントエンドをファイルマウントで配置する方式を採用しています。Go プラグインのビルドは `nakama-pluginbuilder` Docker イメージ内で行うため、本番サーバーに Go コンパイラのインストールは不要です。フロントエンドは開発環境でビルドし rsync で転送するため、本番に Node.js のインストールも不要です。
