# さくらVPS デプロイ手順

tommChat を さくらVPS（Ubuntu）にデプロイする手順。
ローカル開発環境（WSL2）からの移行を想定。

## 自動スクリプト

手動手順の代わりに、以下のスクリプトで一括セットアップできます:

```bash
# VPS デプロイ（セクション 1〜3, 5〜6 を自動化）
./nakama/doSetupVPS.sh

# HTTPS 設定（セクション 4 を自動化）
./nakama/doSetupHTTPS.sh <ドメイン名>
```

> SSH 鍵認証の設定（1.1）は手動で行ってください。

### 事前テスト（WSL2 / 開発環境）

本番実行前に、別ディレクトリでスクリプトの動作を確認できます:

```bash
# リポジトリをクローン（test/log 等の大容量ファイルを含まない）
git clone ~/24-mmo-Tommie-chat ~/test-deploy
cd ~/test-deploy/nakama

# テスト実行（ufw・スワップ・ログローテーションは WSL2 ではスキップされる）
bash doSetupVPS.sh

# 確認
docker compose -f docker-compose.yml -f docker-compose.prod.yml ps
cat .env

# テスト後クリーンアップ
docker compose -f docker-compose.yml -f docker-compose.prod.yml down -v
cd ~ && rm -rf ~/test-deploy
```

各ステップのスキップ条件:

| ステップ | スキップ条件 |
| -------- | ----------- |
| 1. ファイアウォール | ufw が無い、または既に active |
| 2. スワップ | メモリ 2GB 超、または /swapfile 既存 |
| 3. Docker | `docker` コマンドが既に存在 |
| 4. Node.js | `node` コマンドが既に存在 |
| 5. ビルド | 毎回実行 |
| 6. .env | POSTGRES_PASSWORD が localdev 以外なら既に本番設定済み |
| 7. セキュリティ | .env に NAKAMA_SERVER_KEY が既に存在 |
| 8. ログローテーション | /etc/docker/daemon.json が既に存在 |
| 9. 起動 | 毎回実行 |

以下は手動で行う場合の詳細手順です。

## 前提条件

| 項目 | 要件 |
|------|------|
| VPS | さくらVPS（Ubuntu 22.04 / 24.04 推奨） |
| メモリ | 2GB 以上推奨 |
| ドメイン | 任意（HTTPS 必要時は必須） |
| ローカル | `npm run build` でフロントエンドをビルド済み |

## 1. VPS 初期設定

### 1.1 SSH 接続・ユーザー設定

```bash
# root で初回ログイン後、作業ユーザーを作成
adduser deploy
usermod -aG sudo deploy

# SSH 鍵認証の設定（ローカルから）
ssh-copy-id deploy@<VPS_IP>

# パスワードログインを無効化
sudo sed -i 's/^#*PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config
sudo systemctl restart sshd
```

### 1.2 ファイアウォール

```bash
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 22/tcp    # SSH
sudo ufw allow 80/tcp    # HTTP
sudo ufw allow 443/tcp   # HTTPS
sudo ufw enable
```

> **注意:** 7350, 7351, 5432, 9090 等は外部に公開しない。nginx 経由でアクセスする。

### 1.3 スワップ設定（メモリ 2GB 以下の場合）

```bash
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab
```

## 2. Docker インストール

```bash
# 前提パッケージ
sudo apt-get update
sudo apt-get install -y ca-certificates curl gnupg lsb-release

# Docker 公式リポジトリの追加
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg

echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# インストール
sudo apt-get update
sudo apt-get install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin

# ユーザーを docker グループに追加
sudo usermod -aG docker $USER
newgrp docker
```

## 3. アプリケーションのデプロイ

### 3.1 リポジトリの取得

```bash
cd ~
git clone https://github.com/open-tommie/tommie-chat.git
cd tommie-chat
```

### 3.2 フロントエンドのビルド

```bash
# Node.js インストール（未導入の場合）
curl -fsSL https://deb.nodesource.com/setup_24.x | sudo -E bash -
sudo apt-get install -y nodejs

npm install
npm run build
```

### 3.3 環境変数の設定

```bash
cp nakama/.env.example nakama/.env
```

`nakama/.env` を編集し、本番用の値に変更:

```bash
# まずランダムなパスワードを生成（覚える必要なし）
openssl rand -base64 24
# 出力例: xK7m2pQnR9vL4hBwYzT1sDfG
```

生成された文字列を `.env` に貼り付ける:

```bash
POSTGRES_PASSWORD=xK7m2pQnR9vL4hBwYzT1sDfG
```

### 3.4 docker-compose.yml の本番化

`nakama/docker-compose.yml` を以下のように変更:

#### ポートの制限（外部公開を nginx のみに）

```yaml
# postgres
ports:
  - "127.0.0.1:5432:5432"

# nakama
ports:
  - "127.0.0.1:6060:6060"
  - "127.0.0.1:7349:7349"
  - "127.0.0.1:7350:7350"
  - "127.0.0.1:7351:7351"

# web (nginx) — 外部公開
ports:
  - "80:80"

# prometheus
ports:
  - "127.0.0.1:9090:9090"
```

#### 再起動ポリシーの変更

全サービスの `restart` を変更:

```yaml
restart: unless-stopped
```

#### Nakama のセキュリティ設定

entrypoint に以下を追加:

```yaml
--console.username <管理ユーザー名> \
--console.password <管理パスワード> \
--socket.server_key <ランダムな文字列> \
```

> `--socket.server_key` はクライアント側の `SERVER_KEY` と一致させること（`index.html` の `APP_SERVER_KEY`）。

### 3.5 サーバー起動

```bash
cd nakama
docker compose up -d
bash doBuild.sh --fresh
cd ..
```

## 4. HTTPS 対応（Let's Encrypt）

### 4.1 certbot のインストール

```bash
sudo apt-get install -y certbot
```

### 4.2 証明書の取得

```bash
# nginx を一時停止してポート80を解放
cd ~/tommie-chat/nakama
docker compose stop web

# 証明書取得
sudo certbot certonly --standalone -d example.com

# nginx を再起動
docker compose start web
```

### 4.3 nginx.conf の HTTPS 化

`nakama/nginx.conf` を更新:

```nginx
server {
    listen 80;
    server_name example.com;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    server_name example.com;

    ssl_certificate     /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    root /usr/share/nginx/html;
    index index.html;

    gzip on;
    gzip_types text/plain text/css application/javascript application/json image/svg+xml;

    types {
        image/ktx2 ktx2;
    }

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /assets/ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    location /textures/ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }
}
```

`docker-compose.yml` の web サービスに証明書をマウント:

```yaml
web:
  image: nginx:alpine
  restart: unless-stopped
  volumes:
    - ../dist:/usr/share/nginx/html:ro
    - ./nginx.conf:/etc/nginx/conf.d/default.conf:ro
    - /etc/letsencrypt:/etc/letsencrypt:ro
  ports:
    - "80:80"
    - "443:443"
```

### 4.4 証明書の自動更新

```bash
# cron で自動更新（月2回）
echo "0 3 1,15 * * certbot renew --pre-hook 'cd ~/tommie-chat/nakama && docker compose stop web' --post-hook 'cd ~/tommie-chat/nakama && docker compose start web'" | crontab -
```

## 5. クライアント側の接続先変更

`index.html` の接続設定を本番用に変更:

| 設定 | 開発時 | 本番 |
|------|--------|------|
| `APP_SERVER_KEY` | `defaultkey` | docker-compose.yml の `--socket.server_key` と一致 |
| `APP_NAKAMA_HOST` | `127.0.0.1` | `example.com`（ドメイン名） |
| `APP_NAKAMA_PORT` | `7350` | `443`（HTTPS 経由） |
| `APP_NAKAMA_USE_SSL` | `false` | `true` |

> nginx で WebSocket を Nakama にプロキシする場合は `nginx.conf` に以下を追加:
>
> ```nginx
> location /ws {
>     proxy_pass http://nakama:7350;
>     proxy_http_version 1.1;
>     proxy_set_header Upgrade $websocket_upgrade;
>     proxy_set_header Connection "upgrade";
>     proxy_set_header Host $host;
> }
> ```

## 6. 運用

### 6.1 Docker ログローテーション

`/etc/docker/daemon.json` を作成:

```json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }
}
```

```bash
sudo systemctl restart docker
```

### 6.2 PostgreSQL バックアップ

```bash
# 手動バックアップ
docker compose exec postgres pg_dump -U nakama nakama > backup-$(date +%Y%m%d).sql

# cron で日次バックアップ
echo "0 4 * * * cd ~/tommie-chat/nakama && docker compose exec -T postgres pg_dump -U nakama nakama > ~/backups/nakama-\$(date +\%Y\%m\%d).sql" | crontab -
```

### 6.3 状態確認

```bash
cd ~/tommie-chat/nakama

# コンテナ状態
docker compose ps

# Nakama ログ
docker compose logs -f nakama

# リソース使用量
docker stats
```

### 6.4 アップデート手順

```bash
cd ~/tommie-chat
git pull
npm install
npm run build
cd nakama
bash doBuild.sh --fresh    # Go プラグインの再ビルド
docker compose restart nakama
```

## 7. デプロイ方式について

本プロジェクトは公式 Docker イメージ（Nakama, PostgreSQL, nginx, Prometheus）をそのまま使い、Go プラグインとフロントエンドをファイルマウントで配置する方式を採用しています。Go プラグインのビルドは `nakama-pluginbuilder` Docker イメージ内で行うため、本番サーバーに Go コンパイラのインストールは不要です。Node.js はフロントエンドのビルド（`npm run build`）に必要です。

将来的にはカスタム Dockerfile を作成し、Go プラグインとフロントエンドを含んだイメージをレジストリ経由でデプロイする方式も考えられます。その場合、本番に Node.js も不要になります。現段階では開発中のため、スクリプト方式の方がシンプルです。
