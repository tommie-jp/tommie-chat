# さくらVPS デプロイ手順

tommChat を さくらVPS（Ubuntu）にデプロイする手順。
ローカル開発環境（WSL2）からの移行を想定。

## 自動スクリプト

手動手順の代わりに、以下のスクリプトで一括セットアップできます:

```bash
# VPS デプロイ（セクション 1〜3, 5〜6 を自動化）
./nakama/doDeploy.sh

# HTTPS 設定（セクション 4 を自動化）※未テスト
./nakama/doSetupHTTPS.sh <ドメイン名>
```

> SSH 鍵認証の設定（1.1）は手動で行ってください。

### doDeploy.sh の処理内容

| ステップ | 内容 | スキップ条件 |
| -------- | ---- | ----------- |
| 0. 既存コンテナ停止 | `docker compose down -v` でコンテナ・ボリューム削除 | なし（毎回実行） |
| 1. ファイアウォール | ufw で 22/80/443 のみ許可 | ufw が無い、または既に active |
| 2. スワップ | 2GB スワップ作成 | メモリ 2GB 超、または /swapfile 既存 |
| 3. Docker | Docker Engine + Compose インストール | `docker` コマンドが既に存在 |
| 4. Node.js | Node.js 24.x インストール | `node` コマンドが既に存在 |
| 5. 環境変数 | `.env` 生成（PG パスワード・server_key・console パスワード自動生成） | なし（毎回再生成） |
| 6. フロントエンドビルド | `npm install && npm run build`（server_key 自動設定） | なし（毎回実行） |
| 7. Docker ログローテーション | `/etc/docker/daemon.json` 作成 | daemon.json が既に存在 |
| 8. サーバー起動 | `docker compose up -d` + Go プラグインビルド | なし（毎回実行） |

> パスワード・キーは毎回再生成されます（ボリュームも毎回削除するため）。メモ不要です。

### デプロイテスト（WSL2 / 開発環境）

本番実行前に、別ディレクトリでデプロイ〜テストまで一括確認できます。
`doDeployTest.sh` は git clone → doDeploy.sh → doAll.sh（疎通テスト + 全テスト）を一括実行します。

```bash
# 作業ディレクトリを作成
mkdir -p ~/deploy-test && cd ~/deploy-test

# doDeployTest.sh を取得して実行
curl -fsSL https://raw.githubusercontent.com/open-tommie/tommie-chat/main/test/doDeployTest.sh -o doDeployTest.sh
chmod +x doDeployTest.sh
./doDeployTest.sh
```

2回目以降は `./doDeployTest.sh` だけで再実行できます（既存ディレクトリの削除を確認後、再クローンします）。

以下は手動で行う場合の詳細手順です。

| 項目 | 要件 |

| メモリ | 2GB 以上推奨 |
| ドメイン | 任意（HTTPS 必要時は必須） |

## 1. VPS 初期設定

### 1.1 SSH 接続・ユーザー設定

```bash
# root で初回ログイン後、作業ユーザーを作成
adduser deploy
usermod -aG sudo deploy

# SSH 鍵認証の設定（ローカルから）
ssh-copy-id deploy@<VPS_IP>

# パスワードログインを無効化
sudo sed -i 's/^#*PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config
sudo systemctl restart sshd
```

### 1.2 ファイアウォール

```bash
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 22/tcp    # SSH
sudo ufw allow 80/tcp    # HTTP
sudo ufw allow 443/tcp   # HTTPS
sudo ufw enable
```

> **注意:** 7350, 7351, 5432, 9090 等は外部に公開しない。nginx 経由でアクセスする。

#### 本番環境のポート構成

| ポート | サービス | 外部公開 | 説明 |
| ----- | ------- | ------- | ---- |
| 80 (dev) / 8080 (prod) | nginx | **公開** | Web 配信 + Nakama API プロキシ |
| 443 | nginx | **公開** | HTTPS（Let's Encrypt 設定時） |
| 7350 | Nakama API | `127.0.0.1` のみ | nginx が `/v2/`, `/healthcheck`, `/ws` をプロキシ |
| 7351 | Nakama Console | `127.0.0.1` のみ | 管理画面（SSH トンネルでアクセス） |
| 5432 | PostgreSQL | `127.0.0.1` のみ | データベース |
| 9090 | Nakama gRPC | `127.0.0.1` のみ | 内部通信用 |

ブラウザクライアントおよびリモートテストは nginx 経由（ポート 80/8080）で Nakama API にアクセスします。
Nakama API ポート（7350）を外部に公開する必要はありません。

`docker-compose.prod.yml` では Nakama・PostgreSQL のポートを `127.0.0.1` にバインドし、外部からの直接アクセスを防止しています。
開発用の `docker-compose.dev.yml` では全ポートを `0.0.0.0` にバインドしています。

### 1.3 スワップ設定（メモリ 2GB 以下の場合）

```bash
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab
```

## 2. Docker インストール

```bash
# 前提パッケージ
sudo apt-get update
sudo apt-get install -y ca-certificates curl gnupg lsb-release

# Docker 公式リポジトリの追加
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg

echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# インストール
sudo apt-get update
sudo apt-get install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin

# ユーザーを docker グループに追加
sudo usermod -aG docker $USER
newgrp docker
```

## 3. アプリケーションのデプロイ

### 3.1 リポジトリの取得

```bash
cd ~
git clone https://github.com/open-tommie/tommie-chat.git
cd tommie-chat
```

### 3.2 自動デプロイ（推奨）

```bash
cd nakama
bash doDeploy.sh
```

これにより以下が自動で行われます:

- `.env` にランダムなパスワード・server_key・console パスワードを生成
- フロントエンドビルド（server_key を自動埋め込み）
- Docker コンテナ起動（本番構成: `docker-compose.prod.yml`）
- Go プラグインビルド

### 3.3 テスト実行

```bash
cd ../test
set -a; source ../nakama/.env; set +a
./doAll.sh
```

疎通テスト（`doTest-ping.sh`）が最初に実行され、5項目を確認します:

1. コンテナ healthy 待ち
2. HTTP ヘルスチェック
3. server_key 認証
4. RPC getWorldMatch（Go プラグイン確認）
5. Web 配信（nginx）

### 3.4 リモートサーバへのテスト

デプロイ後、別マシンからリモートサーバに対してテストを実行できます。
`--host` / `--port` オプションまたは `NAKAMA_HOST` / `NAKAMA_PORT` 環境変数で接続先を指定します。
リモートホスト指定時は Go ビルド・サーバ再起動・Docker ログ収集が自動スキップされます。

```bash
# 疎通テスト
./test/doTest-ping.sh --host mmo.tommie.jp

# 全テスト（リモート）
NAKAMA_SERVER_KEY=<server_key> ./test/doAll.sh --host mmo.tommie.jp

# ポート指定（HTTPS 経由等）
NAKAMA_SERVER_KEY=<server_key> ./test/doAll.sh --host mmo.tommie.jp --port 443

# 個別テスト
NAKAMA_SERVER_KEY=<server_key> ./test/doTest-snd-rcv.sh --host mmo.tommie.jp -n 100
```

> リモートテストでは `NAKAMA_SERVER_KEY` の手動指定が必要です（ローカルの `.env` は自動読み込みされますが、リモートサーバの server_key と一致させる必要があります）。
> リモートホスト指定時は Go ビルド・サーバ再起動・Docker ログ収集が自動スキップされます。
> テスト間のサーバ再起動もスキップされるため、前テストのプレイヤーが残った状態で次テストが実行されます。
> 各テストは固有のユーザー名を使うため大半のケースで問題ありませんが、厳密なテスト分離が必要な場合はローカルで実行してください。

## 4. HTTPS 対応（Let's Encrypt）

### 4.1 certbot のインストール

```bash
sudo apt-get install -y certbot
```

### 4.2 証明書の取得

```bash
# nginx を一時停止してポート80を解放
cd ~/tommie-chat/nakama
docker compose -f docker-compose.yml -f docker-compose.prod.yml stop web

# 証明書取得
sudo certbot certonly --standalone -d example.com

# nginx を再起動
docker compose -f docker-compose.yml -f docker-compose.prod.yml start web
```

### 4.3 nginx.conf の HTTPS 化

`nakama/nginx.conf` を更新:

```nginx
server {
    listen 80;
    server_name example.com;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    server_name example.com;

    ssl_certificate     /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    root /usr/share/nginx/html;
    index index.html;

    gzip on;
    gzip_types text/plain text/css application/javascript application/json image/svg+xml;

    types {
        image/ktx2 ktx2;
    }

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /assets/ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    location /textures/ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }
}
```

`docker-compose.yml` の web サービスに証明書をマウント:

```yaml
web:
  image: nginx:alpine
  restart: unless-stopped
  volumes:
    - ../dist:/usr/share/nginx/html:ro
    - ./nginx.conf:/etc/nginx/conf.d/default.conf:ro
    - /etc/letsencrypt:/etc/letsencrypt:ro
  ports:
    - "80:80"
    - "443:443"
```

### 4.4 証明書の自動更新

```bash
# cron で自動更新（月2回）
echo "0 3 1,15 * * certbot renew --pre-hook 'cd ~/tommie-chat/nakama && docker compose -f docker-compose.yml -f docker-compose.prod.yml stop web' --post-hook 'cd ~/tommie-chat/nakama && docker compose -f docker-compose.yml -f docker-compose.prod.yml start web'" | crontab -
```

## 5. クライアント側の接続先変更

`doDeploy.sh` が server_key を自動設定するため、手動設定は通常不要です。

HTTPS 対応時に `src/network.ts` 等の接続先を変更:

| 設定 | 開発時 | 本番 |
| ---- | ------ | ---- |
| `SERVER_KEY` | `defaultkey` | `.env` の `NAKAMA_SERVER_KEY`（自動設定済み） |
| `HOST` | `127.0.0.1` | `example.com`（ドメイン名） |
| `PORT` | `7350` | `443`（HTTPS 経由） |
| `USE_SSL` | `false` | `true` |

`nakama/nginx.conf` には以下の Nakama API プロキシが設定済みです:

- `/v2/` — Nakama REST API（認証・RPC）
- `/healthcheck` — ヘルスチェック
- `/ws` — WebSocket（ブラウザ ↔ Nakama のリアルタイム通信）

これにより、ブラウザクライアントおよびリモート疎通テスト（`doTest-ping.sh --host`）が
nginx 経由（ポート 80/8080）で Nakama API にアクセスできます。
Nakama API ポート（7350）を外部に公開する必要はありません。

## 6. 運用

### 6.1 Docker Compose 構成

開発と本番で Docker Compose ファイルを分離しています:

| ファイル | 用途 |
| ------- | ---- |
| `docker-compose.yml` | 基本構成（ポート定義なし） |
| `docker-compose.dev.yml` | 開発用（全ポート `0.0.0.0` バインド） |
| `docker-compose.prod.yml` | 本番用（`127.0.0.1` バインド、Web は 8080、`name: tommchat-prod`） |

```bash
# 開発環境
docker compose -f docker-compose.yml -f docker-compose.dev.yml up -d

# 本番環境
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d
```

本番の Web ポートは 8080 です（既存の nginx が 80 を使用している前提、mmo.tommie.jp からリバースプロキシ）。

### 6.1.1 環境変数ファイル（.env）

開発とサーバで `.env` ファイルが分かれています。

| ファイル | 読み込むツール | 用途 |
| ------- | ------------ | ---- |
| `/.env.development` | Vite（`npm run dev` 時） | 開発ブラウザ用。`VITE_SERVER_KEY=defaultkey` や `VITE_DEBUG=true` 等を設定 |
| `/nakama/.env` | Docker Compose | サーバ用（`POSTGRES_PASSWORD`, `NAKAMA_SERVER_KEY` 等）。`doRestart.sh` / `docker compose up` 時に使用 |

> `/.env` は `doDeploy.sh` がビルド時に自動生成・削除します（手動編集は不要）。
> Vite は `package.json` のあるディレクトリ（プロジェクトルート）の `.env` を読み、
> Docker Compose は `docker-compose.yml` のあるディレクトリ（`nakama/`）の `.env` を読みます。

### 6.2 運用スクリプト

| スクリプト | 用途 |
| --------- | ---- |
| `nakama/doDeploy.sh` | 初回デプロイ（環境構築〜起動） |
| `nakama/doSetupHTTPS.sh` | HTTPS 設定（Let's Encrypt） |
| `nakama/doBuild.sh` | Go プラグイン再ビルド + nakama 再起動 |
| `nakama/doRestart.sh` | 開発環境の再起動 |
| `nakama/doStop.sh` | 全コンテナ停止（dev/prod 両方） |
| `test/doTest-ping.sh` | 疎通テスト（5項目、`--host`/`--port` 対応） |
| `test/doAll.sh` | 全テスト一括実行（デフォルト100人、`--host`/`--port` 対応） |

### 6.3 Docker ログローテーション

`/etc/docker/daemon.json` を作成:

```json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }
}
```

```bash
sudo systemctl restart docker
```

### 6.4 PostgreSQL バックアップ

```bash
# 手動バックアップ
docker compose -f docker-compose.yml -f docker-compose.prod.yml exec postgres pg_dump -U nakama nakama > backup-$(date +%Y%m%d).sql

# cron で日次バックアップ
echo "0 4 * * * cd ~/tommie-chat/nakama && docker compose -f docker-compose.yml -f docker-compose.prod.yml exec -T postgres pg_dump -U nakama nakama > ~/backups/nakama-\$(date +\%Y\%m\%d).sql" | crontab -
```

### 6.5 状態確認

```bash
cd ~/tommie-chat/nakama

# コンテナ状態
docker compose -f docker-compose.yml -f docker-compose.prod.yml ps

# Nakama ログ
docker compose -f docker-compose.yml -f docker-compose.prod.yml logs -f nakama

# リソース使用量
docker stats

# server_key 確認
grep NAKAMA_SERVER_KEY .env
```

### 6.6 アップデート手順

```bash
cd ~/tommie-chat
git pull
npm install
npm run build
cd nakama
bash doBuild.sh --fresh    # Go プラグインの再ビルド + nakama 再起動
```

> `doBuild.sh` は `docker-compose.prod.yml` を自動検出し、適切な compose コマンドで再起動します。

## 7. デプロイ方式について

本プロジェクトは公式 Docker イメージ（Nakama, PostgreSQL, nginx, Prometheus）をそのまま使い、Go プラグインとフロントエンドをファイルマウントで配置する方式を採用しています。Go プラグインのビルドは `nakama-pluginbuilder` Docker イメージ内で行うため、本番サーバーに Go コンパイラのインストールは不要です。Node.js はフロントエンドのビルド（`npm run build`）に必要です。

将来的にはカスタム Dockerfile を作成し、Go プラグインとフロントエンドを含んだイメージをレジストリ経由でデプロイする方式も考えられます。その場合、本番に Node.js も不要になります。現段階では開発中のため、スクリプト方式の方がシンプルです。
