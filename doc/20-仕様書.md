- 通知の案
  - 想定している使い方
    - オセロゲーム
      - 1.新ゲーム開始ボタンでゲーム作成→パネルを閉じる
      - 2.対戦相手が参加ボタンを押した
      - 3.ゲームオーナーへ通知が来る
        - 通知「対戦相手が見つかりました。[オセロへ移動するリンク]」
          - リンクURL: `?ot=123`
            - othello(オセロパネル指定)
            - オセロゲーム番号
              - （想定している場面）対戦相手待ちのゲーム番号のとき、オセロゲーム開始
              - ゲーム中の番号のとき、オセロ閲覧表示
              - それ以外の番号のとき、オセロパネルトップ表示
                - 存在しない番号 / 削除済み / 既に満員 / 相手が離脱後
                - 理由をトースト表示する
                  - 例：「オセロゲーム番号123は相手がオフラインです」
            - リンクURLは一般的に使う想定
              - QRコードに`https://mmo.tommie.jp/?ot=123`を埋め込む
                - ユーザAがtommieChatでQRコードを表示
                - ユーザBがQRコードアプリで読み込み、URLを開くとオセロ対戦画面へ
  - 表示方法
    - 詳細は トースト（吹き出し） セクション参照
  - 通知フォーマット（サーバ→クライアント）

    - content:
      - gameNo: int 例: 123
      - opponentName: string 例: "ユーザB"
    - クライアントでの吹き出し文字列

      - タップ → `?ot={gameNo}`

-URLスキーム規約
    -?ot=123 オセロパネル表示、オセロゲーム番号123を表示
    -?panel=othelloはサポートしない
    -v2以後
        -?sg=123 将棋パネル表示
        -?dn 表示名設定パネル表示
            -?dn=name1 不採用
                -不採用理由：URLでパネル指定はOK、値の書き込みは不採用
                -パネルだけ開く。値は無視
        -?as アバター選択パネル表示
            -?as=123 アバター番号123をアバター選択パネルで表示
        -?rm=123 部屋番号123へ移動
            -v3:`https://mmo.tommie.jp/rm/123`123は部屋番号
            -v3:`https://mmo.tommie.jp/rm123` rm123は部屋名
        -?pl=512,513 座標(512,513)へ移動
        -?pl=場所名1 場所名1へ移動
        -?rm=123&pl=512,513 部屋番号123の座標(512,513)へ移動

    -別サーバ移動（URL全体）
        -`https://mmo.sample.jp` 別サーバへ移動
            -外部サーバに誘導できるとフィッシングリスク。
            -既知サーバのホワイトリストに含まれるドメインのみ許可
            -未知ドメインは「このURLは外部サーバです。移動しますか？」確認ダイアログ

    -QRコード対応
        -`https://mmo.tommie.jp/?ot=123`を埋め込む
        -QRリンク?ot=123をログインしてない状態で開いたとき
            -未ログイン（デバイス認証）でもオセロゲームはできる
            -名前がユーザID（デバイス認証のID:例3a8f)が読みにくいがゲームはできる
                -v1:これは仕様として許容する（誰でもすぐ対戦できるのが利点）
                -v2:表示名設定ダイアログを最初に表示し、設定後、オセロパネルへ自動で移動
        -v1:PWAはサポートしない
            -iOSではQRコードからPWAを開けない
                -safariが開く。
            -safari, PWAでデバイスIDが別になるが、safariとPWAでgoogle認証しておけば解決するので、google認証を推奨する。
            -google認証は既に実装済み。
            -デバイス認証でのiOS safari, PWAが別ユーザ扱いになる件はサポートしない。（google認証推奨）
        -v2:PWAをフルサポートするのはAndroid,PCのみ。iOSでは制限あり。

-URLパラメータ読み取りのタイミング（遅延呼び出し）
    -QR から入ってきた ?ot=123 の処理順序:
        -1.URLパラメータ(?ot=123)をURL_parameter変数へ覚えておく
        -2.ページロード
        -3.authenticateDevice (localStorage or cookie → deviceId)
        -4.socket 接続確立
        -5. 1で設定した変数にパラメータがあるなら処理する
            -URLパラメータを読み、オセロパネルを開く
            -対象 gameNo の状態を RPC で確認 (待機中 / 対局中 / 不在)
            -不在ならトースト表示してパネルトップ
        -メモ
            -URL_parameter変数はメモリ保持のみ（localStorage 退避不要）
            -ページロード完了→socket接続確立、までの遅延を吸収する目的

-socket notification DB永続化(persistent=true) の運用
    -起動時の取り残し回収
        -クライアント起動時（socket接続確立後）にclient.listNotifications(session, 100) で取り残しを取得
        -処理 → deleteNotifications で消化
    -受信ハンドラでも処理後 deleteNotifications を呼ぶ（重複排除）
    -意図：DBに通知が溜まり続ける＆起動のたびに全件再受信を防止

- ✔FIX BUG: iPhone safariで音がならない。
  - 状況：iPadMini, windows11 chromeではオセロ駒を置いたときなど音がする。
  - 原因：iOS Safari は `AudioContext` を user gesture 内で `resume()` しないとサスペンド状態のまま無音になる。iPad は挙動が緩いので問題化しにくい。
  - さらに本アプリは自動ログイン経路（`setTimeout(doLogin, 100)`）が存在するため、`doLogin` 冒頭で `unlockAudio()` を呼んでも gesture 外となり解禁に失敗する。
  - 対処：
    - 共有 `AudioContext` を `UIPanel` 初期化時に用意し、`unlockAudio()` で `resume()` + 無音バッファ再生。
    - `document` に `pointerdown` / `keydown` / `touchstart` の capture リスナーを仕込み、最初の user gesture で `unlockAudio()` を必ず実行（成功したら解除）。
    - `doLogin` 冒頭でも `unlockAudio()` を呼ぶ（ユーザ操作ログイン時の保険）。
    - `othelloPlaySound` は共有 ctx を使い、呼び出し時に suspended なら `resume()` を試みる。
  - 追加対処（HTMLAudioElement 併用）：
    - `unlockAudio()` 内で無音 WAV data URI を `new Audio()` で読み込み `loop=true`・`volume=0`・`playsInline=true` で `.play()` 実行。
    - iOS Safari は HTMLAudioElement を一度でも再生すると audio session が確立し、以降の WebAudio 再生が通りやすくなる。
    -テスト結果：鳴った。
