## 高優先度リファクタリング計画（5件）

### 1. GameScene.ts — Vector3 毎フレーム生成の排除

**対象**: L679, L716, L720, L959

**現状**: 毎フレーム `new Vector3()` を4箇所で生成（60FPS = 秒間240個のGC対象）

**計画**: クラスフィールドに再利用用の `Vector3` を事前確保

```typescript
// クラスフィールドに追加
private _tmpMoveDir = new Vector3();
private _tmpTarget = new Vector3();
private _tmpDirection = new Vector3();
private _tmpPanTest = new Vector3();
```

- L679: `new Vector3(0,0,0)` → `this._tmpMoveDir.setAll(0)`
- L716: `new Vector3(tx, y, tz)` → `this._tmpTarget.set(tx, y, tz)`
- L720: `target.subtract(currentPos)` → `currentPos.subtractToRef(...)` で `_tmpDirection` に書込
- L959: `new Vector3(newX, 0, newZ)` → `this._tmpPanTest.set(newX, 0, newZ)`

**効果**: フレーム毎のオブジェクト生成ゼロ、GC圧力解消

### 2. GameScene.ts — profileHistory の Array.shift() をリングバッファに

**対象**: L828-829

**現状**: `push` + `shift()`。600要素の配列で `shift()` はO(n)＝毎フレーム600要素の再インデックス

**計画**: 固定長配列 + インデックスポインタ

```typescript
// クラスフィールド
private _profileRing = new Array(600);
private _profileRingIdx = 0;
private _profileRingLen = 0;
```

- `push+shift` → `_profileRing[_profileRingIdx] = entry; _profileRingIdx = (_profileRingIdx + 1) % 600`
- 参照側（DebugOverlay等）で `_profileHistory` を使っている箇所をスナップショット取得メソッドに変更

**効果**: O(n) → O(1)

### 3. NakamaService.ts — TextEncoder インスタンス再利用

**対象**: L367,378,389,398,428,438,658 の7箇所

**現状**: 毎送信で `new TextEncoder()` を生成

**計画**: `_decoder` と同様にクラスフィールドに追加

```typescript
private readonly _encoder = new TextEncoder();
```

- 7箇所すべて `new TextEncoder().encode(...)` → `this._encoder.encode(...)`

**効果**: 送信毎のオブジェクト生成ゼロ。移動メッセージは100ms毎なので秒間10回×7箇所分の削減

### 4. main.go — opMoveTarget チャンク移動時の O(N²) ブロードキャストをバッチ化

**対象**: L1775-1802

**現状**: チャンク移動時、全AOIをループして1人ずつ `json.Marshal` + `BroadcastMessage`。100人なら100回のMarshal+送信

**計画**:

1. Enter/Leaveそれぞれ対象者リストを先に収集（`[]runtime.Presence`）
2. `json.Marshal` は Enter用・Leave用それぞれ **1回だけ**
3. 収集したリストで `BroadcastMessage` を **1回** 呼出

```go
// Before: N回の Marshal + N回の Broadcast
// After:
var enterTargets []runtime.Presence
var leaveTargets []runtime.Presence
for otherSID, otherAOI := range ms.AOIs { ... collect ... }
if len(enterTargets) > 0 {
    enterData, _ := json.Marshal(map[string]interface{}{...})
    dispatcher.BroadcastMessage(opAOIEnter, enterData, enterTargets, nil, true)
}
if len(leaveTargets) > 0 {
    leaveData, _ := json.Marshal(map[string]interface{}{...})
    dispatcher.BroadcastMessage(opAOILeave, leaveData, leaveTargets, nil, true)
}
```

**効果**: Marshal N回→1回、BroadcastMessage N回→1回。100人で99%削減

### 5. main.go — collectAOITargets の空間ハッシュ化

**対象**: L1513-1526

**現状**: 毎回全 `ms.Presences` をスキャンしてAOI内かチェック。100人×10Hz = 秒1000回のフルスキャン

**計画**: チャンク座標→セッションIDの逆引きマップを追加

```go
// matchState に追加
ChunkPlayers map[[2]int]map[string]bool  // (cx,cz) → {sid: true}
```

- `opMoveTarget` でチャンク変更時に `ChunkPlayers` を更新（古いチャンクから削除、新しいチャンクに追加）
- `collectAOITargets` は対象者のAOIチャンク範囲を走査する代わりに、`ChunkPlayers[cx][cz]` の結合で取得
- ただし、AOI未登録プレイヤー（参加直後）のフォールバックは維持

**効果**: O(N) → O(AOI面積)。AOIは通常7×7=49チャンクなので、100人でも49回の参照で済む

**注意**: この変更は影響範囲が広い（MatchJoin/MatchLeave/opAOIUpdateでもChunkPlayersを更新必要）。段階的に実装し、まず #4 のバッチ化だけで大きな効果が出る

### 実装メモ

#### #3 TextEncoder再利用 — 完了

- `NakamaService` クラスに `private readonly _encoder = new TextEncoder()` を追加
- 7箇所の `new TextEncoder().encode(...)` を `this._encoder.encode(...)` に置換

#### #1 Vector3再利用 — 完了

- `GameScene` クラスに4つの再利用用フィールドを追加:
  - `_tmpMoveDir`, `_tmpTarget`, `_tmpDirection`, `_tmpPanTest`
- `new Vector3()` を `set()` / `setAll()` / `subtractToRef()` に置換
- 毎フレーム4個のVector3生成（60FPS = 秒間240個）を排除

#### #2 リングバッファ — 完了

- `_profileHistory` 配列（push+shift O(n)）をリングバッファに変更:
  - `_profileRing: (ProfileEntry | null)[]` (固定長600)
  - `_profileRingIdx`, `_profileRingLen` でポインタ管理
- 後方互換の getter `_profileHistory` で順序付き配列を返却
- `main.ts`, `DebugOverlay.ts` のリセット処理も更新

#### #4 AOIバッチ送信 — 完了

- `opMoveTarget` のチャンク変更時AOI通知をバッチ化（`main.go` L1774-1802）
- **変更前**: 各プレイヤーに対し個別に `json.Marshal` + `BroadcastMessage`（N人 = N回ずつ）
- **変更後**:
  - Enter/Leave対象の `[]runtime.Presence` を先に収集
  - AOI_ENTER: `PendingAOIEnter` バッファに積む（tick末にバルクフラッシュ、`opInitPos` と同じパターン）
  - AOI_LEAVE: `json.Marshal` 1回 + `BroadcastMessage` 1回で全対象に送信
- 100人でMarshal N回→1回、BroadcastMessage N回→1回に削減

#### #5 空間ハッシュ化 — 完了

- `matchState` に `ChunkViewers map[[2]int]map[string]bool` を追加
  - 各チャンク(cx,cz)に対し、そのチャンクをAOI内に含むセッションID群を保持
- ヘルパーメソッド `addChunkViewer` / `removeChunkViewer` で AOI範囲内の全チャンクを一括登録/除去
- 更新箇所:
  - `opAOIUpdate`: 旧AOI除去 → 新AOI登録
  - `MatchJoin`（ゴーストキック）: 旧セッションの ChunkViewers を除去
  - `MatchLeave`（通常 + ゴーストキック）: 退出セッションの ChunkViewers を除去
- `collectAOITargets` を書き換え: `ms.Presences` 全走査 → `ChunkViewers[[2]int{cx,cz}]` の参照のみ
- **計算量**: O(N) → O(viewers)。AOI範囲が7×7の場合、100人中AOI内のプレイヤーのみ参照
