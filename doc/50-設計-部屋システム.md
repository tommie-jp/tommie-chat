# 設計: 部屋システム（広場・マイルーム）

アメーバPigg の「広場」「マイルーム」に相当する機能を tommieChat に実装するための設計方針。
**広場（恒久的なテーマ別空間）を先に実装し、マイルーム（個人部屋）は後回しにする。**

---

## 概要

- **広場**: ワールド内の区域としてテーマ別の空間を設ける（Pigg の渋谷・原宿に相当）
- **マイルーム**: 各ユーザーが自分の部屋を持てる（Phase 2 以降、別空間として実装）
- 広場はワールドと同じ Match 内に存在（シーン切替不要・ローディング不要）
- マイルームは将来的に別 Match として実装

---

## 広場（ワールド内区域方式）

### 設計方針: 別空間ではなくワールド内の区域

Pigg では広場を別空間として実装していたが、tommieChat では
**ワールド（1024×1024）内の特定エリアをブロックで囲んで広場とする**方式を採用する。

#### Pigg 式（別空間）との比較

| 観点 | ワールド内区域（採用） | 別空間（Pigg 式） |
| ---- | ---- | ---- |
| 実装コスト | 小（既存の仕組みで実現） | 大（Match/シーン/チャット切替基盤が必要） |
| 空間の連続性 | 歩いて自然に辿り着ける | ローディングが入る |
| チャンク・AOI | 既存のまま使える | 別管理が必要 |
| 環境の変化 | 空の色等は共通 | 広場ごとに変更可 |
| チャット分離 | 広場エリアにチャットルームを紐付けて分離可 | 自然に分離 |
| 人数制限 | チャットルームの参加人数で制限可 | Match 単位で制限 |

**結論: まずワールド内区域で実装し、環境演出が必要になったら別空間に切り出す。**

### 広場の構成

#### 物理的な構造

広場はワールド内の特定座標に**ブロックの塀で囲んだエリア**として存在する。

```text
ワールド (1024×1024)
┌──────────────────────────────────────────┐
│                                          │
│    ┌──────────┐     ┌────────┐           │
│    │ 公園広場  │     │ ビーチ  │           │
│    │  64×64   │     │ 64×32  │           │
│    │ 入口     │     │ 入口   │           │
│    └──────────┘     └────────┘           │
│                                          │
│         自由建築エリア                     │
│                                          │
│    ┌──────────┐     ┌────────┐           │
│    │ 夜の街   │     │  屋上   │           │
│    │  48×48   │     │ 32×32  │           │
│    │ 入口     │     │ 入口   │           │
│    └──────────┘     └────────┘           │
│                                          │
└──────────────────────────────────────────┘
```

- 塀（壁ブロック）で囲むことで視覚的に「広場」と認識できる
- 入口（開口部）を1〜2箇所設け、アバターが歩いて出入りする
- 広場内の装飾ブロックは運営が配置（一般ユーザーは編集不可）

#### 広場一覧（サーバ側で定義）

```go
// サーバ側 Go — 広場定義
type PlazaDef struct {
    ID       string  // "plaza_park", "plaza_beach" 等
    Name     string  // "公園広場", "ビーチ"
    MinX     int     // エリア左端（ワールド座標）
    MinZ     int     // エリア下端
    MaxX     int     // エリア右端
    MaxZ     int     // エリア上端
    SpawnX   float64 // テレポート先（入口付近）
    SpawnZ   float64
    ChatRoom string  // チャットルーム名 "plaza_park"
    MaxUsers int     // チャットルーム人数上限
}
```

初期実装で用意する広場の例:

| ID | 名前 | サイズ | 座標（例） | テーマ |
| ---- | ---- | ---- | ---- | ---- |
| `plaza_park` | 公園広場 | 64×64 | (100,100)-(164,164) | 緑の広場・木や花 |
| `plaza_beach` | ビーチ | 64×32 | (200,100)-(264,132) | 砂浜風ブロック |
| `plaza_night` | 夜の街 | 48×48 | (100,200)-(148,248) | 高い壁・暗い雰囲気 |
| `plaza_rooftop` | 屋上 | 32×32 | (200,200)-(232,232) | 高台に配置 |

### 広場のチャット管理

#### エリア連動チャットルーム

各広場に専用のチャットルームを紐付ける。
プレイヤーの座標が広場エリア内に入ったら自動的にチャットルームを切替。

```text
プレイヤーの位置を監視（既存の移動処理内で判定）

  ワールド自由エリア → "world" チャンネル
  公園広場エリア内   → "plaza_park" チャンネル
  ビーチエリア内     → "plaza_beach" チャンネル
```

| 状況 | チャンネル操作 |
| ---- | ---- |
| ワールド自由エリアにいる | `world` に参加 |
| 広場エリアに入る | `world` から離脱 → `plaza_{id}` に参加 |
| 広場エリアから出る | `plaza_{id}` から離脱 → `world` に参加 |

- 座標ベースの自動切替（入口を通過した時点で切り替わる）
- UIPanel のチャット表示ヘッダーに「公園広場」等の現在地名を表示
- `world` チャンネルのメッセージは広場内では見えない（広場の会話に集中）

#### チャットルームによる人数管理

- 各広場のチャットルームに `MaxUsers` を設定
- 上限に達した場合:
  - 入口付近でツールチップ「この広場は満員です（20/20人）」
  - 物理的には入れるが、チャットに参加できない → 実質的に入場制限
  - または入口にバリア（見えない壁）で物理的にもブロック
- 将来的に同じ広場を複数チャンネル化（ch1, ch2）も検討

### 広場のブロック管理

#### 編集権限

- 広場エリア内のブロックは**一般ユーザーは編集不可**
- サーバ側で広場エリアの座標範囲を管理し、`setBlock` RPC で拒否
- 管理者（特定ユーザーID）のみ広場内のブロックを編集可能
- 広場外の自由エリアは従来通り全員が編集可能

```go
// setBlock RPC 内の権限チェック
func isInPlaza(x, z int) *PlazaDef {
    for _, p := range plazas {
        if x >= p.MinX && x <= p.MaxX && z >= p.MinZ && z <= p.MaxZ {
            return &p
        }
    }
    return nil
}
// 広場内 && 管理者でない → 403
```

#### 広場の装飾配置

- 初期実装: 広場のブロック（塀・装飾）はサーバ起動時に自動生成
- 将来: 管理者が実行時に配置可能（管理者用ブロック編集モード）

### 広場への移動方法

#### テレポート（メニュー経由）

メニューに「広場一覧」を追加。選択するとその広場の入口座標にワープ。

```text
[メニュー] → [広場一覧パネル]
  公園広場     12人 / 50人
  ビーチ        5人 / 30人
  夜の街        8人 / 30人
  屋上          3人 / 20人
              → クリックでテレポート
```

- ローディング不要（同じワールド内のテレポートなので一瞬）
- 人数はチャットルームの参加者数を表示
- 満員の広場はグレーアウト or 「満員」表示

#### 歩いて移動

広場はワールド内にあるため、歩いて自然に辿り着くこともできる。
ワールドの中心付近に広場を集めて配置し、スポーン地点から近い場所にする。

### 広場のエリア判定

```typescript
// クライアント側 — プレイヤー位置から現在の広場を判定
interface PlazaArea {
  id: string;
  name: string;
  minX: number; minZ: number;
  maxX: number; maxZ: number;
  chatRoom: string;
}

function getCurrentPlaza(x: number, z: number, plazas: PlazaArea[]): PlazaArea | null {
  for (const p of plazas) {
    if (x >= p.minX && x <= p.maxX && z >= p.minZ && z <= p.maxZ) return p;
  }
  return null;
}
```

- 移動のたびに判定（軽い処理、毎フレームでなく移動確定時のみ）
- 広場に入った / 出た でチャットチャンネルを切替
- UI に現在地名を表示（「公園広場」「ワールド」等）

### 広場の雰囲気演出

ワールド内区域方式では空・光源はワールド共通のため、
**ブロック装飾で雰囲気を変える**のが基本戦略。

#### ブロックでできる演出（ワールド内区域で対応可）

| 広場テーマ | 演出方法 |
| ---- | ---- |
| 公園広場 | 緑色ブロックの地面、木（茶+緑の柱状ブロック）、花壇 |
| ビーチ | 砂色（黄土色）ブロックの地面、青ブロックの水面 |
| 夜の街 | 高い壁+天井ブロックで空を遮蔽、壁に明るい色ブロック（ネオン風） |
| 屋上 | Y軸の高い位置に床を配置、柵ブロックで囲む |

- **地面の色**: 広場エリアのブロックを砂色・石畳色等で塗り分け
- **壁の高さ**: 高い壁で囲むと外が見えず「屋内感」が出る
- **天井**: 天井ブロックで覆えば空が見えない → 洞窟・室内の雰囲気
- **Y軸**: 高台に広場を配置すれば「屋上」「展望台」の演出
- **色のトーン**: 暗い色のブロックで統一すれば夜っぽい雰囲気

#### 環境演出（空・光源・霧）— クライアント側ローカル切替

Babylon.js の `clearColor`・光源・霧は**シーン作成時だけでなくいつでも動的に変更可能**。
各クライアントが自分の位置に応じてローカルに環境を切り替えることで、
別空間にしなくても広場ごとの雰囲気演出が実現できる。

```typescript
// 広場エリアに入ったときにクライアント側で切替
function applyPlazaEnvironment(scene: Scene, light: DirectionalLight, env: PlazaEnv) {
  scene.clearColor = new Color4(...env.clearColor);   // 空の色
  light.intensity = env.lightIntensity;                // 明るさ
  light.direction = new Vector3(...env.lightDirection); // 光の方向
  scene.fogMode = env.fogMode;                         // 霧
  scene.fogColor = new Color3(...env.fogColor);
  scene.fogStart = env.fogStart;
  scene.fogEnd = env.fogEnd;
}
```

| 広場テーマ | clearColor | 光源 | 霧 |
| ---- | ---- | ---- | ---- |
| 公園広場 | 青空 (0.5, 0.7, 1.0) | 明るい (1.0) | なし |
| ビーチ | 夕焼け (1.0, 0.6, 0.3) | やや暗い (0.7) | 薄い霧 |
| 夜の街 | 夜空 (0.05, 0.05, 0.15) | 暗い (0.2) | 濃い霧 |
| 屋上 | 星空 (0.02, 0.02, 0.1) | 暗い (0.3) | なし |

**ポイント:**

- ワールド全体ではなく**各クライアントがローカルに切替**（他プレイヤーには影響しない）
- 各プレイヤーが自分のいる広場に合った環境を見る
- 広場外に出たらワールドのデフォルト環境に戻す
- 境界付近の違和感対策: 環境値をフェード（lerp で徐々に変化）させる

```typescript
// 境界付近のフェード例（毎フレーム、0.05ずつ補間）
const FADE_SPEED = 0.05;
scene.clearColor = Color4.Lerp(scene.clearColor, targetColor, FADE_SPEED);
light.intensity += (targetIntensity - light.intensity) * FADE_SPEED;
```

これにより別空間への切り出しは不要。ワールド内区域方式のまま
全広場で環境演出が可能。

### この方式のメリットまとめ

- **実装コスト最小**: Match 切替・シーン入替・ローディングが一切不要
- **空間の連続性**: 歩いて広場に辿り着ける自然な体験
- **既存の仕組みを活用**: チャンク・AOI・ブロック描画はそのまま
- **チャット分離可能**: エリア判定 + チャットルーム切替で実現
- **人数制限可能**: チャットルームの参加人数で制御
- **テレポートも可能**: メニューからワープすればローディングなしで瞬時移動
- **環境演出も可能**: 空・光源・霧をクライアント側でローカル切替（別空間不要）

## マイルーム（個人部屋）— Phase 1 以降

マイルームは**個人ごとに持つ小さいワールド**。
現在のワールド（1024×1024）の仕組みを小さくした別空間（16×16〜64×64）として実装する。
マイルームへの移動 = ワールドの切替。既存のワールド処理（ブロック描画、移動、環境設定）を
そのまま流用でき、実装がクリーンになる。

### なぜマイルームはワールド内区域にしないか

| 課題 | 詳細 |
| ---- | ---- |
| 土地の有限性 | 1024×1024 に広場+自由エリア+全員のマイルーム → ユーザー増加で破綻 |
| 土地割り当て | 誰にどの座標を割り当てるか、自動配置ロジックが複雑 |
| 休止ユーザー | ログインしなくなったユーザーの土地が永久に占有される |
| プライバシー | 鍵付き部屋（入室許可制）が困難。壁があっても歩いて入れる |
| サイズ変更 | 隣の部屋と干渉するため、後から広げられない |

広場は数が少ない（4〜10個）のでワールド内で問題ないが、
マイルームは**ユーザー数に比例して増える**ため別空間が必要。

### 1. ワールド切替（シーン管理）

マイルームへの移動 = **小さいワールドへの切替**。
同一の Babylon.js Scene を使い回し、メッシュとデータを入れ替える。

```text
ワールド → マイルームに入る（= 小さいワールドに切替）
  1. ワールド Match から leave
  2. 既存チャンクメッシュを dispose
  3. マイルーム Match に join → ブロックデータ受信
  4. マイルームのメッシュを生成（ワールドと同じ描画処理）
  5. カメラ位置をリセット
  6. 環境設定を適用（空・光源・霧）

マイルーム → ワールドに戻る（= 元のワールドに切替）
  1. マイルーム Match から leave
  2. マイルームメッシュを dispose
  3. ワールド Match に rejoin → チャンクデータ受信
  4. ワールドのメッシュを再生成
  5. 環境設定をワールドデフォルトに戻す
```

- Scene の再作成は不要（メモリ効率が良い）
- ブロック描画・移動処理・AOI など既存コードをそのまま流用
- 移動時に一瞬ローディング画面を表示
- 広場で実装した環境切替（clearColor, 光源, 霧）をそのまま流用
- マイルームは小さい（最大 64×64）ので AOI は不要、全データ送信で十分

### 2. マイルームのデータ管理

マイルームはワールドより**はるかに小さい**閉じた空間。

| 項目 | ワールド | マイルーム |
| ---- | ---- | ---- |
| サイズ | 1024×1024 | 16×16 〜 64×64（可変） |
| 永続化 | サーバメモリ + DB | Nakama Storage API（オーナーごと） |
| 同期方式 | AOI + チャンク差分 | 入室時に全データ送信 |
| 環境設定 | ワールド共通 | 部屋ごとにカスタマイズ可 |

```typescript
interface RoomData {
  roomId: string;          // UUID
  ownerId: string;         // Nakama user ID
  name: string;            // 部屋名
  sizeX: number;           // 幅（ブロック数）16〜64
  sizeZ: number;           // 奥行き（ブロック数）16〜64
  floorType: number;       // 床材テクスチャID
  wallType: number;        // 壁テクスチャID（将来）
  blocks: Uint8Array;      // ブロックデータ（ワールドと同じ形式）
  environment: PlazaEnv;   // 環境設定（広場と同じ形式を流用）
  createdAt: number;       // 作成日時
}
```

- **Nakama Storage API** に永続化（コレクション `rooms`、キー `room_{roomId}`）
- オーナーのみ書き込み可、他ユーザーは読み取り可

### 3. マイルームのチャット

```text
ワールド/広場: "world" or "plaza_{id}" チャンネル
マイルーム:   "room_{roomId}" チャンネル
```

- マイルーム入室時にチャンネルを切替
- 退室時にワールドのチャンネルに戻る

### 4. サーバ側（Nakama Match）

```text
ワールド Match（既存）: 移動・ブロック・AOI を処理（広場も含む）
マイルーム Match（新規）: 部屋内の移動・ブロック・プレゼンスを処理
```

- マイルーム Match は**オンデマンド作成**（誰かが入室したときに作成、全員退出で終了）
- ブロックデータは Match 開始時に Storage API から読み込み
- Match 終了時に変更があれば Storage API に書き戻し
- AOI 不要、チャンク差分同期不要（入室時に全データ送信）
- 同時接続上限を設定可能（例: 最大20人）

### 5. マイルームへの移動方法

**Phase 1（UIメニュー）:**

- メニューに「マイルーム」ボタン → 自分の部屋に移動
- ユーザーリストから「○○の部屋を訪問」→ 他人の部屋に移動
- 部屋内に「ワールドに戻る」ボタン

**Phase 3（ドアブロック）:**

- ワールド上にドアブロックを設置 → クリックでマイルームに移動
- ドアブロックにメタデータ（roomId）を紐付け
- 部屋内の出口ブロック → ワールドに戻る

## 6. 実装ステップ（段階的に進める）

### Phase 1: 共有の小部屋（MVP）— 最優先

誰でも行ける小さい部屋を、ワールド切替で実現する。
環境設定（光源等）は現在のワールドをそのまま流用。

- [ ] サーバ側: 部屋定義（Go 構造体: ID, 名前, サイズ固定 32×32）
- [ ] サーバ側: 部屋用 Match ハンドラ（ワールド Match を簡略化したもの）
- [ ] サーバ側: 部屋ブロックデータの永続化（Storage API）
- [ ] サーバ側: 部屋 Match のオンデマンド作成（入室時に起動、全員退出で終了）
- [ ] クライアント: Match 切替処理（leave → join）← **最大の技術的ハードル**
- [ ] クライアント: メッシュ入替（既存チャンク dispose → 部屋データで再生成）
- [ ] クライアント: ローディング画面
- [ ] クライアント: AOI 無効化（部屋は小さいので全データ送信）
- [ ] クライアント: チャット切替（`world` → `room_{id}`）
- [ ] クライアント: 「ワールドに戻る」ボタン（戻り先: 出発地点の座標を保持）

## 7. 検討事項・注意点

### パフォーマンス

- マイルーム移動時にローディング画面を表示（メッシュ生成に 0.5〜2秒）
- マイルームデータはクライアント側にもキャッシュ（IndexedDB）
- マイルーム Match のメモリ消費に注意（同時に100部屋開くと Go 側のメモリ増大）
- 広場はワールド内区域なのでパフォーマンス影響なし

### セキュリティ

- 広場: ブロック編集は管理者のみ（`setBlock` で座標チェック）
- マイルーム: オーナーのみ編集可（設定で開放可）
- マイルームの公開/非公開設定
- 荒らし対策: 訪問者のキック機能、ブロック設置のレート制限

### UX

- 広場にいるとき: ワールドの `world` チャットから広場チャットに自動切替
- マイルームにいるとき: ワールドのプレイヤーは見えない
- マイルームにいるプレイヤーはユーザーリストで「○○の部屋にいます」と表示
- マイルームから切断した場合: 再接続時にワールドに戻す

### Nakama の制約

- Storage API の1オブジェクト上限: 16KB（デフォルト）→ 大きい部屋は分割保存が必要
- Match 数の上限: Nakama のメモリに依存（1 Match ≈ 数KB〜数十KB）
- チャンネル数に実質的な上限はない

### 3種類の空間の比較

| 観点 | ワールド（自由エリア） | 広場（ワールド内区域） | マイルーム（別空間） |
| ---- | ---- | ---- | ---- |
| 所有者 | なし（共有） | 運営 | 個人ユーザー |
| 実体 | ワールド Match | ワールド Match（同一） | 専用 Match |
| 永続性 | サーバ常駐 | サーバ常駐 | 訪問時のみ起動 |
| 広さ | 1024×1024 | 32×32〜64×64 | 16×16〜64×64 |
| ブロック編集 | 全員可 | 管理者のみ | オーナー |
| 環境演出 | デフォルト | エリア別にローカル切替 | 部屋ごとにカスタマイズ |
| チャット | `world` | `plaza_{id}` | `room_{roomId}` |
| 移動方法 | 歩く | 歩く or テレポート | UIメニュー or ドアブロック |
| ローディング | なし | なし | あり（メッシュ入替） |
