# 設計: 認証システム

tommieChat の認証方式の設計方針。
現状のデバイス認証の限界を整理し、メール / OAuth2 への段階的移行を提案する。

[フレンドシステム設計](52-設計-フレンドシステム.md)の前提となる機能。
フレンド機能を有効化するには、アカウントが揮発しない正規認証が必須。

---

## 1. デバイス認証の限界

現状は Nakama の `authenticateDevice`（デバイス ID を localStorage + Cookie に保存）のみ。
PWA Safari↔standalone 間の引き継ぎ Cookie も実装済みだが、本質的な弱点は残る。

### デバイス認証で起きる問題

| 問題 | 詳細 |
| ---- | ---- |
| **アカウント消失** | ブラウザのストレージクリア / Cookie 削除でアカウントが消える。Cookie 引き継ぎ実装済みでも完全ではない |
| **マルチデバイス不可** | PC とスマホで同一アカウントを使えない。Pigg/Discord で「家でも外でも」が当たり前なのに不可能 |
| **復旧不能** | デバイス紛失・故障でアカウント永久ロスト。フレンドリスト・部屋・ブロックも全部消える |
| **なりすまし** | デバイス ID を盗めばそのアカウントになれる。フレンドが信用できなくなる |
| **同一人物判定不能** | 「あいつ最近来ないな」と思った人が別アカウントで来ていても気付けない |
| **垢 BAN 回避が容易** | ブロックされてもストレージ消せば別アカウント作り放題。ブロック機能が機能不全 |
| **フレンド検索手段がない** | UID しか識別子がない。「リアルで会った人を友達追加」ができない |

### フレンド機能と相性が悪い理由

フレンドの本質は「**この人**を覚えておく」こと。デバイス認証ではアカウント自体が揮発するので、

- フレンド申請を送った相手が翌日には消えている
- 自分のフレンド一覧が突然消える
- ブロックした相手が新アカウントで戻ってくる
- 「久しぶり！」が成立しない

→ **フレンドの前にちゃんとした認証が必須**。

## 2. 認証方式の比較

| 方式 | 復旧 | マルチデバイス | 実装コスト | UX | 匿名性 |
| ---- | ---- | ---- | ---- | ---- | ---- |
| **デバイス認証** (現状) | ✕ | ✕ | 済 | ◎（即ログイン） | ◎ |
| **メール+パスワード** | ◯ | ◯ | 中 | △（入力面倒） | ◯ |
| **メールマジックリンク** | ◯ | ◯ | 中 | ◯（パスワード不要） | ◯ |
| **OAuth2 (Google/Apple)** | ◎ | ◯ | 中 | ◎（ワンクリック） | △（プロバイダに依存） |
| **OAuth2 (X/Discord)** | ◯ | ◯ | 中 | ◎ | △ |
| **パスキー (WebAuthn)** | △ | △ | 高 | ◎ | ◯ |
| **電話番号 (SMS)** | ◯ | ◯ | 高（コスト） | △ | ✕ |

### Nakama がサポートする認証

Nakama は標準で以下を提供（Go プラグイン不要）:

- ✅ Device ID（実装済み）
- ✅ Email + Password
- ✅ Custom（任意の外部 ID 連携 → OAuth2 のブリッジに使う）
- ✅ Facebook / Google / Apple / Steam / Game Center / Facebook Instant Game
- ✅ アカウントの**リンク**機能（同一アカウントに複数認証手段を紐付け）

→ **Nakama 標準でほぼ全部できる**。Go コード追加なしでメール認証も Google/Apple 連携も実装可能。

## 3. 段階的移行戦略

今のデバイス認証を捨てるのではなく、その上に正規認証を「リンク」する。

### Phase 0: 現状（デバイス認証のみ）

```text
ブラウザ起動 → デバイスID生成 → 即ログイン → 仮アカウント
```

UX は最高だが、匿名・揮発・復旧不能。

### Phase 1: 正規認証の追加（アカウントリンク）

```text
ブラウザ起動 → デバイスIDで仮ログイン (現状維持)
                   ↓
        ユーザーが「アカウント保存」を押す
                   ↓
        メール or OAuth2 で認証
                   ↓
        Nakama LinkEmail / LinkCustom でデバイスIDアカウントに紐付け
                   ↓
        以後、別デバイスからもメール/Google でログイン可能
```

ポイント:

- **既存ユーザーを失わない** — デバイス ID → メール認証への移行で、フレンド・部屋・ブロックを維持
- **匿名利用も継続可能** — 「アカウント保存」を押さないユーザーは今まで通り
- **段階的アップグレード** — ゲームに慣れてから本登録を促せる

### Phase 2: フレンド機能の有効化

- 「アカウント保存済み」のユーザー同士のみフレンド申請可能
- 仮アカウント同士はフレンド不可（揮発するので無意味）
- フレンド申請を送るときに「アカウント保存しますか？」を促す

### Phase 3: 新規ユーザーは正規認証を推奨

- ログイン画面に Google/Apple ボタンを大きく配置
- 「ゲストとして遊ぶ」を小さく（既存のデバイス認証）
- ゲストは 7 日後に自動削除など

## 4. 推奨する認証手段の組み合わせ

tommChat の方向性（チャット主体 SNS）から考えて:

| 認証 | 採用 | 理由 |
| ---- | ---- | ---- |
| **Google OAuth2** | ◎ 第一推奨 | スマホユーザーの大半がアカウント持ち、ワンクリック、実装容易、無料 |
| **メール (マジックリンク)** | ◯ 推奨 | パスワード忘れ問題なし、Google/Apple 使えない人向け |
| **デバイス認証 (ゲスト)** | ◯ 維持 | 「とりあえず触ってみる」UX のため残す |
| **Apple Sign In** | △ 条件付き | **iOS App Store 公開時のみ必須**（12章参照）。Web 配信なら不要 |
| **X / Discord OAuth** | △ 後回し | ゲーマー向けには魅力だが OAuth プロバイダ追加コスト（10章参照） |
| **メール + パスワード** | ✕ 不要 | マジックリンクで代替可、パスワード管理はユーザー負担 |
| **電話番号 SMS** | ✕ 不要 | コストかかる、プライバシー懸念 |

## 5. 「仮アカウント → 正規アカウント」昇格 UX の工夫

Pigg・他 MMO の事例から:

1. **昇格のメリットを明示** — 「アカウントを保存するとデバイスをまたいで遊べます」「フレンド機能が使えます」「アバターが消えません」
2. **適切なタイミングで促す** — 初回ログイン直後ではなく、5回目のログイン時、フレンド申請しようとした時、部屋を作ろうとした時
3. **ワンクリック化** — Google ボタンだけで終わらせる
4. **「後で」を許可** — 強制せず、毎回ちょっとだけ促す
5. **昇格後のメリットを即体験** — 昇格直後にフレンド機能を解禁、特別バッジ付与等

## 6. データモデル

Nakama 標準の `users` テーブルに以下が記録される（追加開発不要）:

```text
users
├ id              ← 内部 UID（変わらない）
├ username        ← @user_xxx 形式
├ display_name    ← 表示名
├ create_time
├ update_time
└ ...

user_device       ← デバイス認証
├ id (device_id)
└ user_id

user_email        ← メール認証
├ email
├ password_hash
└ user_id

user_google       ← Google OAuth
├ google_id
└ user_id

user_apple        ← Apple Sign In
├ apple_id
└ user_id
```

同一の `users.id` に複数の認証レコードを紐付けられる（リンク機能）。

### 「保存済み」判定

クライアントから「自分のアカウントは保存済みか」を取得する必要がある。
判定は Nakama の `account.email`, `account.google_id`, `account.apple_id` のいずれかが
非空かどうか。これをラップする RPC または `getAccount` API の戻りで判定できる。

## 8. Phase 1 実装の見積もり

最小実装（Google OAuth2 のみ追加）:

- [ ] Nakama 側: Google OAuth2 クライアント ID / Secret を環境変数で設定（`GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET`）
- [ ] サーバ: `linkGoogleByCode` RPC を Go プラグインに追加（認可コード→ID トークン交換 → `LinkGoogle`）
- [ ] サーバ: 「保存済みアカウントか」フラグを判定する RPC（フレンド機能の前提チェック用）
- [ ] クライアント: `public/oauth-callback.html` + `public/js/oauth-callback.js`（postMessage で親に code を送る）
- [ ] クライアント: `public/js/google-oauth.js`（方式 B ポップアップ + 方式 A リダイレクトフォールバック）
  - クリックハンドラ内で**同期的に** `window.open` を呼ぶ（iOS Safari のポップアップブロッカー対策）
  - `popup === null || popup.closed` ならリダイレクト方式へフォールバック
  - リダイレクト前に `sessionStorage` に `roomId` / `position` / `cameraTarget` を保存
  - 戻った後に `sessionStorage` から状態復元 → 元の部屋・位置に再入室
- [ ] クライアント: `NakamaService.linkGoogleByCode(code)` ラッパ追加
- [ ] クライアント: 設定パネルに「Google でアカウント保存」ボタン + 「アカウント情報」表示（メール、リンク済み認証手段）
- [ ] CSP: nginx の `script-src` / `connect-src` / `frame-src` に Google ドメイン追加
  - `connect-src https://oauth2.googleapis.com`（トークン交換、最低限）
  - One Tap も併用するなら `script-src https://accounts.google.com` / `frame-src https://accounts.google.com`
- [ ] UI: ログイン画面リデザイン（ゲスト/Google/メールの 3 択）
- [ ] **ステージング検証**: `mmo-test.tommie.jp` 上で iPhone Safari（通常タブ）と PWA standalone の両方で実機確認
  - ポップアップが開くか
  - ブロックされた場合のリダイレクトフォールバックが動くか
  - リダイレクト戻り後に元の部屋・位置に復元できるか

## 9. 注意点・トレードオフ

### プライバシー

- OAuth2 はプロバイダ（Google/Apple）に「tommieChat を利用している」事実が記録される
- 完全匿名にこだわるユーザーにはデバイス認証（ゲスト）を残す
- メールアドレスはサーバに保存されるので適切に保護（パスワードハッシュ化、メール再送制限）

### CSP / OAuth2 ポップアップ

- Google OAuth2 はポップアップウィンドウまたはリダイレクトを使う
- 現在の CSP `connect-src 'self' wss://*.tommie.jp https://cdn.babylonjs.com` には Google のドメインを追加する必要あり
- `script-src` への追加も検討（Google Identity Services の JS を読み込む場合）

### 既存ユーザーの扱い

- Phase 1 移行時、既存のデバイス認証ユーザーは全員「未保存ゲスト」扱い
- 全員に「アカウントを保存してください」を促す
- 一定期間後、未保存ゲストの自動削除をするか、永続化するかの判断が必要

### 子供ユーザー

- COPPA（米国）/ GDPR-K（EU）対応: 13歳未満の OAuth2 利用に制限あり
- 国内サービスとして始めるなら年齢確認不要だが、将来的に必要

### 認証方式の追加コスト

- Apple Sign In: Apple Developer Program 加入必要（**$99/年、約14,800円**）→ 詳細は 12章
- Google OAuth2: 無料、Google Cloud Console で設定
- メール送信: SendGrid / AWS SES など（無料枠あり）

## 10. X (Twitter) OAuth のコスト詳細

X OAuth は他の OAuth プロバイダより**コストが高めで不安定**。
開発者個人として X を使っているなら自分用には魅力的だが、本番ユーザー向けの提供は慎重に判断する。

### 10.1 金銭的コスト（X API ティア）

Elon 買収後（2023〜）に大きく変わり、現在も変動が続いている。

| ティア | 月額 | 内容 |
| ---- | ---- | ---- |
| **Free** | $0 | 投稿のみ（月1500 post）、ユーザー認証は使えるが取得情報が極限定 |
| **Basic** | $200/月 | 読み取り API 一部、月10000 read |
| **Pro** | $5000/月 | 本格的な API アクセス |
| **Enterprise** | 要問合せ | 大規模利用 |

#### 「Sign in with X」だけならどうか

OAuth 2.0 PKCE で**ログイン認証だけ**ならティアの縛りは緩い。
ただし、ログイン後にユーザー情報（表示名、アイコン URL 等）を取りたい場合は
`GET /2/users/me` を呼ぶ必要があり、これが**読み取り API** に該当する。

- Free ティアでは `users/me` は呼べるが、過去に何度も仕様変更されている
- 「無料で Sign in できたはずが、ある日突然 Basic 必須になった」という事例が頻発
- **X のせいで OAuth 提供をやめたサービスは多数**（一部の開発ツール、Web フォーム連携サービス等）

→ **「現在は無料でも、明日から有料化されるリスクが常にある」**のが X 認証の最大コスト。

#### Google / Apple との比較

| プロバイダ | API 料金 | 安定性 |
| ---- | ---- | ---- |
| Google OAuth2 | **完全無料** | 安定（10年以上仕様維持） |
| Apple Sign In | **完全無料**（Apple Developer Program $99/年は別途必要） | 安定 |
| X OAuth2 | **無料 or $200/月**（変動） | 不安定（年に数回仕様変更） |

### 10.2 実装コスト

#### Nakama の標準サポート状況

| プロバイダ | Nakama 標準 API |
| ---- | ---- |
| Google | ✅ `authenticateGoogle(token)` 一発 |
| Apple | ✅ `authenticateApple(token)` 一発 |
| Facebook | ✅ `authenticateFacebook(token)` |
| Steam | ✅ `authenticateSteam(token)` |
| **X / Twitter** | ❌ **なし** |
| Discord | ❌ なし |

X は Nakama 標準に**ない**ので、自前で OAuth 2.0 フローを実装し、`authenticateCustom` で繋ぐ必要がある。

#### 自前実装で必要な作業

```text
1. クライアント側
   - X OAuth 2.0 PKCE フロー実装（code_verifier 生成、認可リクエスト）
   - 認可コールバック処理（リダイレクト or popup）
   - state パラメータ検証（CSRF対策）

2. サーバ側（Go プラグイン追加）
   - 認可コードを access_token に交換する RPC（X の token endpoint 叩く）
   - access_token で users/me を呼んで X user_id を取得
   - X user_id を customId として nakama.AuthenticateCustom

3. 設定
   - X Developer Portal でアプリ登録
   - Callback URL 登録（dev/staging/prod それぞれ）
   - Client ID / Client Secret を環境変数管理

4. テスト
   - OAuth フロー全体の E2E テスト
   - トークン期限切れ時のリフレッシュ
```

Google なら 1〜2 時間で済む実装が、X は **1〜2 日**かかる。

### 10.3 運用コスト

X 特有の運用負担:

- **API 仕様変更が頻繁** — 年に数回 Breaking Change がある
- **アプリ審査** — Developer Portal でアプリ登録時に用途説明を求められる（数日〜数週間）
- **アカウント停止リスク** — Developer アカウントが突然停止される事例あり
- **コールバック URL の制限** — 登録した URL のみ。dev/staging/prod それぞれ別アプリ登録が必要な場合あり
- **レート制限** — 認証エンドポイントにもレート制限あり、ユーザー急増時に問題化
- **権限スコープの変更** — `read:users` だったのが `users.read` になる等、地味に壊れる

Google / Apple は安定運用できる:

- Google: API 仕様が安定、ドキュメントが整備、Stack Overflow に情報豊富
- Apple: 厳しいが規約は明確、変更事前通知あり
- X: 「今動いてるから OK」が来週通用するか分からない

### 10.4 ユーザー側の制約

X アカウント特有の問題:

| 問題 | 詳細 |
| ---- | ---- |
| アカウント凍結 | X のアカウント凍結が tommChat ログイン不可に直結 |
| サブアカウント文化 | ユーザーが複数 X アカウントを持っており「どれで登録したっけ？」問題 |
| メール非公開 | X アカウントにメールが紐付いていないユーザーもいる |
| 年齢層 | Google/Apple より若年層が薄い |
| 国際性 | 日本では強いが、地域によってはほぼ使われていない |

### 10.5 結論: X OAuth の現実的な選択肢

#### 案 A: 当面見送る

- 金銭・実装・運用すべてのコストが Google より高い
- API 仕様が不安定で「ある日壊れる」リスク
- ユーザーの大半は Google アカウントも持っているので、Google で代替できる
- フレンド機能の開始だけなら Google + メールで十分

#### 案 B: Phase 後半で追加

```text
Phase 1: Google + メール（フレンド機能解禁の最低ライン）
Phase 2: Apple Sign In（iOS 配布時に必須）
Phase 3: ↑ 安定運用が確認できたら X OAuth 検討
Phase 4: Discord OAuth（ゲーマー層向け）
```

X 追加の判断基準:

- ユーザーから「X でログインしたい」という要望が複数件来てから
- API ティアが安定したことを確認してから
- ROI が見込める規模になってから

#### 案 C: どうしても X 認証したい場合の最小実装

開発者本人だけが使うなら、コストを最小化する方法:

1. X Developer Portal で Free ティアのアプリを作成
2. OAuth 2.0 PKCE で `users/me` だけ呼ぶ（追加 API は使わない）
3. Nakama `authenticateCustom` に `x_{user_id}` を渡す
4. 表示名・アイコンは初回ログイン時にコピーして以後 X API は呼ばない（API 依存度を最小化）
5. 仕様変更で壊れたら一旦無効化、メール認証に誘導

これなら**実質 $0** で動かせるが、いつ壊れるか分からない前提で運用する必要がある。

### 10.6 ハイブリッド運用の推奨

開発者個人として X を使っている前提なら、**ドッグフーディング用に「案 C」で実装する**のは理にかなっている。

ただし、tommChat の正規認証としてユーザーに提供するなら**案 A（当面見送り）or 案 B（Phase 3 以降）**が安全。

「自分のドッグフーディング用に使いつつ、本番ユーザーには Google を推す」というハイブリッド運用がベスト。

## 12. Apple Developer Program 必須要件

「Sign in with Apple」を提供するには **Apple Developer Program ($99/年、約14,800円) への加入が必須**。
無料の Apple ID では Service ID を作成できず、Sign in with Apple の設定画面が塞がれている。

### 12.1 必要なもの

| 項目 | 必須 | コスト |
| ---- | ---- | ---- |
| Apple ID | ◯ | 無料 |
| Apple Developer Program | ◎ | **$99/年（約 14,800 円）** |
| Service ID（OAuth 用識別子） | ◎ | Developer Program 加入後に作成可 |
| Sign in with Apple Key（JWT 署名用 .p8 鍵） | ◎ | 同上 |
| 検証済みドメイン | ◎ | DNS TXT レコードで証明 |

無料の Apple Developer アカウント（サインアップだけ）でできること:

- ✅ Xcode で自分のデバイスにアプリをインストール
- ❌ Service ID を作れない
- ❌ Sign in with Apple Key をダウンロードできない
- ❌ App Store に公開できない

→ Sign in with Apple の Service ID 作成画面が「Developer Program に加入してください」で塞がれているため、回避不能。

### 12.2 $99/年で何が手に入るか

```text
Apple Developer Program ($99/年)
├ App Store 公開権
├ TestFlight ベータ配布
├ Sign in with Apple ←★ これが目当て
├ Push Notification (APNs)
├ MapKit / CloudKit 等の各種 API
├ ベータ版 OS / Xcode へのアクセス
└ 全 Apple ハードでのアプリ実機テスト
```

iOS アプリを出さなくても、**Sign in with Apple だけ目当てで $99/年払う**ことは可能。

### 12.3 tommieChat にとって必須かどうか

#### Web のみで配信する場合 → 必須ではない

- Apple Sign In がなくても tommChat は普通に動く
- Google + メール認証だけで十分
- Apple ユーザーは Google アカウントも大体持っている

#### iOS アプリ / PWA を App Store に出す場合 → 実質必須

Apple の App Store Review Guideline **4.8** に以下のルールがある:

> アプリで第三者のログインサービス（Google, Facebook, Twitter 等）を主要な認証手段として提供する場合、**Sign in with Apple も提供しなければならない**。

つまり:

- tommChat を iOS App Store に出す
- 既に Google ログインを実装している
- → **Apple Sign In も必須**になり、Developer Program 加入必須

例外:

- 自社アカウントのみ（メール+パスワードのみ）の場合は不要
- Apple Sign In 単独でも不要（他に何もない場合）
- 教育機関・医療機関向けで他の認証方式を使う場合は不要

### 12.4 tommieChat の現状を踏まえた判断

| シナリオ | Apple Sign In | Developer Program |
| ---- | ---- | ---- |
| 当面 Web のみ（mmo.tommie.jp） | 不要 | 不要 |
| PWA を「ホーム画面に追加」してもらう | 不要 | 不要 |
| iOS App Store に公開 + Google ログインあり | **必須** | **必須** |
| Android のみ Play ストア公開 | 不要 | 不要 |

### 12.5 推奨

現状の方向性（Web/PWA 配信、App Store 進出は未定）なら、**Apple Sign In は当面見送って OK**。

#### Phase 別の優先度

```text
Phase 1: Google OAuth + メール認証（フレンド機能解禁）
         → Developer Program 不要、$0/年

Phase 2: ↑ で運用してユーザー数を見る

Phase 3: iOS App Store 公開を判断
         → 出すなら Apple Sign In 追加 + Developer Program 加入
         → 出さないなら Apple Sign In はスキップ可
```

#### Google で十分な理由

- iOS ユーザーも大半が Google アカウントを持っている
- Web ブラウザ経由なら Apple Sign In がなくても誰も困らない
- 「Apple Sign In がないからログインできない」というユーザーは現実的にはほぼゼロ
- $99/年を払う ROI が出るのは、iOS App Store 公開で明確な収益見込みがあるとき

### 12.6 結論

| 質問 | 答え |
| ---- | ---- |
| Apple Sign In に Developer Program 必須？ | **はい、$99/年必須** |
| Web のみなら払う価値ある？ | **ない**（Google で代替可） |
| iOS App Store 進出時は？ | **必須**（Guideline 4.8） |
| tommieChat で今すぐ必要？ | **不要** |

## 13. OAuth フロー方式の選定（リダイレクト vs ポップアップ vs One Tap）

OAuth2 を組み込むときに必ず迷うのが「認証画面への遷移方法」。
3 つの方式があり、tommieChat のようにリアルタイム WebSocket を維持したいゲームでは選定が重要。

### 13.1 3 つのフロー方式

#### 方式 A: フルページリダイレクト（伝統的）

```text
tommChat ─→ accounts.google.com にページ遷移 ─→ 認証 ─→ tommChat にリダイレクト戻り
```

| 項目 | 影響 |
| ---- | ---- |
| ページ状態 | **完全アンロード**。Babylon シーン破棄、メモリリセット |
| WebSocket | **切断される**（match から退出扱い） |
| 他プレイヤーから見え方 | 一旦ゴースト化 → 退室 → 再入室 |
| Nakama セッション JWT | localStorage / Cookie に残るので、戻った後に**自動再ログイン可能** |
| 実装難度 | 低い（純粋に URL 遷移） |
| iOS Safari | ◎ 動く |
| 欠点 | ゲーム中の認証で「会話の途中で消える」体験になる |

#### 方式 B: ポップアップウィンドウ（推奨）

```text
tommChat（裏で動き続ける）
  └─ 子ウィンドウで accounts.google.com を開く
        └─ 認証完了後、postMessage で親に code を通知
        └─ 子ウィンドウは自動で閉じる
```

| 項目 | 影響 |
| ---- | ---- |
| ページ状態 | **維持**。Babylon シーンも WebSocket もそのまま |
| 他プレイヤーから見え方 | **何も起きない**（完全シームレス） |
| 実装難度 | 中（postMessage、ポップアップブロッカー対策） |
| iOS Safari | △ ポップアップブロッカーが厳しめ。ユーザー操作直後の `window.open` は通る |
| 欠点 | モバイルでポップアップが小さい / ブロックされる可能性 |

#### 方式 C: Google One Tap（GIS — Google Identity Services）

```text
tommChat 内に小さな「Google でサインイン」ダイアログが出る
  └─ クリック → インライン認証（ページ遷移なし）
  └─ ID Token を直接 JS に返す
```

| 項目 | 影響 |
| ---- | ---- |
| ページ状態 | **完全維持**（iframe 内で完結） |
| 他プレイヤーから見え方 | 何も起きない |
| 実装難度 | 低（Google が提供する `<script>` を読み込むだけ） |
| 対応プロバイダ | **Google のみ**（Apple / X には同等の仕組みなし） |
| iOS Safari | ◎ 動く |
| 欠点 | サードパーティ Cookie ブロックで動かない場合あり、Google 専用 |

### 13.2 比較表

| 観点 | A: リダイレクト | B: ポップアップ | C: One Tap |
| ---- | ---- | ---- | ---- |
| ゲーム接続維持 | ❌ 切断 | ✅ 維持 | ✅ 維持 |
| 実装コスト | 低 | 中 | 低 |
| ブラウザ対応 | ◎ | ◯ | △ |
| モバイル UX | ◯ | △ | ◎（Google 限定） |
| iOS Safari | ◎ | △ | ◎ |
| プロバイダ | 全部 OK | 全部 OK | Google のみ |
| 推奨度 | フォールバック | **第一推奨** | Google 専用の追加策 |

### 13.3 Nakama セッションは消えない

重要な前提として、**OAuth2 のリダイレクトは Nakama セッションに直接影響しない**。

- Nakama ログイン = JWT トークン発行（localStorage / Cookie に保存）+ WebSocket 接続
- OAuth2 は Google 側の認可フローであり、tommChat の JWT を消すわけではない
- 方式 A でページが破棄されても、戻ってきたときに JWT を読み直せば**再ログイン不要**
- ただし WebSocket は切れているので、再接続して match に再入室する必要がある

つまり方式 A の問題は「ログアウトする」ことではなく、「**WebSocket とゲーム状態が一旦リセットされる**」こと。

### 13.4 tommieChat への推奨: 方式 B（ポップアップ）

ゲーム接続を保ったまま認証できるのが最大のメリット。

#### サンプル実装（クライアント側）

```javascript
// public/js/google-oauth.js
async function linkGoogleAccount() {
    const params = new URLSearchParams({
        client_id: GOOGLE_CLIENT_ID,
        redirect_uri: location.origin + "/oauth-callback.html",
        response_type: "code",
        scope: "openid email profile",
        state: crypto.randomUUID(),
    });
    const popup = window.open(
        `https://accounts.google.com/o/oauth2/v2/auth?${params}`,
        "google-oauth",
        "width=500,height=600"
    );
    return new Promise((resolve, reject) => {
        const handler = (e) => {
            if (e.origin !== window.location.origin) return;
            if (e.data.type === "google-oauth-code") {
                window.removeEventListener("message", handler);
                popup.close();
                resolve(e.data.code);
            }
        };
        window.addEventListener("message", handler);
    });
}
```

#### コールバックページ（最小）

```html
<!-- public/oauth-callback.html -->
<!DOCTYPE html>
<html><head><meta charset="utf-8"><title>認証中...</title></head>
<body>認証中...
<script src="/js/oauth-callback.js"></script>
</body></html>
```

```javascript
// public/js/oauth-callback.js
const params = new URLSearchParams(location.search);
const code = params.get("code");
if (code && window.opener) {
    window.opener.postMessage({ type: "google-oauth-code", code }, location.origin);
}
```

呼び出し側:

```typescript
const code = await linkGoogleAccount();
// Nakama 側 RPC で code → access_token 交換 → linkGoogle
await nakama.rpc("linkGoogleByCode", { code });
// WebSocket は維持されたまま、リンクだけ完了
```

### 13.5 必要な CSP 追加

ポップアップ方式でも Google のドメインを許可する必要がある:

```text
script-src  'self' 'wasm-unsafe-eval' https://cdn.babylonjs.com https://accounts.google.com
connect-src 'self' wss://*.tommie.jp https://cdn.babylonjs.com https://oauth2.googleapis.com
frame-src   https://accounts.google.com
```

- `script-src` — One Tap (方式 C) を併用するなら必須。ポップアップだけなら不要
- `connect-src` — トークン交換エンドポイント
- `frame-src` — One Tap の iframe 用。ポップアップだけなら不要

### 13.6 方式 A をフォールバックとして残す

ポップアップがブロックされる環境（iOS Safari の一部、組織ポリシー）に備え、方式 A も用意する:

```typescript
async function linkGoogleAccount() {
    const popup = window.open(...);
    if (!popup || popup.closed) {
        // ポップアップブロックされた → リダイレクト方式に切替
        sessionStorage.setItem("oauth_resume", JSON.stringify({
            roomId: currentRoomId,
            position: player.position,
            cameraTarget: camera.target,
        }));
        location.href = `https://accounts.google.com/...`;
        return;
    }
    // ...ポップアップ処理
}
```

戻った後:

```typescript
const resume = sessionStorage.getItem("oauth_resume");
if (resume) {
    const state = JSON.parse(resume);
    sessionStorage.removeItem("oauth_resume");
    // 元の部屋・位置・カメラに復元
    await rejoinRoom(state.roomId, state.position);
}
```

### 13.7 まとめ

| 推奨度 | 方式 | 用途 |
| ---- | ---- | ---- |
| ★★★ | **B: ポップアップ** | デフォルト。WebSocket を維持してシームレスに認証 |
| ★★ | A: リダイレクト | ポップアップブロック時のフォールバック。`sessionStorage` で状態復元 |
| ★ | C: One Tap | Google 専用の追加 UX。「初回ログイン画面」だけで使う |
