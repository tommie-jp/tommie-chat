# セキュリティレビュー

2026-04-08 実施。サーバー（Go）・クライアント（TypeScript）・インフラ（Docker/nginx）の3領域を確認。

## 前提

- Web アプリである以上、クライアントは信用できない（DevTools、ブックマークレット、プロキシ等で誰でも RPC 実行可能）
- iOS Safari でもコンソールなしで RPC 実行は可能（Mac Web Inspector、Shortcuts アプリ等）
- これは tommieChat 固有の弱点ではなく、全 Web アプリ共通の制約
- 防御の基本は **サーバー権威 + レート制限 + BAN 運用**

## 深刻度: 高

### H-1. XSS 脆弱性（クライアント＋サーバー） ✅ 対応済み

`escapeHtml` 関数が存在せず、ユーザー入力が `innerHTML` にエスケープなしで挿入されていた。

**対応内容（2026-04-08）:** 2層防御で対策。

**Layer 1 — サーバー側（`main.go`）:**

- `sanitizeText()`: `html.EscapeString` + 制御文字除去。displayName, チャットメッセージに適用
- `sanitizeColor()`: `#[0-9a-fA-F]{3,8}` のみ許可。nameColor に適用
- `sanitizeTextureUrl()`: `/s3/avatars/` で始まるパスのみ許可。textureUrl に適用
- 適用箇所: `opInitPos`, `opDisplayName`, `opAvatarChange`, `opChat`, `rpcUpdateDisplayName`, `rpcCreateRoom`

**Layer 2 — クライアント側（`src/utils.ts`）:**

- `escapeHtml()` / `sanitizeColor()` を作成
- 適用箇所: `UIPanel.ts`（チャット表示、ユーザーリスト、システムメッセージ）、`DebugOverlay.ts`（uid表示）、`NPCSystem.ts`（NPC チャット）

### H-2. pprof 認証なし公開（サーバー）

`net/http/pprof` がポート 6060 で認証なしに起動。メモリダンプから機密情報漏洩の可能性。

**該当:** `main.go` pprof 起動箇所

**対策:** Docker でポート非公開にするか、認証ミドルウェアを追加。本番 prod.yml では `127.0.0.1` バインド済みだが、コード上は無防備。

### H-3. rpcDeleteUsers に管理者権限チェックなし（サーバー） ✅ 対応済み

`ENABLE_TEST_RPC=true` 時、認証済みユーザーなら誰でも任意ユーザーを削除可能。

**対応内容（2026-04-08）:** H-4 と合わせて `docker-compose.prod.yml` で `ENABLE_TEST_RPC=false` を明示。本番では `deleteUsers` RPC 自体が登録されない。

### H-4. ENABLE\_TEST\_RPC=true がベース設定にハードコード（インフラ） ✅ 対応済み

`docker-compose.yml` の環境変数に `ENABLE_TEST_RPC=true` があり、本番用 compose で上書きされていなかった。

**対応内容（2026-04-08）:**

- `docker-compose.prod.yml` に `ENABLE_TEST_RPC=false` を追加
- `docker-compose.yml` のコメントを「dev 専用」に明確化

### H-5. /s3/ パスのアクセス制限なし（インフラ） ✅ 対応済み

nginx が `/s3/` を MinIO にプロキシしていたが、全 API が無制限で公開されていた。

**対応内容（2026-04-08）:** `doDeploy.sh` の nginx 設定を修正。

- `/s3/avatars/` の GET/HEAD のみ許可（`limit_except`）
- `/s3/` の他パスは全て 403 で拒否
- `proxy_set_header Host` を `minio:9000` に変更（S3 互換のため）
- `vite.config.ts` の dev proxy も `/s3/avatars` のみに制限

### H-6. dev 用ポートが 0.0.0.0 バインド（インフラ）

`docker-compose.dev.yml` で PostgreSQL (5432)、Nakama 管理コンソール (7351)、Prometheus (9090)、MinIO (9000/9001) が全て外部公開。

**該当:** `docker-compose.dev.yml` ports

**対策:** 全ポートを `127.0.0.1:xxxx:xxxx` に変更。

## 深刻度: 中

### M-1. RPC にレート制限なし（サーバー） ✅ 対応済み

マッチメッセージ（move, block, chat）にはレート制限があるが、RPC 関数にはない。`syncChunks` や `getGroundChunk` の連続呼び出しでサーバー負荷増大。

**対応内容（2026-04-08）:** トークンバケット方式のユーザーごとレート制限を導入。

- `withRateLimit()` ラッパー関数で全 `RegisterRpc` を一括ラップ
- デフォルト: 秒間10回、バースト上限20回（`RATE_LIMIT_RPC`, `RATE_LIMIT_RPC_BURST` で変更可能）
- 超過時は `RESOURCE_EXHAUSTED` エラーを返し、RPC 関数は実行されない
- 古いバケットは5分間未使用で自動GC

### M-2. ワールド作成に上限なし（サーバー）

`createRoom` に作成数制限がない。各ワールドがチャンクマップをメモリ保持するため、大量作成でメモリ枯渇。

### M-3. displayName / textureUrl / nameColor にバリデーションなし（サーバー） ✅ 対応済み

長さ制限・フォーマットチェックがない。巨大文字列や不正な URL を設定可能。

**対応内容（2026-04-08）:** 入力長さ制限を導入。

- `truncateRunes()` で Unicode 安全な文字数切り捨て（sanitize 前に適用）
- displayName: 最大30文字（`MAX_DISPLAY_NAME_LEN`）— `opInitPos`, `opDisplayName` で切り捨て、`rpcUpdateDisplayName` ではエラー返却
- textureUrl: 最大128バイト（`MAX_TEXTURE_URL_LEN`）— `opInitPos`, `opAvatarChange` で切り捨て
- loginTime: 最大30文字にハードコード切り捨て
- nameColor: 既存の `sanitizeColor()` で `#[0-9a-fA-F]{3,8}` のみ許可（実質9バイト上限）
- `rpcCreateRoom` の部屋名も `maxDisplayNameLen` を参照するように統一

### M-4. 未知オペコードがレート制限なしで全員ブロードキャスト（サーバー）

定義外のオペコードのメッセージがそのまま全接続クライアントに転送される。

### M-5. syncChunks で大量データ生成可能（サーバー）

最大 64x64=4096 チャンク分（約 6MB）のレスポンスを生成可能。

### M-6. 配列サイズ制限なし（サーバー） ✅ 対応済み

`opProfileRequest` の sessionIds、`rpcGetDisplayNames` の userIds に配列サイズ上限がない。

**対応内容（2026-04-08）:** 配列サイズ上限を導入。

- `maxArraySize=100`（`MAX_ARRAY_SIZE` で変更可能）
- `opProfileRequest`: sessionIds を最大100件に切り捨て
- `rpcGetDisplayNames`: userIds を最大100件に切り捨て
- 超過分は無視（エラーは返さない）

### M-7. XSS + localStorage でアカウント乗っ取り（クライアント） — H-1 対応により緩和

XSS 脆弱性と組合せて `localStorage` のデバイスID を窃取し、`authenticateDevice` で他人のアカウントにログイン可能。
H-1 の XSS 対策により、この攻撃経路は大幅に緩和された。

### M-8. nginx セキュリティヘッダーなし（インフラ） ✅ 対応済み

以下のヘッダーが未設定だった:

- `Content-Security-Policy`
- `Strict-Transport-Security` (HSTS)
- `X-Frame-Options`
- `X-Content-Type-Options`
- `Referrer-Policy`

**対応内容（2026-04-08）:** `doDeploy.sh` の nginx.conf 生成に4ヘッダーを追加。

- `X-Content-Type-Options: nosniff` — MIME スニッフィング防止
- `X-Frame-Options: DENY` — iframe 埋め込み拒否
- `Referrer-Policy: strict-origin-when-cross-origin` — URL 漏洩制限
- `Content-Security-Policy` — インラインスクリプト禁止、接続先制限（XSS 被害軽減の本丸）

HSTS はホスト側 nginx（certbot 管理）の管轄のため未対応。

### M-9. nginx レート制限なし（インフラ）

`limit_req` が設定されていない。DDoS やブルートフォースに対する nginx レベルの防御がない。

### M-10. デフォルト認証情報のフォールバック（インフラ）

`.env` 未設定時に PostgreSQL (`localdev`)、MinIO (`minioadmin/minioadmin`)、Nakama コンソール (`admin/password`) が使用される。`doDeploy.sh` 経由なら安全だが、`docker compose up -d` 直接実行時にリスク。

### M-11. 本番 Origin チェックに localhost 含む（インフラ）

nginx の Origin チェックで `localhost` / `127.0.0.1` が許可されている。本番環境では不要。

### M-12. ownerUID 空のワールドは誰でも削除可能（サーバー）

`deleteRoom` の権限チェックで `OwnerUID` が空文字列の場合、チェックがスキップされる。

### M-13. エラーメッセージに DB 内部情報（サーバー）

`rpcUpdateDisplayName` で `err.Error()` をそのままクライアントに返しており、DB 構造が漏洩する可能性。

## 深刻度: 低

### L-1. 移動速度チェックなし（サーバー）

座標はワールド範囲内にクランプされるが、移動速度の検証がない。テレポート可能。チャット主体なので影響は限定的。

### L-2. ブロック配置にオーナーシップなし（サーバー）

認証済みユーザーなら任意チャンクにブロック配置/削除可能。荒らし対策がない。

改善案 — setBlock の楽観的ロールバック:
クライアントはブロック配置時に即座に描画（楽観的更新）するが、サーバーがレート制限等で拒否した場合、クライアント側に不整合が残る。対策として：

- サーバー: レート制限で弾いた時に `OP_BLOCK_REJECT` を送信者に返す（新opcode）
- クライアント: 受信したら該当ブロックをロールバック（配置前の状態に戻す）

現状は `RATE_LIMIT_BLOCK=20`（秒間20回）に引っかかるのは異常操作のみで、正規ユーザーには影響しない。荒らしに対しては不整合が残ること自体が罰則的に機能するため、優先度は低い。

### L-3. loginRateLimiter がグローバル（サーバー）

全ユーザー合算で秒間 50 回。攻撃者が枠を使い切り、他ユーザーのログインを阻害可能。

### L-4. TLS プロトコル・暗号スイート未指定（インフラ）

`ssl_protocols`、`ssl_ciphers` の明示指定がなく、nginx デフォルトに依存。

### L-5. server_tokens off 未設定（インフラ）

レスポンスヘッダーに nginx バージョン情報が露出。

### L-6. certbot メール未設定（インフラ）

`--register-unsafely-without-email` で実行。証明書期限切れの通知が届かない。

## 良好な点

- SQL パラメータバインド使用 ✓
- チャットの username/userId はサーバーが付与（なりすまし不可）✓
- 本番ポートは `127.0.0.1` バインド ✓
- `.env` は `.gitignore` 対象 ✓
- WSS は HTTPS 時に自動適用 ✓
- Cookie に `Secure`（HTTPS時）/ `SameSite=Strict` 設定 ✓
- デプロイスクリプトでパスワードをランダム生成 ✓
