# 検討事項: UGC（ユーザー生成コンテンツ）

プレイヤーが写真・3Dモデル・音声をアップロードして使用する機能の設計メモ。

---

## 対応フォーマット

| 種類 | フォーマット | 用途 |
| ---- | ---- | ---- |
| 写真 | JPEG, PNG | 現在の板ポリテクスチャとして使用 |
| 3Dモデル | .glb (glTF) | 将来の3Dアバター |
| 3Dモデル | .vrm | VRoid 等で作成したアバター |
| 音声 | MP3, OGG | エモーション効果音（3〜5秒） |

---

## セキュリティ

| リスク | 対策 |
| ---- | ---- |
| 悪意あるファイル | サーバ側でフォーマット検証（マジックバイト確認、拡張子だけ信頼しない） |
| 巨大ファイル | サイズ上限（写真: 2MB、3D: 10MB、音声: 100KB 等） |
| 不適切画像/音声 | 初期は手動モデレーション + 通報機能。規模拡大後に AI 検出 |
| XSS / スクリプト埋め込み | SVG 禁止。画像は再エンコード（sharp 等でリサイズ）して元データ破棄 |
| 3Dモデルの悪用 | ポリゴン数上限（例: 10,000）、テクスチャサイズ上限 |
| 他人の著作物 | 利用規約で自己責任を明記 |

---

## ストレージ構成

```
アップロード → nginx → Node.js API → 検証・変換 → S3 / ローカルストレージ
                                         ↓
                                   Nakama Storage に URL を保存
```

Nakama Storage にバイナリを直接保存しない（DB が膨らむ）。
ファイル本体は外部ストレージに置き、URL だけを Nakama に保存する。

| 規模 | ストレージ |
| ---- | ---- |
| 小規模（〜1000人） | nginx で静的配信（`/uploads/` ディレクトリ） |
| 中規模 | S3 互換（MinIO セルフホスト、または Cloudflare R2） |
| 大規模 | CDN 経由の S3 |

---

## 写真アバターの処理パイプライン

```
ユーザー → JPEG/PNG アップロード
  → マジックバイト検証（本当に画像か）
  → リサイズ（256x384 等、統一サイズ）
  → KTX2 変換（GPU 圧縮テクスチャ、現在の形式と統一）
  → ストレージ保存
  → Nakama Storage に URL 記録
  → 他プレイヤーに textureUrl を通知（AOI_ENTER で配信済み）
```

現在の仕組み（`textureUrl` を `AOI_ENTER` で送信）がそのまま使える。

---

## 3D モデルの処理パイプライン

```
ユーザー → .glb/.vrm アップロード
  → glTF バリデーション（glTF-Validator）
  → ポリゴン数チェック（上限超過は拒否）
  → テクスチャサイズチェック
  → ストレージ保存
  → Nakama Storage に URL + メタデータ記録
```

---

## エモーション音声

エモーション（拍手、笑い、挨拶等）実行時に短い音声を再生する機能。

### 方式

| 方式 | 内容 | リスク |
| ---- | ---- | ---- |
| A. プリセットのみ | 運営が用意した音声から選択 | なし |
| B. ユーザーアップロード | 自分の声や効果音を録音/アップ | 中（不適切音声） |

Phase 1 ではプリセットのみで実装し、仕組みが安定してからアップロードを追加する。

### 制限

| 項目 | 値 |
| ---- | ---- |
| 最大長さ | 3〜5秒 |
| 最大サイズ | 100KB |
| フォーマット | MP3 / OGG（ブラウザ互換） |
| 再生範囲 | AOI 内のプレイヤーのみ |

短い音声のため著作権リスクは低い（楽曲として成立しない長さ）。
ただし不適切な音声（暴言等）の通報機能は必須。

### 処理フロー

```
エモーションボタン押下
  → クライアントが OP_EMOTE 送信（emoteId + audioUrl）
  → サーバが AOI 内に配信
  → 受信側が audioUrl を再生（Web Audio API）
  → 音量はプレイヤー間の距離で減衰
```

距離による音量減衰で 3D 空間の没入感を演出する。

### 通信事業者届出について

1対1のボイスメッセージは電気通信事業の届出が必要になる可能性がある。
エモーション音声は「不特定多数への一方的な再生」であり、通信ではなく
コンテンツ配信に近いため届出は不要と考えられる。

## パフォーマンス対策

1000人が全員カスタムテクスチャだと重いため、以下の対策が必要。

| 対策 | 内容 |
| ---- | ---- |
| テクスチャアトラス | 近くにいるアバターのテクスチャを1枚にまとめる |
| 遅延読み込み | AOI に入ったときに初めてテクスチャをダウンロード |
| キャッシュ | 一度読んだテクスチャはブラウザキャッシュ（Cache-Control: immutable） |
| サムネイル | 遠距離は低解像度版を使用 |
| デフォルト表示 | ダウンロード完了まではデフォルトテクスチャで表示 |

## 実装の優先順

| 段階 | 内容 |
| ---- | ---- |
| Phase 1 | 写真アップロード（JPEG/PNG → テクスチャ差し替え） |
| Phase 2 | サイズ・フォーマット検証、リサイズ |
| Phase 3 | エモーション音声（プリセット → UGC アップロード） |
| Phase 4 | 3D モデル対応（.glb） |
| Phase 5 | モデレーション、通報機能 |

Phase 1 は現在の `textureUrl` の仕組みに最小限の追加で実装できる。
アップロード API を1つ追加し、ユーザーごとの画像を保存すれば動く。

Phase 3 のエモーション音声はプリセットであればアップロード不要で、
クライアント側に音声ファイルを同梱するだけで実装できる。
