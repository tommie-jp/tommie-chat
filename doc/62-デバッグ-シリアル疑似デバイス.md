# 62-デバッグ-シリアル疑似デバイス

外部CPU接続機能（[59-設計-外部CPU接続.md](59-設計-外部CPU接続.md)）を、Pico 2 などの実機なしで開発・検証するための手段。

## 用途

- ブラウザ側の Web Serial / WebUSB ロジックを単体テストしたい
- UART プロトコル（`M D3\r\n` 等）の挙動を手打ちで確認したい
- 大会前にリバーシ通信ロジックを実機抜きで詰めたい

## 方法 1: HHD Free Virtual Serial Ports（仮想 null-modem ペア） ★推奨

Windows 上で **COM1 ⇔ COM2** のような仮想ペアを作る。
端末ソフトが片側、ブラウザが反対側を開くと、両者が通信できる。

**2026-04 時点の採用方針**: com0com の代わりに
[HHD Free Virtual Serial Ports](https://freevirtualserialports.com/) を使う。
Win11 での署名問題（コード 52／メモリ整合性 OFF／F7 起動等）が発生せず、
インストール一発で動く。実機（COM1 ⇔ COM2）で動作確認済み。

```text
VSCode Serial Monitor (COM2) ⇔ 仮想ヌルモデム ⇔ ブラウザ Web Serial (COM1)
```

- **COM1**: tommieChat 側（ブラウザ Web Serial が開くポート）として想定
- **COM2**: Windows 11 VSCode Serial Monitor で接続してデバッグ

### セットアップ

1. <https://freevirtualserialports.com/> からインストーラ DL
   - 個人利用無料（商用は別途ライセンス）
   - インストーラは **管理者として実行**
2. 起動 → 「Create virtual ports pair」等で COM1 ⇔ COM2 のペアを作成
   - 番号は他で使われていなければ自由。1/2 が使われている場合は空き番号を指定
3. デバイスマネージャーで **「ポート(COM と LPT)」配下に COM1/COM2 が出ているか確認**
4. VSCode Serial Monitor: COM2 に接続（後述）
5. ブラウザ: <https://mmo-test.tommie.jp/test-web-serial-api.html> を開いて
   API=自動 or Native、新接続 → COM1 を選択
6. 端末ソフトでタイプした文字 → ブラウザ側の textarea に表示
   ブラウザの送信欄から打った文字 → 端末ソフトに表示

### 端末ソフトは VSCode Serial Monitor を推奨

- **VSCode 拡張**「Serial Monitor」（Microsoft 製）が最も確実
- TeraTerm はバージョン・設定によって仮想ポートを開けない場合がある
- VSCode Serial Monitor は Port ドロップダウンに COM ポートがそのまま表示され開ける
- Baud rate / Line ending を UI で切替可能
- 改行送信は `Send as Text` + Line ending=CRLF が便利

### メリット

- ハード不要
- 双方向通信を即試験できる
- プロトコル試作のイテレーションが超速い
- HHD 版は Win11 の署名問題なし（com0com の「コード 52」で詰まらない）

### 注意

- Native Web Serial 限定。WebUSB polyfill は仮想 COM を見ない
- → Windows Chrome の Native API 利用が前提

### 代替ツール

| ツール                                | 価格                    | 署名                   | 特徴                                                   |
| ------------------------------------- | ----------------------- | ---------------------- | ------------------------------------------------------ |
| **HHD Free Virtual Serial Ports**     | 個人無料                | Win11 現行署名         | 本ドキュメントの主軸。インストール一発、署名問題なし   |
| **com0com**                           | 無料 (OSS)              | 旧式・コード52対応必要 | 署名問題で詰まりやすいが無料・OSS                      |
| **Eltima Virtual Serial Port Driver** | 有料 ($170〜)           | フル対応               | 商用クオリティ、サポート有                             |
| **VSPE (Eterlogic)**                  | 32bit 無料 / 64bit 有料 | 古め                   | TCP/UDP ブリッジも可                                   |

#### com0com を使う場合の注意（過去の運用メモ）

署名問題で詰まったときの参考として残す。

- 配布物: `com0com-3.0.0.0-i386-and-x64-signed.zip`（Win11 は署名付き版必須）
- Setup GUI でペア作成時、CNCA0/CNCB0 両方で **「use Ports class」にチェック**
  （外すとデバイスマネージャーの「ポート(COM と LPT)」に出ず、アプリから開けない）
- 既定の COM5/COM6 が予約済みだと Windows が空き番号（COM4/COM7 等）を自動割当
- Win11 で「コード 52」が出る場合:
  1. 設定 → Windows セキュリティ → デバイス セキュリティ → コア分離 →
     **メモリ整合性 OFF** → 再起動
  2. それでもダメなら 設定 → 回復 → PC の起動をカスタマイズ → 再起動 →
     トラブルシューティング → 詳細オプション → スタートアップ設定 → 再起動 →
     **「7」キー**でドライバ署名強制を無効化（一回限り）
  3. Secure Boot 有効時は `bcdedit /set testsigning on` は不可

## 方法 2: USB-UART 2本ループバック（ハード派）

USB-UART 変換アダプタを 2 本（FTDI / CP2102 / CH340 等）。
TX↔RX をクロス接続して、それぞれ別 COM ポートとして認識させる。

```text
PC -- [USB-UART#1 COM3] --TX─┐ ┌─RX-- [USB-UART#2 COM4] -- PC
                              X
                          --RX─┘ └─TX--
```

- TeraTerm: COM3
- ブラウザ: COM4

ハード持ってるならこちらでも可。ただし配線ミス・GND 接続忘れ注意。

## 方法 3: socat（Linux/WSL 系）

Linux/WSL 上で：

```bash
socat -d -d pty,raw,echo=0 pty,raw,echo=0
```

→ 2 つの `/dev/pts/N` ペアが生成される。
ただし **Windows 側 Chrome の Web Serial からは見えない** ので、ブラウザも WSL 上で動かす必要あり（GUI 環境必要）。Windows メインなら方法 1 を推奨。

## 推奨ワークフロー

1. **方法 1 (com0com) で UART プロトコル仕様を固める**
   - `M D3\r\n`, `PASS\r\n`, `RESIGN\r\n` 等を手打ちで投げて挙動確認
   - 改行コード、エコーバック有無、タイムアウトなどの仕様を詰める
2. **Pico 2 (実機) に同じプロトコルを実装**
3. **大会本番**: Pico 2 直結でデモ

## 応用: Python リバーシAI を疑似 CPU として使う

仮想シリアルペアを使えば、**Pico 2 を作らずとも tommieChat で対戦テストが可能**。

```text
Python リバーシAI (COM2) ⇔ HHD 仮想ペア ⇔ tommieChat (COM1, ブラウザ Web Serial)
```

### 実行環境

**bot.py は Windows 11 側で実行する。** WSL2 Ubuntu からは動かせない。

- HHD（および com0com）の仮想 COM ドライバは Windows ネイティブ。
  WSL2 からは `/dev/ttyS*` として見えない
- `usbipd-win` は実 USB デバイス用で、仮想 COM には使えない
- 従って pyserial で `COMx` を開くコードは Windows 側の Python で動かす

セットアップ:

```powershell
# Windows 11 PowerShell
py -m pip install pyserial
py bot.py
```

### 構成例

`bot.py`（pyserial 利用、最小例）:

```python
# Windows 11 で実行する。VSCode Serial Monitor の代わりに COM2 をこの bot が占有し、
# ブラウザ側（tommieChat）は COM1 を開いて対戦する。
import serial
ser = serial.Serial('COM2', 115200, timeout=1)
print('Bot 起動: COM2 で待機')
while True:
    line = ser.readline().decode().strip()
    if not line:
        continue
    print(f'受信: {line}')
    # 簡易ロジック: 受信したらランダムな手を返す
    if line.startswith('M '):
        # 相手の手を受け取った → 自分の手を返す
        my_move = compute_next_move(line[2:])  # 自作AI
        ser.write(f'M {my_move}\r\n'.encode())
```

### この使い方のメリット

- **大会前に対戦フローを完全検証可**
  - tommieChat ↔ Python AI で何試合でもプレイ可能
  - プロトコル不具合・タイムアウト挙動・終局判定をすべて潰せる
- **大会当日の保険デモ**
  - Pico 2 が動かなくても、Python AI で「動くリバーシ通信」をデモできる
  - 「自作CPUは間に合わなかったが、本来こう繋がる」と説明可能
- **強い AI を試せる**
  - Python なら edax / minimax / ニューラルネット等の既存実装を流用できる
  - 自作 CPU の対戦相手として準備しておけば「観客 vs Python AI」も成立

### この使い方の注意

- HHD / com0com は Windows 専用なので、bot.py も Windows 側で動かす
- WSL2 Ubuntu から Windows の仮想 COM は見えない
  （`usbipd-win` は実 USB 向けで、仮想 COM ドライバには効かない）
- macOS / Linux だけで完結させたい場合は socat で `/dev/pts/N` ペアを作り、
  ブラウザもその OS 上の Chrome で動かす必要がある（方法 3 参照）
