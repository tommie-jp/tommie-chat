# 62-デバッグ-シリアル疑似デバイス

外部CPU接続機能（[59-設計-外部CPU接続.md](59-設計-外部CPU接続.md)）を、Pico 2 などの実機なしで開発・検証するための手段。

## 用途

- ブラウザ側の Web Serial / WebUSB ロジックを単体テストしたい
- UART プロトコル（`M D3\r\n` 等）の挙動を手打ちで確認したい
- 大会前にオセロ通信ロジックを実機抜きで詰めたい

## 方法 1: com0com（仮想 null-modem ペア） ★推奨

Windows 上で **COM4 ⇔ COM7** のような仮想ペアを作る（番号は環境依存）。
端末ソフトが片側、ブラウザが反対側を開くと、両者が通信できる。

```text
VSCode Serial Monitor (COM4) ⇔ 仮想ヌルモデム ⇔ ブラウザ Web Serial (COM7)
```

### セットアップ

1. <https://sourceforge.net/projects/com0com/> から DL
   - Windows 11 は署名付き版必須: `com0com-3.0.0.0-i386-and-x64-signed.zip`
   - インストーラは **管理者として実行**
2. インストール → Setup GUI でペアを作成
   - 既存の "pair 0 (COM5 <-> COM6)" は **CNCA0/CNCB0 両方で「use Ports class」にチェック**
   - チェックしないとデバイスマネージャーの「ポート(COM と LPT)」に出ず、アプリから開けない
   - COM5/COM6 が他で予約済みだと、Windows が空き番号（例: COM4/COM7）を自動割当
3. デバイスマネージャーで **「ポート(COM と LPT)」配下に COM4/COM7（または COM5/COM6）が出ているか確認**
4. 端末ソフト（VSCode Serial Monitor 推奨、後述）: 片側（例 COM7）で接続
5. ブラウザ: <https://mmo-test.tommie.jp/test-web-serial-api.html> を開いて API=Native、Connect → 反対側（例 COM4）を選択
6. 端末ソフトでタイプした文字 → ブラウザ側の textarea に表示
   ブラウザの送信欄から打った文字 → 端末ソフトに表示

### 端末ソフトは VSCode Serial Monitor を推奨

- **VSCode 拡張**「Serial Monitor」（Microsoft 製）が最も確実
- TeraTerm はバージョン・設定によって com0com 仮想ポートを開けない場合がある
- VSCode Serial Monitor は Port ドロップダウンに `COM7 - com0com - serial port emulator (COM7)` のように表示され、そのまま開ける
- Baud rate / Line ending を UI で切替可能
- 改行送信は `Send as Text` + Line ending=CRLF が便利

### Win11 ドライバ署名エラー（コード 52）の対処

インストール後、デバイスマネージャーで「！」が付いて「コード 52」と出る場合:

1. 設定 → Windows セキュリティ → デバイス セキュリティ → コア分離 → **メモリ整合性 OFF** → 再起動
2. これで直らない場合: 設定 → 回復 → PC の起動をカスタマイズ → 再起動 → トラブルシューティング → 詳細オプション → スタートアップ設定 → 再起動 → **「7」キー**でドライバ署名強制を無効化（一回限り）
3. Secure Boot 有効環境では `bcdedit /set testsigning on` は使えない（Secure Boot ポリシーで拒否される）

### メリット

- ハード不要
- 双方向通信を即試験できる
- プロトコル試作のイテレーションが超速い

### 注意

- Native Web Serial 限定。WebUSB polyfill は仮想 COM を見ない
- → Windows ChromeのNative API利用が前提

### 代替ツール（com0com が動かない / 面倒なとき）

| ツール                                | 価格                    | 署名                   | 特徴                                                   |
| ------------------------------------- | ----------------------- | ---------------------- | ------------------------------------------------------ |
| **com0com**                           | 無料 (OSS)              | 旧式・コード52対応必要 | 本ドキュメントの主軸。動けば問題なし                   |
| **HHD Free Virtual Serial Ports**     | 個人無料                | Win11 現行署名         | GUI 今風、インストール一発。署名問題なし。**有力候補** |
| **Eltima Virtual Serial Port Driver** | 有料 ($170〜)           | フル対応               | 商用クオリティ、サポート有                             |
| **VSPE (Eterlogic)**                  | 32bit 無料 / 64bit 有料 | 古め                   | TCP/UDP ブリッジも可                                   |

おすすめは **[HHD Free Virtual Serial Ports](https://freevirtualserialports.com/)**:

- 個人利用無料（商用は別途）
- Win11 で署名問題なし（メモリ整合性 OFF / F7 起動が不要）
- 仮想ペア・スプリッタ・マージなど機能豊富

ただし現状 com0com で動いているなら**乗り換えるメリットは小さい**。
新規マシンへのセットアップや、署名問題で詰まったときの代替案として覚えておく。

## 方法 2: USB-UART 2本ループバック（ハード派）

USB-UART 変換アダプタを 2 本（FTDI / CP2102 / CH340 等）。
TX↔RX をクロス接続して、それぞれ別 COM ポートとして認識させる。

```text
PC -- [USB-UART#1 COM3] --TX─┐ ┌─RX-- [USB-UART#2 COM4] -- PC
                              X
                          --RX─┘ └─TX--
```

- TeraTerm: COM3
- ブラウザ: COM4

ハード持ってるならこちらでも可。ただし配線ミス・GND 接続忘れ注意。

## 方法 3: socat（Linux/WSL 系）

Linux/WSL 上で：

```bash
socat -d -d pty,raw,echo=0 pty,raw,echo=0
```

→ 2 つの `/dev/pts/N` ペアが生成される。
ただし **Windows 側 Chrome の Web Serial からは見えない** ので、ブラウザも WSL 上で動かす必要あり（GUI 環境必要）。Windows メインなら方法 1 を推奨。

## 推奨ワークフロー

1. **方法 1 (com0com) で UART プロトコル仕様を固める**
   - `M D3\r\n`, `PASS\r\n`, `RESIGN\r\n` 等を手打ちで投げて挙動確認
   - 改行コード、エコーバック有無、タイムアウトなどの仕様を詰める
2. **Pico 2 (実機) に同じプロトコルを実装**
3. **大会本番**: Pico 2 直結でデモ

## 応用: Python オセロAI を疑似 CPU として使う

com0com を使えば、**Pico 2 を作らずとも tommieChat で対戦テストが可能**。

```text
Python オセロAI (COM5) ⇔ com0com ⇔ tommieChat (COM6, ブラウザ Web Serial)
```

### 構成例

`bot.py`（pyserial 利用、最小例）:

```python
import serial
ser = serial.Serial('COM5', 115200, timeout=1)
print('Bot 起動: COM5 で待機')
while True:
    line = ser.readline().decode().strip()
    if not line:
        continue
    print(f'受信: {line}')
    # 簡易ロジック: 受信したらランダムな手を返す
    if line.startswith('M '):
        # 相手の手を受け取った → 自分の手を返す
        my_move = compute_next_move(line[2:])  # 自作AI
        ser.write(f'M {my_move}\r\n'.encode())
```

### この使い方のメリット

- **大会前に対戦フローを完全検証可**
  - tommieChat ↔ Python AI で何試合でもプレイ可能
  - プロトコル不具合・タイムアウト挙動・終局判定をすべて潰せる
- **大会当日の保険デモ**
  - Pico 2 が動かなくても、Python AI で「動くオセロ通信」をデモできる
  - 「自作CPUは間に合わなかったが、本来こう繋がる」と説明可能
- **強い AI を試せる**
  - Python なら edax / minimax / ニューラルネット等の既存実装を流用できる
  - 自作 CPU の対戦相手として準備しておけば「観客 vs Python AI」も成立

### この使い方の注意

- com0com は Windows 専用なので、開発機が Windows である必要あり
- macOS なら socat + Chrome を Linux VM で動かす等の工夫が必要
